## 单窗口失控处理规则

这轮工作已经证明，一个窗口长时间连续推进会带来上下文腐烂、任务漂移和“看起来一直在干活，实际越跑越偏”。以后不要再让一个窗口长期无人值守式推进。

出现下面任一情况，就开新窗口：

- 当前窗口已经发生上下文压缩，还要从调研转到写作、从写作转到审查，或从方法转到样本。
- AI 两次以上用解释、道歉或原则承诺回应，但没有新增可检查验收。
- 同一窗口连续改过方法论、文档标准、样本、skill、verifier 或 harness 多个层级。
- 用户开始看不清它到底改了什么、如何验证、哪里还没证明。
- 任务已经从“做一个最小改动”变成“顺便把整个系统整理好”。

以后按多窗口角色推进，但默认只开两个必要窗口：

| 窗口 | 职责 | 是否改文件 | 退出条件 |
|:---|:---|:---:|:---|
| 执行窗口 | 只做一个层级的一个任务 | 可以改 | diff 可审、验证完成、提交或说明未提交原因 |
| 审查窗口 | 只读审查 diff 和文档质量 | 默认不改 | 输出问题清单和证据，不顺手修 |

主控不是常驻第三个窗口。只有当执行和审查结论冲突、任务跨多个层级、或需要整理下一轮路线时，才开一个短的主控窗口，用来定目标、锁层级、合并判断和更新交接。

调研窗口和回归窗口也不是默认常驻角色。需要外部资料时开调研窗口，只交付“资料到判断”；需要跑样本、verifier、渲染或 smoke 时开回归窗口，只交付结果和失败原因。

核心规则：一个窗口只解决一个层级的一类问题。跨层问题写成后续回收，不顺手修改。

## 硬约束

- 不读取或引用 `archive/`，包括 `output/**/archive/**`。存档只供人工溯源。
- archive 禁读指 Agent 不得手动打开、搜索、复制或引用历史/中间文件；脚本通过 `--prof-dir` / `ProfDirResolver` 读取当前 active `_internal/archive/<ts>` 不属于手动读取 archive。
- 不提交 `.codex/` 或未明确纳入项目的本地目录，例如 `精髓转移/`。
- 导师成品根目录只能暴露 `00_材料导读.md`、`01_基础画像.md`、`02_领域地图.md`、`03_论文路线.md`、`04_学习向导.md` 和 `_internal/`。
- 来源必须可追溯；缺失写 `[未找到]`；弱推断写 `需人工复核`。
- 不做导师评价，不写匹配度、推荐申请、是否值得报考等判断。
- 每个里程碑只解决一类问题：成品质量、证据可靠性、教学设计、可视化、端到端自动化、项目瘦身或 verifier 底线。
- verifier 只做 smoke，不冒充学术质量。学术质量来自证据、蓝图、自审、回归和人工反馈。

## 已稳定的 harness 设计

### Fact Pack

阶段 1 仍是有价值的事实层资产：`phase1_init.py`、Google Scholar / OpenAlex / arXiv 采集、`risk_gate.py`、受控身份修正、受控论文剔除、`render_profile.py` 和 `verify_profile.py`。

新导师从三行输入初始化：

```bash
python src/phase1/phase1_init.py \
  --university "<大学>" \
  --institute "<学院或研究所>" \
  --department "<部门>" \
  --name "<中文名>" \
  --official-url "<导师官网 URL>"
```

后续统一使用脚本输出的 `prof_dir` 和 `--prof-dir` 参数。不要手动拼 `_internal/archive/<ts>`。

### Cognitive Blueprint

每位导师在写 `02-04` 前必须维护 `_internal/blueprint.md`。蓝图不是过程日志，而是认知设计合同，至少覆盖：

- 读者起点
- 导师当前方向一句话
- 目标论文及选择理由
- 核心图或图组
- 平台链路
- 课程到论文的学习桥
- 可视化计划
- 证据风险和人工复核

蓝图薄弱时先继续检索和核证，不要继续润色正文。成品正文必须消费蓝图，而不是只让 verifier 看到蓝图字段存在。

### 成品质量底线

每份成品文档都按“小白执行 AI + 小白学生读者”设计，必须自包含：说明本文解决什么问题、为什么这样读、读完应获得什么能力；关键术语和缩写首次出现时展开；不能假设读者读过项目规则、聊天记录或其他导师样本。

可视化是理解工具，不是装饰。不要把所有内容画成 Mermaid；领域定位优先层级树或定位表，论文路线优先问题-方法-角色矩阵，学习向导优先课程-能力-论文桥接表，真实流程或平台链路才用小型 Mermaid 或文本链路。

### 讨论即沉淀

当前进入逐文档、逐小节的人工式质量审查。顺序是先讨论整份文档职责和大纲，再讨论小节正文逻辑、图文配合和出口能力；单份文档完成后回看整篇衔接，五份文档完成后回看 `00-04` 整体认知阶梯。

每次用户和 AI 讨论出稳定结论后，先判断它处在哪一层。属于五份成品结构和章节逻辑的，先进 `quality-workbench/文档标准/`；属于学习设计、信息筛查、证据分级、图文表达等跨文档方法的，先进 `quality-workbench/支撑方法/`。当前阶段不要把每个小反馈立刻升格到正式 Harness。只有跨导师可复用、能稳定提高质量、且归属明确的结论，才在统一回收时进入 `docs/计划书.md`、`.agents/skills/research-advisor/references/`、`assets/templates/`、verifier 或 tests。

`quality-workbench/文档标准/00_材料导读.md` 已完成第一轮结构设计并提交为 `b77c4ed3 docs: finalize material guide workbench`。当前五节结构是 `材料定位`、`阅读顺序`、`读后自检`、`来源核查`、`参考资料`；文首有目录和显式 anchor；底部保留 `已确认规则`、`成品试写时验证`、`后续落地清单`。早期命名和入口结构不再作为当前 `00` workbench 结构。

当前协作模式是用户实时反馈、AI 逐步讨论和修改；旧的“长时间无人值守自进化”目标只作历史背景，不再作为默认工作方式。下一步是继续 `quality-workbench/文档标准/01_基础画像.md` 讨论，而不是直接重构正式模板或 verifier。

- 支撑方法讲原则，自己却没有从真实问题走到可执行动作。
- 参考资料放在文末，正文看不出资料怎样改变判断。

- 可视化最容易变成图形配额：把表格包装成图，把平台、时间、学习都画成相似流程，把论文角色和证据边界都画成节点连线。
- AI 遇到反馈后容易承认问题，再做局部修补，但没有把反馈变成新的验收用例。

后续每次修订都要把用户反馈转成可失败、可检查的门：

```text
这次要阻止什么具体失败？
为什么之前会失败？
资料或项目经验支持哪个判断？
执行者下一次要做什么不同动作？
审稿人怎样看出它没有再犯？
这条规则暂时不覆盖什么？
```
