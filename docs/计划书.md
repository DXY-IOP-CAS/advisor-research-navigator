## 文件规范

本计划书统一按以下规则组织。

**编号体系**：h2（##）用于章（第X章），h3（###）用于节（X.Y），h4（####）用于节内子段（X.Y.Z）。禁止 h5+。

**每节三段式**（仅工具设计章强制，h3 下三个 h4 子段）：
- 工作流程 — 做什么，自然语言描述
- 技术原理 — 方法、API、异常处理
- 输出规格 — 文件名、模板、格式

**h4 子段内只用正文段落**。不插入块引用、粗体伪标题、列表嵌套或其他制造新层级的格式。

**代码块限三类**：示例输出、API 请求/响应、模板/骨架。

**已沉淀 / 待观察二分**：所有"已确认"内容写进正文；所有"未决定 / 待研究 / 需要批量验证"事项写进附录 B，不混进正文。

**目录只列到 h3**。

## 第1章: 需求背景

### 1.1 起因 / 工具目标

每年硕士/博士生选导师是刚需。每年研一 3-4 月学生面临报志愿和套磁，核心痛点在四个环节。官网信息粗糙：只有论文列表和标题，没有科普级解读。有论文看不懂：论文要求研究生以上知识，学生只有本科基础。资源散乱无索引：论文、教材、视频各在各处。调研易绕圈：不知道该读什么、按什么顺序。

工具用四段嵌套解决这些问题。每一段的输出为后一段提供认知上下文，范围逐段收窄，理解逐段加深。工具同时承担三个递进职能：调研（全面收集信息）、综述（分析串联知识）、教学（设计递进学习路线）。

项目采用先单点验证、再批量扩展的路线。当前先解决用户本人申请单个导师的真实需求；下一步扩展到中国科学院物理研究所一个中心或整个物理所；再下一步扩展到中国科学院大学相关专业；长期目标是覆盖全国高校和科研院所的相关专业导师。这个扩展目标要求所有阶段都必须标准化、可复用、可验证，不能依赖某次人工提示或某个 agent 的临场发挥。

| 段 | 做什么 | 做法（高层次） | 目的 |
|:---|:---|:---|:---|
| 第1段 | 老师基础信息调研 | 地毯式收集官网、学术平台、中文百科、采访稿等多维信息，多源交叉比对 | 输出基础画像，明确领域范围 |
| 第2段 | 领域发展调研 | 基于第1段的领域范围，追溯子领域发展脉络和当前阶段 | 输出认知地图，理解位置 |
| 第3段 | 论文串联与定位 | 以第2段认知框架重新审视论文序列 | 标注每篇论文在领域树上的位置 |
| 第4段 | 从零到前沿向导 | 从高数+线代起步反向设计学习路线 | 输出递进式向导（小字典） |

嵌套逻辑：大领域一>小领域一>老师定位一>学习路径。后一段建在前一段的认知之上。

技术实现细节在第2章和附录 A。

### 1.2 目标用户画像

工具的目标用户是大二/大三物理本科生，正在准备申请硕博选导师。核心场景是研一 3-4 月报志愿前，用工具理解导师当前研究方向、论文路线和进组前学习准备，而不是给导师打分或计算匹配分数。

工具假设用户从高数和线代起步。实际用户可能已有四大力学、固体物理等基础，但所有高阶知识（复变、数理方法、高量、群论、量子多体）本质上是高数和线代的逐步推广。从根推导才能零跳步建立真理解。

教材在 AI 时代不再是逐页通读的课本，而是字典。本科到研一的教材体系分三个层面：

| 层面 | 课程 | 定位 |
|:---|:---|:----|
| 数学基础 | 高数、线代、概率论、复变、数理方法 | 推导起点 |
| 物理基础 | 普物、四大力学、固体物理 | 物理直觉来源 |
| 高阶延伸 | 高量、群论、量子多体、量子场论、规范场论 | 大字典边界—领域触及的范围 |

教材从课本转向字典，有两个原因。第一，教材面向通用人才培养，覆盖完整体系，但具体方向只用其中部分知识点。逐页通读在无关内容上浪费大量时间。第二，学生可在 AI 辅助下按需读取字典条目，带着研究方向去学，既能深化具体知识点理解，又能强化整体框架认知。

第4段产出的讲义是个性化小字典。它从教材体系（大字典）中只提取第1-3段调研确认需要的知识点，串联成一条从高数+线代到前沿的逻辑链。小字典扎根于大字典，以教材体系为参照系，约束 AI 创作边界，防止偏离轨道。

一个可扩展的方向：未来单独开发教学 Agent，教用户如何配合 AI 递进学习。AI 的优势是能根据用户水平动态适配，但多数人不擅长用 AI 辅助学习。教学 Agent 不属于当前导师研究方向 Harness 的范围；当前项目先把导师事实锁定、领域地图、论文路线和学习路径四份文档稳定做好，并完成中国科学院物理所批量化后，再进入教学 Agent 项目。阶段 4 只负责产出可交给学生和未来教学 Agent 使用的知识地图、学习路径和小字典，不承担实时教学、个性化诊断或互动训练。

### 1.3 工具差异化与定位

18+ 同类工具调研后确认：市场无直接竞品。现有工具分三类，各占一个不重叠的位置。

| 类别 | 代表工具 | 做什么 | 与我们的重叠 |
|:-----|:---------|:-------|:-----------:|
| 导师评价 | 导师评价网、GradChoice、advisor-calculator | 口碑评分（人品、管理、毕业） | 不重叠——我们不碰评价 |
| 导师匹配 | PhdFit、StreamlinedAI、PhDuo | 从学生 CV 推荐导师 | 不重叠——我们不做推荐 |
| 学者画像 | supervisor、SciAtlas、ai-talent-graph | 论文+主页一>信息聚合报告 | 第1段部分重叠 |

学者画像工具停在"导师长什么样"的层面。我们继续往前推到领域发展、论文路线和教学讲义。

工具的准确定位包含四层含义。第一，通用模板引擎：对任何机构的任何老师适用，跨老师复用零边际成本，首个实例为 IPHY 超快物质科学中心某位导师。第二，开源方法论：Claude Code Skill + `src/phase1/` 脚本 + Markdown 模板形态，不是网站/App，零部署。第三，从调研走到教学：调研+综述+教学三职能递进。第四，调研驱动设计：每项决策经同类工具验证，不凭空发明。

批量化不是简单循环运行脚本。批量化的核心是质量一致：同一阶段在不同导师、不同学科、不同机构上承担同一认知职责，使用同一证据等级和同一失败处理策略。阶段一锁定事实底座；阶段二必须建立领域地图；阶段三必须解释论文在领域路线中的位置；阶段四必须把前沿倒推成学生可执行的学习路径。

一条根本差异区分了我们和所有竞品：同类工具帮学生找谁合适（匹配引擎）或知道老师什么样（画像工具）；我们帮学生学懂一个方向。

工具六不做：不写教材正文；不替代主动学习；不点评老师或推荐方向；不捏造论文/教材链接；不评价导师口碑；不搭建网站。

## 第2章: 四阶段流水线设计

**技术落地细节见** `src/phase1/pipeline.md`（CLI 命令、字段格式、API 参数）。本章只记录设计决策，不重复实现细节。

### 2.0 端到端质量重构基线

本项目的目标不是“跑通 phase”，而是从最小输入 `姓名 + 机构路径 + 官网 URL` 端到端产出质量过关、可读、可审查、可批量复刻的 `00-04` 五份导师文档。旧 phase、旧脚本、旧模板和旧 verifier 都只是候选资产；能服务最终质量就保留，妨碍质量、可审查性或项目瘦身就降级、合并或废弃。

当前架构审计结论如下。

| 类别 | 当前资产 | 判断 | 后续处理 |
|:---|:---|:---|:---|
| 事实层资产 | `phase1_init.py`、三源采集、身份锁定、`risk_gate.py`、受控身份/论文复核脚本、`render_profile.py`、`verify_profile.py` | 这些资产解决身份、论文归属和目录状态，仍服务端到端质量 | 保留，但新入口优先使用 `phase1_init.py + --prof-dir`；`run.py` 只作兼容捷径，不作为新 workflow 的核心 |
| 写作层债务 | `02/03/04` 模板只有章节骨架，缺少写作前的认知设计合同 | 容易导致张鹏举与汪非凡深度不一致，AI 按模板填空而不先理解导师主线 | 在成品前新增 `_internal/blueprint.md` 作为认知蓝图；模板只消费蓝图，不替代判断 |
| 过程沉积 | `docs/e2e/`、`docs/superpowers/specs/`、过往回归记录、旧 quickstart 类文档 | 对溯源有价值，但不能成为执行 AI 的主入口 | 保留为历史记录或待瘦身对象；长期入口只认 `AGENTS.md`、本计划书、`research-advisor` skill、`src/phase1/pipeline.md`、`docs/上下文交接.md` |
| 确定性底线 | `verify_phase_docs.py`、`verify_source_metadata.py`、`verify_mermaid_render.py` | 能拦截结构、引用、目录、长段和部分 DOI 错误，但不能证明学术质量 | 降级为底线 smoke；真正质量来自证据包、认知蓝图、写作自审和样本回归 |

新端到端主路径采用六段闭环。

| 段 | 产物 | 解决的问题 | 失败时怎样恢复 |
|:---|:---|:---|:---|
| 1. 最小输入 | `prof_dir` 和 `_internal/latest.txt` | 只让用户给姓名、机构路径、官网 URL | 输入不全时只补这三项，不要求 GS/OA/ORCID/论文列表 |
| 2. Fact Pack | `_internal/archive/<ts>/` 中的身份、履历、论文和风险状态 | 锁定“这个人是谁、哪些论文属于他、哪些数据有风险” | `risk_gate` 进入 conservative 时按 `next_actions` 定向补证据或剔除，不靠顺滑文字绕过 |
| 3. Cognitive Blueprint | `_internal/blueprint.md` | 在写成品前固定导师主线、目标论文、核心图、平台链路、学习路径、证据风险和可视化计划 | 蓝图薄弱时暂停写作，继续检索官网、论文、综述、课程或图源 |
| 4. Product Draft | `00-04` 五份 Markdown | 把蓝图翻译成自包含、面向新手、图文结合的读者材料 | 不允许只填模板；每份文档必须消费蓝图中对应的目标、图示和证据判断 |
| 5. Review Loop | `_internal/evidence/key_claims.md` + 自审记录 + verifier | 区分直接证据、交叉证据、弱推断和需人工复核，机械门只兜底 | smoke 通过但内容薄弱时记录学术风险，回到蓝图或检索，而不是继续润色 |
| 6. Regression & Slimming | 张鹏举黄金样本 + 非张样本回归 + 去重审计 | 检查质量能否复刻，并清掉被正式规则取代的过程沉积 | 回归失败先归类：事实、证据、教学、可视化、自动化或瘦身，不混成无边界大改 |

`_internal/blueprint.md` 不是新的过程日志，而是成品前的认知设计合同。它至少回答八个问题：读者起点是什么；导师当前方向一句话是什么；目标论文为什么选它；核心图或图组是什么；平台链路如何画；从课程到目标论文的学习桥是什么；哪些判断只有弱证据；每份成品用什么可视化最省认知负担。蓝图可以先用 Markdown 表格维护，只有当自动化需要稳定字段时再考虑 JSON schema。

本基线吸收三类外部方法，但不生搬硬套。[OpenAI Harness Engineering](https://openai.com/index/harness-engineering/) 强调把仓库内知识变成 agent 可见的事实系统，并用反馈环降低漂移；[Claude Code Skills](https://docs.anthropic.com/en/docs/claude-code/skills) 文档强调 skill 主入口要短，支撑文件按需加载，并用真实 prompt 做质量评估；[Backward Design](https://cft.vanderbilt.edu/guides-sub-pages/understanding-by-design/) 要求先确定学习终点和可接受证据，再设计学习活动。本项目对应为：仓库长期规则只保留少数入口，`research-advisor` skill 负责渐进式披露，`04_学习向导.md` 从目标论文、核心图和平台问题反推课程路径。

### 2.1 整体工作流

用户提供导师姓名 + 机构 + 官网 URL，工具执行四份文档流水线。每份文档的认知上下文来自前一份，深度逐段加深。早期需求曾按三层结构描述为“信息汇总 → 领域认知 → 学习向导”；当前四份文档不是推翻三层结构，而是把“论文串联与定位”从领域认知和学习向导之间独立出来，作为从导师事实走向学生学习路径的中间桥梁。

```
阶段1：基础画像 → 结构化画像 + 按学术履历分组的论文列表
    ↓
阶段2：领域地图 → 当前申请方向的学科位置、发展脉络、关键问题
    ↓（认知回环，非单向）
阶段3：论文路线 → 用领域框架解释相关论文的研究路线和技术细节
    ↓
阶段4：学习向导 → 从高数+线代到前沿的递进学习路径
```

阶段 1 的论文按学术履历阶段分组，不是为了按年份归档，而是为了观察导师在不同身份、单位和合作网络下研究方向如何微调或转向。学生申请导师时，最需要理解的是导师当前单位、当前课题组中最可能从事的研究领域；早期履历的价值在于解释当前方向从哪里来、哪些能力和问题意识被延续下来。

阶段 2 建立当前方向的领域认知，反过来让阶段 3 的论文串联更有深度。阶段 3 不是简单的时间线排列，而是在“老师当前做什么领域”的基础上讲清“哪些论文属于当前主线、哪些是前史积累、为什么这些技术路线会收敛到现在的课题”。阶段 4 再把前三段反向翻译成学生可执行的学习路径。

**阶段一览**：

| 阶段 | 输入 | 产出 | 核心任务 | 人闸 |
|:-----|:-----|:-----|:---------|:----:|
| 1 基础画像 | 姓名+机构+官网URL | `01_基础画像.md` | 官网提取身份+履历 → 三源论文采集 → 同名过滤 → `career_stages.json` 按履历切分论文 → 9 节画像 | 无（AI 自主 + verify 门控） |
| 2 领域地图 | 画像+论文列表 | `02_领域地图.md` | 快速回顾导师学术路径，重点解释当前申请方向在学科层级中的位置、发展脉络、关键问题和前沿 | AI 自主 + phase docs smoke + 内容门审阅 |
| 3 论文路线 | 领域框架+论文列表 | `03_论文路线.md` | 用阶段 2 的专业框架重读相关论文，区分当前主线、前史积累、旁支论文和需复核弱证据 | AI 自主 + DOI 元数据门 + 内容门审阅 |
| 4 学习向导 | 前三段全部产出 | `04_学习向导.md` | 从高数+线代和本科物理出发，反向设计进入当前方向前沿的递进学习路径，并用阶段产出物检查学生是否正在接近目标论文阅读能力 | AI 自主 + phase docs smoke + 内容门审阅 |

### 2.2 阶段 1：原理与策略

#### 2.2.1 三源互补设计

三个数据源各取所长，不追求任一源独立完整。

**GS 是论文列表主源，OpenAlex 做元数据补充，arXiv 做预印本时间差填补。** 选择 GS 做主源的判断来自一个根本事实：GS profile 由学者本人维护，OpenAlex 是机器聚类。对于任何学者，前者的论文完整性远高于后者。

| 对比维度 | Google Scholar | OpenAlex | arXiv |
|:---------|:---------------|:---------|:------|
| 论文列表来源 | 学者本人维护 | ML 算法聚类 | 作者自提交 |
| 对中文作者覆盖 | 完整（学者自己添加） | 不完整（22-38%） | 依学科而定 |
| 元数据（DOI/期刊/作者） | 无（只有标题+年份+引用数） | 完整 | 部分（DOI 可选） |
| 同名干扰 | 无（邮箱验证） | 高（需多级过滤） | 高（只靠 au: 文本匹配） |
| 限速风险 | CAPTCHA | 100K/天免费 | 3 秒/请求 |
| 可获得性 | GS ID 公开或可搜到 | OpenAlex ID 公开 | 全网公开 |

**采信优先级**：GS > OA > arXiv。多源重合论文以 GS 标题为准，OA 补充 DOI/期刊/作者，arXiv 补充预印本。

**GS 不可用时的降级**：如果找不到 GS profile 或被封锁，OpenAlex 做论文列表替补。AI 在画像开篇标注"未找到 GS profile，论文覆盖可能不完整"。三个数据源全部不可用则输出空列表。

#### 2.2.2 身份锁定（Phase A）

身份确认是整条流水线的基础。ID 错了，后面全是别人的数据。

**锁定协议**（四层递进，任一通过即停）：

- Tier 1（金标准）：官网 email 域名与 GS profile 的 verified email 域名匹配 → 身份锁定，GS profile 内所有论文属于此人，不执行同名过滤。这一层覆盖绝大多数教授。
- Tier 2（银标准）：ORCID 匹配。通过 OpenAlex 或官网获取 ORCID，与来源数据中的 ORCID 交叉验证。
- Tier 3（铜标准）：高度特定论文匹配。官网论文标题与 GS 论文标题精确匹配 3 篇以上 → 高度疑似同一人。
- Tier 4：全降级。以上都不通 → 跳过 GS，走 OpenAlex + arXiv。

**身份锁定完成后 AI 创建两个文件**：
- `verified_ids.json`：记录每个来源 ID、验证方式、通过哪一 Tier
- `career_stages.json`：从官网履历提取的结构化学术生涯记录

#### 2.2.3 学术阶段划分（core design decision）

**论文按履历分组的目的不是按年份归档，而是看每个职业阶段的研究方向演变。**

划分原则：**每个（时间区间 + 机构 + 职位）变化是一个独立阶段。** 机构没变但职位升了、方向变了，也要拆开。不能因为都在同一家单位就把多年揉成一团。

`career_stages.json` 同时承担三份职责：
1. 控制 §2 学术履历表的生成（脚本自动从 JSON 渲染表格）
2. 控制 §4 论文分组（脚本按年份区间把论文分配到对应阶段）
3. 给 AI 提供阶段叙事的锚点（每个阶段表格前写 1-2 句说明该阶段研究主题）

**它是学术履历的唯一事实源。** AI 写一份 JSON，脚本同时用它生成 §2 表格 + §4 分组。两条输出从同一份数据来，不可能不一致。旧方法（AI 手写 §2 表格 + 脚本默认 5 年分桶）已被放弃，因为两处数据会漂移。

#### 2.2.4 同名干扰过滤

过滤只在 GS profile 无法提供锚点（邮箱未验证或 GS 不存在）时触发。触发时使用三层网络评分，不依赖学科关键词（跨学科通用）：

- **合著者**：OA 独有论文与 GS 已确认论文共享合著者 → +2 分
- **期刊**：期刊名在已确认论文的期刊列表中 → +1 分
- **机构**：作者机构在已确认论文的机构列表中 → +1 分

总分 < 1 → 疑似同名干扰，过滤。≥ 1 → 保留。

arXiv 独有论文使用更简单的门控：有 DOI 的保留（可能是尚未被 GS 收录的预印本），无 DOI 的过滤（无法交叉验证）。

无 GS 锚点时的特例：整个 pipeline 只有 OpenAlex 数据。此时跳过过滤，全部保留，在画像中标注"未经合著者/期刊网络过滤，建议核查"。

#### 2.2.5 渲染与验证（Harness Loop）

渲染分两层：

1. **脚本生成骨架**（render_profile.py）：从 merged.json 读取导师信息和论文列表，从 career_stages.json 读取阶段配置 → 自动生成 §1 身份标识、§2 学术履历表、§4 论文分组表格、§8 数据质量统计。§3（研究方向）、§5（合作网络）、§6（公开信息）、§7（引用分析）、§9（验证来源）留占位符给 AI Edit。

2. **AI 填充叙事**：AI 通过 Edit 命令逐个替换占位符。不改论文表格行、不改格式，只写内容。

3. **verify 门控**（verify_profile.py）：最终质量检查，9 项全部通过才能声称完成。未通过时，AI 回到第 2 步修复，形成闭环：

```
渲染 → verify 检查 → 通过 → 完成
                    ↓ 未通过
                AI 看具体哪项失败 + 修复指引 → 修数据或改叙事 → 重新渲染 → 回到 verify
```

verify 的 9 项检查每项失败了会给出修复命令。Harness 不代替 AI 判断，但告诉 AI 往哪个方向修。

#### 2.2.6 arXiv 精确匹配流程

arXiv 的 `au:` 搜索基于字符串匹配，同名噪声率极高。优先使用更精确的方式：

1. **ORCID author identifier feed**（最精确）：如果作者已将 ORCID 关联到 arXiv 账号，访问 `https://arxiv.org/a/{ORCID}.atom2` 返回该作者全部论文（零噪声）。这是 opt-in 机制，不是所有作者都有。

2. **au: 搜索的优化**（前一步失败时使用）：将 `姓_名`（Wang_Shili）拆成 `au:Wang+AND+au:Shili` 匹配作者字段中同时出现姓和名的论文。加上 `-c` 学科分类过滤可进一步降低噪声。

3. **结果过滤**（render_profile.py 执行）：arXiv 独有论文无 DOI 的视为同名噪声过滤；有 DOI 的保留（可能是尚未被 GS 收录的预印本）。

**所有实现细节（CLI 命令、字段格式、API 参数、质量门清单）见 `src/phase1/pipeline.md`。**

项目采用三层文档结构，各层不重复：

| 层 | 受众 | 文件 | 内容 |

### 2.4 阶段 2/3/4 设计原则

#### 2.4.1 工作流程

阶段 2/3/4 的共同目标，是让学生从“知道一位导师的基本信息”过渡到“理解自己进组后最可能面对的当前研究领域”。三份文档不是并列报告，而是围绕同一条认知链逐步收敛：导师履历中的研究方向演化 → 当前申请方向的领域地图 → 相关论文在该地图上的位置 → 从本科基础进入该前沿的学习路径。

这三份文档的第一性原理是：学生缺的不是更多资料，而是一条从陌生到能理解导师当前研究的认知路径。阶段 2 要把当前领域讲清，阶段 3 要把导师当前研究内容和论文群内部逻辑讲清，阶段 4 要把前沿倒推成可学习路线。任何阶段如果只是在拼贴前文、堆检索结果或填模板，都不算完成。

V2 基线增加一个更强的默认假设：执行 AI 是小白，读者也是小白。执行 AI 不能靠当前聊天记忆补齐规则；读者也不能靠项目背景、其他导师材料或开发文档补齐理解。每份成品文档都必须自包含，开头说明本文解决什么问题、为什么要这样读、读完应获得什么能力；关键缩写和专业术语第一次出现时必须展开并简要解释。不能把“知其然也知其所以然”的认知框架只沉淀在 `00_材料导读.md` 或项目规则里。

项目执行遵循“从需求出发，而不是从技术出发”。凡是研究判断、学习路线、可视化、证据核查或 harness 设计拿捏不准，先调研官方文档、公开 skills、同类项目、论文或领域资料，再判断是否吸收。站在巨人肩膀上扬长避短，但不生搬硬套；不为了实现某个技术而引入技术。

长时迭代时必须防止“上下文越长、规则越散、方向越偏”。每轮迭代都从用户需求和当前成品痛点出发，先判断问题属于内容质量、证据可靠性、可读性、目录洁净、执行规则还是确定性验证；再决定改成品、改 skill/reference、改脚本、改测试或只记录待观察。不能为了显得工程化而增加新文件、新框架或新检查；只有能跨导师稳定复用、能降低幻觉或能提高学生理解质量的规则，才沉淀进 harness。

阶段 2（领域地图）先用短段落回顾导师整个人生和学术路径，让学生建立人物认知和时间框架；随后迅速收敛到导师当前单位、当前课题组最可能开展的研究领域。早期方向只作为当前方向的前史和能力来源处理，除非它仍直接延续到当前课题组。阶段 2 要回答：这个导师现在所在的方向属于哪个一级/二级/更细子领域，领域在解决什么问题，关键技术路线是什么，导师当前工作处在什么位置。

阶段 3（论文路线）用阶段 2 的领域框架进入导师自己的相关论文群。它不复述每篇论文摘要，也不要求把每篇论文都详细拆解，而是建立“研究对象—核心过程—方法平台—论文角色—相邻论文关系”的阅读坐标系。阶段 3 要回答：导师当前研究内容是什么，相关论文如何构成研究路线，学生拿到任意一篇当前相关论文时如何判断它处在这条路线中的什么位置。

阶段 4（学习向导）把前三段反向翻译成学习路径。它不是通用课程表，也不是教材正文，而是从高数、线代和本科物理出发，抽取读懂当前方向所需的知识点，串联成个性化小字典。阶段 4 要回答：学生如何从已知基础逐步进入导师当前前沿，哪些概念必须补，哪些高阶主题只需按需查。

#### 2.4.2 技术原理

阶段 1 的按履历分组是阶段 2/3/4 的关键输入。导师身份、单位、职位和合作网络变化，常常对应研究对象、实验平台、技术路线或问题意识的微调；即使方向没有显著变化，分开观察也能避免把多年工作揉成一个静态标签。对申请学生而言，越接近当前单位和当前课题组的工作，通常越能代表进组后可能接触的领域；早期阶段的意义在于解释当前方向的来源，而不是平均展开全部历史方向。

阶段 2 的领域地图必须以“当前方向”为中心，而不是写成大百科。写作时从大到小定位学科层级：一级学科、二级方向、更小子领域、关键技术路线、当前前沿。阶段 3 只在这个地图上深入导师当前相关论文群，不另起一个不相关的论文综述，也不退回阶段一式论文列表。阶段 4 只为这个地图、这些论文和学生能力跃迁服务，不写通用本科到研究生课程大纲。

证据强度要贯穿三段。来自官网、Google Scholar、OpenAlex、ORCID、DOI、论文页面、教材、讲义或综述的判断优先；单源论文、同名风险论文、来源缺失内容只能作为弱证据或标注“需人工复核”。来源缺失时写 `[未找到]`，不得为了补齐叙事捏造 URL。正文采用论文式上标引用，优先用 `<sup><a href="#o1">[O1]</a></sup>`、`<sup><a href="#p2">[P2]</a></sup>`、`<sup><a href="#r3">[R3]</a></sup>` 等可点击引用键，完整链接集中放在“参考文献与资料”，避免裸 URL 干扰阅读。

四份文档统一采用“正文论文式上标引用 + 文末参考文献表”的引用规范。正文原则上不放裸 URL，而使用 `<sup><a href="#p3">[P3]</a></sup>` 这类可点击上标引用标明资料类型并跳转到文末参考文献行；文末设置 `参考文献与资料`，按官方与身份资料、论文与数据资料、综述教材与课程资料、背景与辅助资料分组，并统一使用 `编号 / 文献或资料 / 支撑内容 / 链接 / 类型` 五列表格，其中 `编号` 列写成 `<a id="p3"></a>[P3]` 形式。该设计保留可追溯 URL，同时让正文渲染效果接近学术论文引用。它不是正式论文参考文献格式，而是面向学生和批量质检的学术 Markdown 格式。

批量化以后，论文题名错误会比链接失效更隐蔽：AI 容易把相邻论文、会议题名、截断题名或记忆中的机制写进来源表，而 URL 仍然看起来可追溯。因此阶段 2/3/4 增加 DOI 元数据门：对 `doi.org`、Nature article URL 等可还原 DOI 的来源，优先用 Crossref 核对题名；Crossref 查不到的高校仓储 DOI 再用 DataCite 兜底。这个检查不判断论文解释是否正确，只防止来源表题名和真实 DOI 元数据不一致。

V2 基线把反幻觉从“有链接”推进到“关键判断可审查”。成品文档保持可读，不把每个来源核查过程塞进正文；内部目录 `_internal/evidence/` 维护关键判断级证据核对表，覆盖身份锁定、领域定位、论文关系、学习路径、核心图和平台链路等会影响学生理解的判断。每条核对记录必须说明判断、来源、来源支撑了什么、证据强度，以及是否需要人工复核。

实现上采用 harness 化设计，而不只是 Skill 化。`SKILL.md` 只保留触发、路由、硬约束和第一性原理；阶段细节放在 `references/` 按需加载；确定性检查放在 `scripts/`；模板只固定骨架，不替代内容理解；输出目录、证据表、图源、测试和回归样本共同构成 harness。这样既让 AI 稳定执行，又方便用户直接审查中文规则。

#### 2.4.3 输出规格

`00_材料导读.md` 的建议结构：这套材料解决什么问题；建议阅读顺序；如何阅读引用和证据标记；起步讨论入口；文件定位；使用边界；参考文献与资料。建议阅读顺序必须先粗读 `01_基础画像.md`，再读 `02/03/04`，最后回看 `01_基础画像.md` 核查事实和来源风险。

`02_领域地图.md` 的建议结构：资料概览；导师路径速览；当前方向学科定位；领域怎样发展到当前问题；关键问题和技术路线；当前前沿；证据与待复核点；参考文献与资料。该文档可以提早期履历，但篇幅服务于解释当前方向，不平均介绍所有历史方向。

`03_论文路线.md` 的建议结构：资料概览；先抓住论文在回答什么问题；论文线怎样连成研究路线；当前主线论文；前史积累论文；旁支与弱证据；从论文路线倒推出的学习准备；参考文献与资料。该文档允许进入技术细节，但技术细节必须服务于理解导师当前研究内容和阶段 4 的学习路径。内部可以用对象、过程、方法、角色和路线关系做诊断，但正文不应先展示分类框架或总表口吻，而应先讲清论文群正在推进的问题链。

`04_学习向导.md` 的建议结构：资料概览；读者起点；终点能力；课程知识地图；进组前起步闭环；若干段由浅入深的问题链；回到阶段三论文路线；进组后先从哪里接上；卡住时如何判断该补什么；资源指针；参考文献与资料。默认读者只有高等数学、线性代数和普通物理基础，量子力学、光学、波动、谱学和具体前沿方向都可能不稳。它应从学生已学课程出发，说明每一步知识属于哪门课、为什么要补、补到什么程度、怎样连接到目标论文、核心图和进组问题；不能只是 `03_论文路线.md` 的改写。

五份成品文档都需要可视化理解构件，但不强制每处都用 Mermaid，也不强制每份都画流程图。可视化必须服务理解，不做装饰；一张图或一个表只回答一个问题。流程图只用于真实流程；领域定位优先层级树、定位表或小型 mindmap；论文路线优先“论文-问题-方法-角色”矩阵；学习向导优先“课程基础-要补概念-解锁论文/图-卡住时回退”桥接表；平台链路可用小型 Mermaid 或文本链路图。单个 Mermaid 图优先控制在 5-9 个节点，超过约 12 个节点、横向过长或字号被压小时，应拆成总览图 + 局部图，或改成表格/矩阵/层级树。

V2 中把可视化检查拆成两层：`verify_phase_docs.py` 检查正文是否存在可视化理解构件，并在 Mermaid 存在时检查图类型开头；`verify_mermaid_render.py` 只负责把真实存在的 Mermaid 代码块用本机 `mmdc` 或显式 `--use-npx` 渲染成 SVG。没有 Mermaid 时渲染门跳过；有 Mermaid 但没有渲染器时必须记录为未通过，不能把可视化构件 smoke 当成 Mermaid 可渲染证明。

排版可读性同样分两层处理。普通正文段落超过 420 字时，`verify_phase_docs.py` 直接失败；代码块、表格、标题、列表和文末参考文献区不纳入这个机械阈值，避免误伤证据表和来源表。这个阈值只拦截明显的信息堆叠，不能替代人工判断段落之间的递进、并列和因果关系是否真正清楚。

## 第4章: 使用最佳实践

本章写给使用者（你自己）。**纠正两点：AI 不能替代 session 管理，工具不是万能胶。**

### 4.1 session 分割原则

每轮 session 只做一类工作。不要在同一轮里同时"设计文档 + 写代码 + 修 bug + 整理目录"——做混了就是我在这个窗口犯的错。

推荐分法：

| session | 任务 | 时长参考 |
|:--------|:-----|:--------|
| 设计 | 写计划书章节、定 workflow | 30-60 分钟 |
| 编码 | 写脚本、测试 | 60-90 分钟 |
| 整理 | 归档、改目录结构 | 15-30 分钟 |
| 调研 | 搜索资料、读工具文档 | 30-45 分钟 |

### 4.2 checkpoint 机制

每完成一个独立任务，写一段 checkpoint 到 `计划/上下文交接.md`。格式很简单：

```
2026-06-30 checkpoint: 跑通 identity_resolver 的 OpenAlex 查询（有 bug 未修）
```

写完后 `/clear` 再开下一个任务。

### 4.3 什么时候该停

- 纠正同一件事两次后 → 不继续，直接写 checkpoint 然后 `/clear`
- AI 开始自行加文档/重构/提议 → 停下，明确说"只做 X，别做 Y"
- 超过 45 分钟 → 该 session 已经够长了，收尾 or `/compact`

### 4.4 `/compact` 的正确用法

```
/compact 保留以下决策：A.1-A.7（数据源体系和架构形态），附录 C（部署清单），2.2.1（阶段 0/1 workflow）。丢弃：过往对话历史、旧代码输出、调试记录。
```

不要在自动 compact 触发后才处理。自动 compact 会丢关键早期决策。

### A.6 架构形态 v2：Harness 设计哲学（已确认）

**Agent = Model + Harness**（Trivedy 2026）。本项目是 Harness，不是 Application。

传统软件工程试图在代码里覆盖所有边界情况，导致臃肿、难以适应突发状况。Harness 方式相反：把"智能"留给 AI Agent，把"结构"留给工具和规则。

| 维度 | Application 方式（不采用） | Harness 方式（采用） |
|:-----|:--------------------------|:-------------------|
| 错误处理 | 代码里 try/except 全覆盖 | 失败快、报错明、交给 AI 修复 |
| 编排逻辑 | 主流程脚本串联一切 | 每个 CLI 独立，AI 决定顺序和重试 |
| 校验 | 前置校验（gate）阻止错误 | 后置校验（verify）暴露错误供修复 |
| 状态 | 内存或 DB | JSON 文件（可观察、可恢复、可检查） |
| 扩展 | 加代码、改架构 | 加脚本、改配置 |

**Harness 三要素**：
1. **工具**（tools）：每个 step 是独立 CLI，入出都是 JSON。run.py 只是快捷方式，AI 可跳过它单独调用任何 step。
2. **规则**（rules）：AGENTS.md、research-advisor skill、pipeline.md 和 references 定义"怎么做"和"不能做什么"。
3. **评价标准**（eval）：verify_profile.py、verify_phase_docs.py、verify_source_metadata.py 和 verify_mermaid_render.py 是确定性 smoke。未通过 → AI 修 → 重验；通过也不等于学术质量最终通过。

**Loop 工作原理**：
```
Phase A → Phase B → C（渲染）→ verify 通过？→ 完成
                              ↓ 未通过
                      AI 诊断 → 修复 → 再渲染 → 回到 verify
```

**不做什么**（来自实践教训）：
- validate_career_stages.py 不做强门控（只警告，不阻止）。verify 才是门控。
- run.py 不做全能编排（出错就报，不自己处理降级）。AI 看到错误自行决定下一步。
- 不缓存中间结果（每次从头查 API，archive/ 只供人工溯源）。
- 不给每个脚本加 schema 校验（校验成本超过犯错成本，AI 检查输出更快）。

**为什么必须这样做**：产出是 Markdown 文档，不是软件。文档的质量标准（内容准确、格式规范、来源可溯）靠 AI 凭判断力保证，不是代码能硬编码的。代码只保证数据获取正确、格式不被破坏。两个层面分离，才能各自演进。

### A.7 自进化迭代协议（已确认）

V2 自进化迭代不是让 AI 无限润色，也不是不断添加工具。它是一套闭环：用户反馈或样本文档暴露问题 → 判断问题类型 → 定位归属 → 最小修改 → 验证 → 提交 → 更新交接。每一轮都必须能回答“这次改动让下一位完全小白的执行 AI 更稳定了吗？让完全小白的学生更容易理解了吗？”

问题归属按以下规则处理。设计原因和取舍写进 `docs/计划书.md`；当前状态、风险和下一步写进 `docs/上下文交接.md`；阶段执行细节写进 `.claude/skills/research-advisor/references/`；可机械判断的问题写进 `scripts/` 和测试；只影响单个导师表达的问题先改成品，不立刻升格成全局规则。反复出现、跨导师复现、且能被清楚描述的问题，才进入 harness。

迭代优先级固定为质量先于效率。第一层是事实和证据：身份、论文、领域定位、核心图、平台链路和学习路径判断必须可追溯。第二层是教学质量：每份文档自包含、术语首次解释、由浅入深、从学生已知课程接到前沿。第三层是排版和可视化：短段、表格、矩阵、层级树或小型 Mermaid 降低信息负担。第四层才是批量效率：模板、脚本和验证器帮助复刻质量，但不能替代研究判断。

每次拿捏不准都先外部调研。可调研对象包括官方文档、公开 agent skills、同类工具、学术论文、教材、课程讲义、综述和领域资料。调研结果只吸收能服务本项目第一性原理的部分；不因为某个工具先进、流行或复杂就引入它。新增工具必须说明用途、适用边界、验证方式和回退路径。

目标模式或长时间无人值守时，Agent 必须按里程碑工作。每个里程碑只解决一类问题，结束时运行相应验证、提交独立 commit、更新 `docs/上下文交接.md`，并明确剩余学术质量风险。若同一阻塞条件连续出现，或验证器通过但内容判断仍薄弱，必须记录风险并转入调研或请用户反馈，而不是用顺滑文字掩盖。
