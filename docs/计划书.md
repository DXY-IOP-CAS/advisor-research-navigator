## 第1章: 需求背景

### 1.1 起因 / 工具目标

每年硕士/博士生选导师是刚需。每年研一 3-4 月学生面临报志愿和套磁，核心痛点在四个环节。官网信息粗糙：只有论文列表和标题，没有科普级解读。有论文看不懂：论文要求研究生以上知识，学生只有本科基础。资源散乱无索引：论文、教材、视频各在各处。调研易绕圈：不知道该读什么、按什么顺序。

工具用四段嵌套解决这些问题。每一段的输出为后一段提供认知上下文，范围逐段收窄，理解逐段加深。工具同时承担三个递进职能：调研（全面收集信息）、综述（分析串联知识）、教学（设计递进学习路线）。

| 段 | 做什么 | 做法（高层次） | 目的 |
|:---|:---|:---|:---|
| 第1段 | 老师基础信息调研 | 地毯式收集官网、学术平台、中文百科、采访稿等多维信息，多源交叉比对 | 输出基础画像，明确领域范围 |
| 第2段 | 领域发展调研 | 基于第1段的领域范围，追溯子领域发展脉络和当前阶段 | 输出认知地图，理解位置 |
| 第3段 | 论文串联与定位 | 以第2段认知框架重新审视论文序列 | 标注每篇论文在领域树上的位置 |
| 第4段 | 从零到前沿讲义 | 从高数+线代起步反向设计学习路线 | 输出递进式讲义（小字典） |

嵌套逻辑：大领域一>小领域一>老师定位一>学习路径。后一段建在前一段的认知之上。

技术实现细节在第2章和附录 A。

### 1.2 目标用户画像

工具的目标用户是大二/大三物理本科生，正在准备申请硕博选导师。核心场景是研一 3-4 月报志愿时，用工具判断与导师研究方向的匹配度。

工具假设用户从高数和线代起步。实际用户可能已有四大力学、固体物理等基础，但所有高阶知识（复变、数理方法、高量、群论、量子多体）本质上是高数和线代的逐步推广。从根推导才能零跳步建立真理解。

教材在 AI 时代不再是逐页通读的课本，而是字典。本科到研一的教材体系分三个层面：

| 层面 | 课程 | 定位 |
|:---|:---|:----|
| 数学基础 | 高数、线代、概率论、复变、数理方法 | 推导起点 |
| 物理基础 | 普物、四大力学、固体物理 | 物理直觉来源 |
| 高阶延伸 | 高量、群论、量子多体、量子场论、规范场论 | 大字典边界—领域触及的范围 |

教材从课本转向字典，有两个原因。第一，教材面向通用人才培养，覆盖完整体系，但具体方向只用其中部分知识点。逐页通读在无关内容上浪费大量时间。第二，学生可在 AI 辅助下按需读取字典条目，带着研究方向去学，既能深化具体知识点理解，又能强化整体框架认知。

第4段产出的讲义是个性化小字典。它从教材体系（大字典）中只提取第1-3段调研确认需要的知识点，串联成一条从高数+线代到前沿的逻辑链。小字典扎根于大字典，以教材体系为参照系，约束 AI 创作边界，防止偏离轨道。

一个可扩展的方向：工具可附带教用户如何配合 AI 递进学习。AI 的优势是能根据用户水平动态适配，但多数人不擅长用 AI 辅助学习。加入"如何用 AI 学"的引导可降低门槛。此方向未确定，作为后续讨论参考。

### 1.3 工具差异化与定位

18+ 同类工具调研后确认：市场无直接竞品。现有工具分三类，各占一个不重叠的位置。

| 类别 | 代表工具 | 做什么 | 与我们的重叠 |
|:-----|:---------|:-------|:-----------:|
| 导师评价 | 导师评价网、GradChoice、advisor-calculator | 口碑评分（人品、管理、毕业） | 不重叠——我们不碰评价 |
| 导师匹配 | PhdFit、StreamlinedAI、PhDuo | 从学生 CV 推荐导师 | 不重叠——我们不做推荐 |
| 学者画像 | supervisor、SciAtlas、ai-talent-graph | 论文+主页一>信息聚合报告 | 第1段部分重叠 |

学者画像工具停在"导师长什么样"的层面。我们继续往前推到领域发展、论文定位和教学讲义。

工具的准确定位包含四层含义。第一，通用模板引擎：对任何机构的任何老师适用，跨老师复用零边际成本，首个实例为张鹏举（IPHY 超快物质科学中心）。第二，开源方法论：Claude Code Skill + `src/phase1/` 脚本 + Markdown 模板形态，不是网站/App，零部署。第三，从调研走到教学：调研+综述+教学三职能递进。第四，调研驱动设计：每项决策经同类工具验证，不凭空发明。

一条根本差异区分了我们和所有竞品：同类工具帮学生找谁合适（匹配引擎）或知道老师什么样（画像工具）；我们帮学生学懂一个方向。

工具六不做：不写教材正文；不替代主动学习；不点评老师或推荐方向；不捏造论文/教材链接；不评价导师口碑；不搭建网站。

**技术落地细节见** `src/phase1/pipeline.md`（执行步骤、接口格式、错误处理）。本章聚焦设计原理和缘由，不重复技术细节。

### 2.1 整体工作流

用户提供导师姓名 + 机构 + 官网 URL，工具执行四份文档流水线。每份文档的认知上下文来自前一份，深度逐段加深。

**流水线逻辑**：

```
阶段1：信息检索 → 论文按学术履历阶段分组 → 结构化画像 + 完整论文列表
    ↓
阶段2：领域脉络调研 → 子领域发展树、里程碑、当前前沿
    ↓（认知回环，非单向）
阶段3：以领域框架重审论文 → 标注每篇论文在领域树上的位置
    ↓
阶段4：从高数到前沿的递进学习路径
```

阶段2建立的领域认知，反过来让阶段3的论文串联更有深度——不是简单的时间线排列，而是在"老师做了什么研究"的基础上讲清"为什么做、为什么这么做"。四份文档每一份都服务于下一份，阶段3回馈阶段2的认知积累形成闭环。

**阶段一览**：

| 阶段 | 输入 | 产出 | 核心任务 | 人闸 |
|:-----|:-----|:-----|:---------|:----:|
| 1 基础画像 | 姓名+机构+官网URL | `01_基础画像.md` | 官网提取身份信息 → GS 搜论文列表 → OA 补元数据 → arXiv 预印本 → 同名过滤 → 按履历分组 → 9 节画像 | 无（AI 自主） |
| 2 领域脉络 | 画像+论文列表 | `02_领域脉络.md` | 追溯子领域发展树和里程碑 | TBD |
| 3 论文串联 | 领域框架 | `03_论文定位.md` | 每篇论文在领域树上的位置标注 | TBD |
| 4 学习讲义 | 前三段全部产出 | `04_学习讲义.md` | 从高数+线代到前沿的递进学习路径 | TBD |

**四条设计原则**贯穿全部阶段：
1. 来源必标 URL：每条外部信息末尾带来源链接
2. 缺失优于编造：找不到的字段标 `[未找到]`，不捏造
3. 双源才标已验证：单一来源的信息标注"仅X来源"
4. 冲突列出不仲裁：不同源数据矛盾时并列写出，你判断

**第五条原则（硬约束）—— 存档不是缓存，每次必须全新运行**：
archive/ 目录下的中间文件仅用于错误回溯和结果复现。每次运行必须从头查询所有 API（GS / OpenAlex / arXiv），不得从历史存档中加载数据充当缓存。违反此规则会导致：旧数据静默覆盖新结果、发现错误后无法通过重新运行修复。宁可 API 限速等待，不可偷懒读缓存。

#### 2.1.1 工作流程

用户提供导师姓名 + 机构 + 官网 URL 后，工具按四份文档流水线逐段执行。每份文档的认知上下文来自前一份，深度逐段加深：阶段 1 收集全部论文和学术信息 → 阶段 2 梳理领域发展脉络 → 阶段 3 在领域树上定位每篇论文 → 阶段 4 设计从高数到前沿的学习路径。

#### 2.1.2 技术原理

**数据收集架构**（v3.3 修订：替代 v3.2 的自适应两步走）

论文采集采用 GS 为主、OpenAlex 为辅的串联策略。核心判断：**GS 是学者本人维护的论文列表，OpenAlex 是机器聚类的论文数据库。对于任何学者，GS profile 的论文完整性远高于 OpenAlex。**

```
Step 1: Google Scholar 获取论文列表
  → 找 GS profile URL（Web Search 发现 / 用户提供）
  → 爬取全部论文（标题 + 年份 + 引用数）
  → 55 篇（张鹏举）/ 60 篇（李自翔）vs OpenAlex 的 14/13 篇
  
Step 2: OpenAlex 元数据补充
  → 用每篇论文标题搜索 OpenAlex
  → 补 DOI、期刊名、作者列表、引用数
  → 不依赖于 OpenAlex 的 works_count（覆盖不完整是常态）

Step 3: 同名干扰过滤
  → 合著者网络 + 期刊范围 + 领域关键词三筛
  → 仅在 GS 搜索结果明显包含多领域论文时触发（如"张鹏举"案例）
  → 名字独特时（如"Zi-Xiang Li"）可直接跳过

Step 4: arXiv 补充预印本
  → 补 GS 可能遗漏的最新预印本
  → 需同名干扰过滤
```

**为什么 GS 永远优于 OpenAlex 做论文列表主源**（v3.3 核心认识变更）：

| 对比维度 | Google Scholar | OpenAlex |
|:---------|:---------------|:---------|
| 论文列表来源 | 学者本人维护 | ML 算法聚类 |
| 对中文作者覆盖 | 完整（学者自己添加） | 不完整（22-38%） |
| 元数据（DOI/期刊/引用） | 无（只有标题+年份+引用数） | 完整 |
| 同名干扰 | 学者用邮箱验证，无干扰 | 需要论文标题比对消歧 |
| API | 无（需爬取 HTML） | 完整 REST API |
| 限速风险 | CAPTCHA 触发 | 100K/天免费 |

结论：**GS 做主源（论文列表完整性），OA 做辅助（元数据丰富性）。** 两个源各取所长。

**GS 不可用时的降级策略**。如果学者未维护 GS profile（或 GS 页面被 CAPTCHA 封锁），则：
1. 恢复 OpenAlex 做主源（原 v3.2 的路径 A）
2. arXiv 按姓名搜预印本
3. 告知用户 GS profile 的重要性，请用户补充链接

**同名过滤逻辑**（v3.4 修正：替代 v3.3 的"合著者白名单硬排除"）。

GS profile 绑定了机构邮箱（如 @iphy.ac.cn）时，该 profile 内所有论文视为已验证属于该学者，不执行同名过滤——学者的职业阶段转换（如从碰撞物理转向阿秒物理）是正常职业发展，不意味着是不同人。

仅在 GS profile 无法验证机构邮箱时，使用多维评分做辅助判断：

```
合著者重叠（+2 分）  ：与已知论文共享合著者
机构匹配（+1 分）    ：论文中的机构与履历一致
期刊范围（+1 分）    ：论文发表在学科主流期刊
领域关键词（+1 分）  ：论文标题含研究方向关键词

总分 ≥ 3 → 保留（可信）
总分 < 3 → 标记"疑似同名干扰"
```

**广域搜索与深度检索的分工**（搜索策略的核心架构）

两条搜索链不是互相验证的关系，而是**发现 vs 确认**的分工：

```
广域搜索（Step 2）— 只做"发现"
  输入：姓名 + 机构
  搜索范围：百度百科、新闻、知乎、采访、实验室主页、公开 profile 链接
  产出：候选链接列表（GS/ORCID/ResearchGate 链接 + 公众报道）
  核心原则：不判断真伪，只负责找到。找到的链接是否属于目标学者、
  事实是否准确——这些不由广域搜索判断。

深度检索（Step 3-7）— 只做"确认"
  输入：官网 email（Step 1 提取）
  锚点：官网 email → GS profile 邮箱校验
  → 一旦邮箱匹配，该 GS profile 内的所有论文视为已确认身份
  → OA/arXiv/S2 的数据围绕这个已确认的身份补充
  核心原则：不负责发现新信息，只负责在已确认身份的基础上做深挖。

交叉验证（渲染时）— 筛选广域结果
  用已确认的身份信息（name + email + institution + GS ID）去过滤
  广域搜索的发现结果：
  → 匹配此身份的新闻报道/采访 → 纳入画像的"公开信息"节
  → 提到同名不同人的结果 → 丢弃
  → 来源可疑且无法与身份交叉验证的 → 标记"未验证"
```

**不需要"互相验证"的理由**。广域搜的任务是"发现有什么"，深度搜的任务是"确认是谁"。两者依赖同一个锚点——官网 email。这就够了。广域搜找到的 GS 链接用官网 email 校验，通过的就是这个人的，不通过的就是同名其他人的。不存在"广域需要深度才能验证"的矛盾。官网 email 就是那个打破循环的第一信任锚点。

| 数据源 | 覆盖范围 | 对中文作者的实际覆盖 |
|:-------|:---------|:-------------------|
| OpenAlex（充足时） | 316M 作品 | 英语论文 90%+，中文作者论文 50-80% |

#### 2.2.2 技术原理

**画像 9 节结构**：

画像由 9 个节组成：身份标识（姓名/职称/机构/邮箱/各平台 ID，来源官网+API）、学术履历（教育→工作含导师/机构/起止年份，来源 ORCID+官网+Web Search）、研究方向（总体方向+具体方向中英文关键词，来源官网+论文自动提取）、全部论文按履历分组（每阶段一组，每篇含标题/期刊/年份/引用数/角色/TLDR，来源 OpenAlex+S2+arXiv+GS 降级）、影响力指标（总引用/总论文/h-index，来源 OpenAlex 或 GS）、合作网络（博士导师/高频合作者/机构，来源 OpenAlex 论文共著 + DOI 反查）、公开信息（新闻/采访/讲座/B站/知乎，来源 Web Search）、技术关键词（实验技术/理论工具/研究对象/应用方向，来源论文关键词）、来源说明（各数据源状态成功/失败/未启用）。

**GS 主源采集算法**（v4.0 技术架构更新）：

```
Step 1: scholarly 获取 GS 论文列表（一行调用，替代旧版 gs_scraper.py）
  → pip install scholarly
  → from scholarly import scholarly
  → author = scholarly.search_author_id(gs_id)
  → filled = scholarly.fill(author, sections=["publications", "indices"])
  → 输出：56 篇论文 + h-index + 引用数 + email_domain（身份金标准）
  → GS 的 email_domain 匹配官网域名即确认身份，不执行额外身份验证

Step 2: OpenAlex API 获取论文列表 + 元数据（step3_openalex.py）
  → /authors/{oa_id} → profile：h-index、ORCID、机构
  → /works?filter=authorships.author.id:{id} → 全部论文
  → 补 DOI、期刊名、作者列表
  → 输出格式与 GS 统一（source: "openalex"）

Step 3: 同名干扰过滤
  在合并步骤（step6_merge.py）自动处理。不需要单独的过滤步骤。
  合并时 P0:DOI → P1:arXiv ID → P2:归一化标题。
  OA 数据在合并中补充 GS 缺失的元数据，不改变 GS 论文列表完整性。
  arXiv 噪声通过 DOI/标题匹配 GS或OA已确认的论文来自动过滤。

Step 4: arXiv 补充（预印本时效）
  au:姓名拼音 → 最新预印本 → 与 GS/OA 论文合并去重
  → 同名干扰在合并时通过标题匹配自动过滤

Step 5: 合并与去重（step6_merge.py）
  教授信息按字段优先级合并（GS 占 h-index/i10-index/引用，OA占DOI/ORCID）
  论文去重 P0:DOI → P1:arXiv ID → P2:归一化标题
  引用数优先 OpenAlex（日更新），GS 引用数做参考
```

**同名干扰过滤方法**（GS 降级时的必要步骤）：

```
合著者过滤：
  取官网已确认论文中的合著者列表作为白名单。
  候选论文至少与 1 位白名单合著者共著。零共著标"疑似同名"。

期刊范围过滤：
  按学科预设主流期刊列表。跨学科论文需合著者过滤通过。
  不在期刊列表中的论文标"期刊存疑"。

领域关键词过滤：
  研究方向关键词正匹配。关键词命中数为 0 的论文标"主题存疑"。
  
汇总：三筛全通过 → 可信高；仅通过合著者筛 → 可信中；
      仅通过关键词筛 → 可信低；零通过 → 作为同名干扰排除。
```

**合并去重规则**。三条优先级：P0 DOI 完全相同；P1 arXiv ID 相同（版本无关）；P2 归一化标题子串匹配。

**合并后字段选取**。引用数优先 OpenAlex（日更新），S2 备选，GS 备选。摘要/TLDR 优先 S2。发表日期优先 OpenAlex，arXiv 版本日期备选，GS 年份备选。作者列表优先 OpenAlex，S2 备选。期刊名优先 OpenAlex，S2 venue 备选。DOI 优先 OpenAlex，CrossRef 兜底。

**共证可信度标记**：一篇论文在 N 个源中出现 → 标记 N/4。从 GS 独有、未被其他源验证的论文标记 "仅 GS 来源"。

**覆盖预期（修正版，v3.2）**：

```
OpenAlex works_count ≥ 20 时:
  覆盖预期与之前一致：多源合并后 90-95%+
  
OpenAlex works_count < 20 时（中文作者常见）:
  GS + OpenAlex 合并后预期捕获该学者 GS profile 中的全部论文
  （但 GS profile 需要学者自己维护，非所有人都更新）
  仍可能缺失：学者未维护 GS profile 时、近 1-2 个月新发论文
  
  无论哪种情况，画像首段标注："论文来自 {源}。OpenAlex 对中文作者
  覆盖不完整（Zheng et al. 2025），如有遗漏请补充 GS/ORCID 链接。"
```

**原创性声明**（巨人给的 vs 我们造的）。巨人给的：论文骨架（标题/DOI/引用/期刊/年份）来自 OpenAlex API，论文内容理解（TLDR）来自 S2 API，预印本发现来自 arXiv API，履历结构来自 ORCID API。我们造的：学科关键词字典（config/sources.json，本地维护），多源合并编排（AI 决定何时查哪个、降级策略），身份消歧（官网 ground truth + 命中率规则），职业阶段分组（AI 解析履历分配论文期），渲染输出（Markdown 结构化+来源标URL）。

**论文分组逻辑**：

```
解析 ORCID 履历或官网 CV 中的起止年份：
  - 博士阶段: start_phd ~ end_phd
  - 博后阶段: start_postdoc ~ end_postdoc
  - 独立阶段: start_independent ~ present

每篇论文的发表年份落入哪个时间段就归入哪组。
如无精确年份，按机构 affiliation 推断。
跨阶段工作（如博士工作延至博后才发表）归属于实际完成阶段。
```

## 第4章: 使用最佳实践

本章写给使用者（你自己）。**纠正两点：AI 不能替代 session 管理，工具不是万能胶。**

### 4.1 session 分割原则

每轮 session 只做一类工作。不要在同一轮里同时"设计文档 + 写代码 + 修 bug + 整理目录"——做混了就是我在这个窗口犯的错。

推荐分法：

| session | 任务 | 时长参考 |
|:--------|:-----|:--------|
| 设计 | 写计划书章节、定 workflow | 30-60 分钟 |
| 编码 | 写脚本、测试 | 60-90 分钟 |
| 整理 | 归档、改目录结构 | 15-30 分钟 |
| 调研 | 搜索资料、读工具文档 | 30-45 分钟 |

### 4.2 checkpoint 机制

每完成一个独立任务，写一段 checkpoint 到 `计划/上下文交接.md`。格式很简单：

```
2026-06-30 checkpoint: 跑通 identity_resolver 的 OpenAlex 查询（有 bug 未修）
```

写完后 `/clear` 再开下一个任务。

### 4.3 什么时候该停

- 纠正同一件事两次后 → 不继续，直接写 checkpoint 然后 `/clear`
- AI 开始自行加文档/重构/提议 → 停下，明确说"只做 X，别做 Y"
- 超过 45 分钟 → 该 session 已经够长了，收尾 or `/compact`

### 4.4 `/compact` 的正确用法

```
/compact 保留以下决策：A.1-A.7（数据源体系和架构形态），附录 C（部署清单），2.2.1（阶段 0/1 workflow）。丢弃：过往对话历史、旧代码输出、调试记录。
```

不要在自动 compact 触发后才处理。自动 compact 会丢关键早期决策。

### A.1 整体方向（已确认）

**A.1.1 信息源不是单一的**。早期讨论陷入"OpenAlex 是主源，其他补充"的死胡同。修正：物理学科按学科分层数据源优先级。

**A.1.2 身份消歧不走纯 ML**。OpenAlex 等 ML 聚类对中文名+跨机构不可靠（实测张鹏举 25 个同名候选）。我们的方法：官网论文做 ground truth，论文标题直接比对。

**A.1.3 必须有人闸**。身份消歧不允许全自动。任何 ID 锁定前用户确认，避免连锁错误。

**A.1.4 不报错退出**。所有 API/数据源失败时降级到其他源，不抛异常。

### A.2 数据源体系（已确认）

按学科+ID 类型分层：

**跨学科通用 ID 体系**：
- ORCID（by email，注册过就有，最可靠）
- OpenAlex Author ID（ML 聚类，覆盖广但中文不准）
- Semantic Scholar Author ID（ML 聚类，覆盖广，TLDR 资源）
- ArXiv Author ID（用户主动 claim 后生成，物理/数学/CS 常见）
- Google Scholar Profile（用户自建，无官方 API）
- CrossRef

**物理学科专用**：
- **INSPIRE-HEP** — 高能物理/粒子物理/核物理领域事实标准，API 全
- **NASA ADS** — 天体物理事实标准，需 API token

**网页抓取层**：
- 机构官网（HTTP fetch）
- Web Search（Serper/Exa/Tavily）
- 百度百科/知乎/B站（Web Search）

**不引入**：Scopus、Web of Science（付费）；CNKI/万方（需授权）；DBLP（CS 专用）；Neo4j（重）。

### C.5 数据流向（阶段 1）

```
用户输入 (姓名/机构/官网URL)
       │
       ▼
阶段 A：AI 广域搜索 + 身份确认
  MCP Fetch 读官网 → 提取 email/名称/研究方向
  MCP Serper/Exa 搜 GS/ORCID/RG 链接
  AI 交叉验证 → verified_ids.json
       │
       ▼
阶段 B：脚本数据获取
  python src/phase1/step2_gs.py {gs_id} → gs.json
  python src/phase1/step3_openalex.py {oa_id} → oa.json
  python src/phase1/step5_arxiv.py {拼音} → arxiv.json
  python src/phase1/step6_merge.py gs.json oa.json arxiv.json → merged.json
       │
       ▼
阶段 C：AI 渲染
  读 merged.json
  写 output/<机构>/<部门>/<姓名>/01_基础画像.md
  中间产物存入 archive/<timestamp>/
```
