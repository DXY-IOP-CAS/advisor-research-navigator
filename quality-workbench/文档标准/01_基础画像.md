## 文档职责

`01_基础画像.md` 是事实底座。它回答“老师是谁、资料是否可靠、履历和论文集合是什么”。

它只保留事实底座的职责，不写成传记、领域综述、论文路线或导师评价。学生先在这里建立一套以后能回看的事实判断框架：

- 身份是否锁定。
- 当前机构、职务和履历阶段是什么。
- 论文集合从哪些数据库来，覆盖和噪声风险在哪里。
- 哪些判断来自直接证据，哪些只是交叉证据、解释性推断或需人工复核。
- 后续读 `02/03/04` 时，应该怎样回到 `01` 检查自己有没有把早期方向、旁支论文或弱证据误当成当前主线。

`01` 的阅读方式应是“两次阅读”：

1. 第一遍粗读：只抓身份、履历阶段、论文集合和来源风险，形成导师事实轮廓。
2. 读完 `02/03/04` 后回看：用已经建立的领域、论文和学习路径理解，反过来核查事实底座是否支撑当前主线。

### 整篇读法

`01` 可以按“先抓轮廓，再查明细”的方式读。

第一遍先看前两节：`画像定位`、`资料与身份核查`。学生先知道这份文档怎么看、资料大致稳不稳、身份和论文集合能不能合并阅读，以及后文要带着哪些事实疑问往下读。

中段再查导师信息：`学术履历`、`方向入口`、`论文集合`、`合作与公开信息`。这些小节负责把前面的资料判断落到履历、论文和公开线索上。读完主体后先做 `主体自检`，判断自己能不能带着事实底座进入 `02`。

后两节是进一步核查区：`数据库指标说明` 说明数据库数字和元数据该怎样读，`来源与参考资料` 给出关键事实的回查入口。第一次阅读时如果主体已经清楚，可以先进入 `02`；遇到论文归属、数据库口径或来源强弱疑问时，再回到这两节细查。

读完 `02/03/04` 后再回看 `01`：检查当前主线、目标论文和学习路径，是否真的被身份、履历、论文集合和来源证据支撑。

## 通用设计原则

这一层只放所有小节共用的方法，先不绑定到某一节。以后统一回收时，可以把它们拆进写作规则或审查清单；每一节的具体写法放到“小节设计”里。

### 入口、资料核查、主体明细与核查区

`01` 应采用“入口 + 资料核查 + 主体明细 + 核查区”的结构：

1. 入口让学生知道这份文档怎么看：`画像定位`。
2. 资料核查让学生先判断资料和身份是否稳：`资料与身份核查`。
3. 主体明细让学生了解导师事实轮廓：学术履历、方向入口、论文集合、合作公开信息。
4. 主体自检收束主体阅读，判断学生能不能带着事实底座进入 `02`。
5. 核查区服务细查：数据库指标说明和来源参考放在主体之后。

这样做的原因是：学生第一遍不能被长论文表压垮，也不能在身份和来源没看清时直接进入导师信息。入口负责降低进入门槛，资料核查负责建立资料可信度，主体明细负责形成导师事实轮廓，核查区负责在读者需要时提供数据库和来源复核。

### 事实与解释分层

每个判断都要区分：

- 事实：官网、论文、数据库直接给出的信息。
- 交叉证据：多个来源共同支持的身份、论文归属或阶段判断。
- 解释性推断：AI 根据履历和论文集合总结出的方向入口判断。
- 风险：来源缺失、同名噪声、数据库不一致、单源论文、弱推断。

正文不能把解释性推断写得像直接事实。需要时用“从当前资料看”“可作为初步线索”“需人工复核”这类表达标出证据边界。

### 第一遍与回看

`01` 的阅读目标分两次：

- 第一遍：建立导师身份、履历阶段、论文集合、公开交叉线索和主要风险的粗轮廓；数据库与来源细节可在需要时再查。
- 回看：读完 `02/03/04` 后，再检查当前主线是否真的被身份、履历、论文集合和来源支撑。

因此正文既要照顾“第一次读的人”，也要留下“以后回来查证的人”需要的结构。长表格可以存在，但前面必须有读法提示。

### 表格与可视化

`01` 不优先使用复杂 Mermaid。它更适合短表、证据链表和文本链路，因为这里的核心是事实分层和证据边界，不是机制过程。

表格要克制使用。表格能帮执行 AI 固定字段，但最终成品要先服务学生阅读；如果每一节都用表，学生会觉得自己在读表单，难以形成一条自然的理解线。

适合 `01` 的视觉构件：

- 资料状态表。
- 身份证据链表。
- 履历阶段表。
- 论文集合读法提示。
- 公开交叉佐证表。
- 数据口径表。
- 来源表。
- 主体自检表。

每张表必须回答一个具体阅读问题。入口、承接、解释原因和收束边界优先用短段落；如果表格只是把正文切成格子，就改回短段或短列表。

成品试写时按以下规则收缩：

- 每个小节默认最多一张主表。
- 超过 5 列的表要警惕，学生横向读起来会累。
- 超过 6 行的表前必须有读法提示。
- 同一小节如果需要两张表，必须服务两个不同动作，例如 `资料与身份核查` 中的“资料状态”和“身份证据链”。
- workbench 里可以保留表格骨架；最终成品要按阅读节奏决定哪些表格转成自然语言。

如果确实需要图，只使用小型文本链路，例如“官方身份 -> 作者档案 -> 论文集合 -> 履历阶段 -> 后续 `02/03/04`”。不要把简单顺序画成横向很长的流程图。

### 术语与缩写

第一次出现的机构缩写、领域缩写和专业术语必须展开。展开方式要短，不把 `01` 写成百科：

- 机构：日本理化学研究所（RIKEN）。
- 领域：原子、分子与光学物理（Atomic, Molecular and Optical Physics, AMO）。
- 方法：高次谐波产生（High Harmonic Generation, HHG）——用强激光产生极紫外或阿秒脉冲的一类技术。

后续可以使用缩写，但不能假设学生已经从其他文档知道含义。

<a id="section-design"></a>
## 小节设计

本节逐节细化。大纲总览看整篇是否顺，通用原则看各节共用什么方法；这里看每一节自己要解决什么、正文怎么走、图表怎么用、读完检查什么、最容易写偏在哪里。这样后续转成模板、skill 规则或人工审查清单时，不需要再从聊天记录里捞判断。

每个小节尽量使用同一套内部检查项：

- 要解决的问题：这一节为什么存在。
- 正文怎么写：正文按什么顺序展开。
- 图表怎么用：是否需要表格、层级树、文本链路或 Mermaid。
- 读完检查：读者读完后应能说出什么或做什么。
- 容易写偏的地方：执行 AI 最可能跑偏的位置。

### 画像定位

要解决的问题：让学生拿到 `01` 后先知道这份文档怎么用。这里像开头段，不承担核查细节；它只告诉学生：先确认理解对象和资料边界，再去看导师履历、论文集合和后续 `02/03/04`。

本节要补足三层进入感，但都要短：

- 这份文档管什么：老师是谁、资料是否可靠、履历和论文集合是什么。
- 第一遍怎么读：先抓身份、阶段、论文集合和风险，不急着精读论文。
- 以后怎么回看：读完 `02/03/04` 后，回来检查当前主线、目标论文和学习路径是否被事实支撑。

正文怎么写：

1. 承接 `00`：学生已经知道五份文档要按认知阶梯阅读，现在进入第一份具体内容。
2. 说明 `01` 的任务：先建立事实底座，不评价导师，也不完整解释研究方向。
3. 给出第一遍读法：先看资料和身份是否稳，再看履历、方向入口和论文集合。
4. 给出回看读法：读完后续文档后，用 `01` 反查事实支撑和风险。

图表怎么用：默认不用图表。开头段的任务是定读法；一上来放风险表，读者容易感觉还没进入正文就被方法论拦住。确实需要视觉提示时，可以用一句文本链路：

```text
先知道怎么看 -> 再看资料是否稳 -> 再看导师履历和论文集合 -> 后续回看核查
```

读完检查：读者应能说出：“我读 `01` 是为了确认理解对象和资料边界。第一遍我只抓身份、阶段、论文集合和风险；读完后续文档后，我还要回来核查当前主线是否被这些事实支撑。”

容易写偏的地方：

- 把开头写成导师传记。
- 一上来堆专业方向。
- 只写“不评价导师”，却没有说明正面用途。
- 忘记说明读完 `02/03/04` 后要回看 `01`。
- 只说“本文是事实底座”，但没有解释不做事实底座会导致哪些具体误判。
- 让学生误以为第一遍必须读完整个论文长表。
- 把资料状态、身份证据、学术履历、论文集合或来源参考的具体方法提前讲完。
- 把项目内部词如 Fact Pack、Cognitive Blueprint 直接写给学生看。

### 资料与身份核查

要解决的问题：给学生一把进入正文前的“秤”。这一节先看资料来源、核查时间和风险，再看官网身份、作者档案和论文集合能不能合并阅读。读者读完后，心里要知道哪些事实可以暂时使用，哪些地方后文要保守。

它承接 `画像定位`。上一节只告诉学生这份文档怎么看；这一节开始检查资料本身是否站得住。它完成后，后面的 `学术履历`、`方向入口` 和 `论文集合` 才有可靠入口。

正文怎么写：

1. 开头先说明本节功能：先看资料状态，再看身份证据，最后给后文使用边界。
2. 给出资料状态表：列当前身份、官方入口、核查时间、论文集合、履历阶段、数据源覆盖和主要风险。
3. 给出身份证据链表：分开官方身份、联系方式/ID、作者档案、论文指纹和同名排除。
4. 写自然语言身份结论：例如“较稳：官网身份、作者档案和主要论文指纹互相支撑”；证据弱时写“需人工复核”。
5. 写后文使用边界：哪些事实可暂用，哪些论文、档案或指标要保守处理；接下来读 `学术履历` 和 `论文集合` 时先看什么。

图表怎么用：使用两张短表，分别完成两个动作。第一张是资料状态表，回答“这份材料现在是什么状态”；第二张是身份证据链表，回答“这些材料为什么可以放在同一位老师名下阅读”。两张表可以在同一节里连续出现，但不要互相替代。

资料状态表骨架：

| 项目 | 当前状态 | 读者怎么看 |
|:---|:---|:---|
| 当前身份 | <中文名>（<英文名或 [未找到]>），<当前机构 / 部门> | 先确认后文讨论的是同一位老师 |
| 官方入口 | <官网 URL> | 身份、职务和官方信息优先回官网核查 |
| 核查时间 | <YYYY-MM-DD 或本轮生成日期> | 公开信息会变化，隔一段时间要重新检索 |
| 论文集合 | <论文数> 篇，来自 <GS/OA/arXiv 等> | 第一遍只看规模、年份和阶段分布 |
| 履历阶段 | <阶段数> 个阶段 | 后文用它理解论文分组和方向演变 |
| 数据源覆盖 | <哪些源可用，哪些源缺失> | 知道哪些判断证据较稳，哪些要复核 |
| 主要风险 | <单源论文 / 同名噪声 / 来源缺失 / 收录延迟等> | 后文遇到相关判断时保留疑问 |
| 下一步读法 | 进入 `学术履历` | 带着资料风险看履历阶段和论文集合 |

身份证据链表骨架：

| 核查点 | 主要来源 | 支撑什么 | 风险处理 |
|:---|:---|:---|:---|
| 官方身份 | 官网、学院页、实验室页 | 当前机构、职位、姓名写法 | 来源缺失时标 `[未找到]`，不补写 |
| 联系方式与 ID | 邮箱域名、ORCID、个人主页链接 | 官网身份和作者档案是否能连上 | 邮箱或 ID 不匹配时标 `需人工复核` |
| 作者档案 | Google Scholar、OpenAlex、ORCID | 论文集合是否指向同一作者 | 数据库消歧可疑时保守使用 |
| 论文指纹 | 题名主题、合作者、机构署名、年份连续性 | 单篇论文是否应归入本文论文集合 | 单源或弱匹配论文不直接当强证据 |
| 同名排除 | 同名作者页面、明显不同机构或方向的论文 | 哪些资料不能混进来 | 记录排除原因，必要时留给人工复核 |

“论文指纹”第一次出现时必须解释。可以写成：“论文指纹指一组可核对线索，例如题名主题、共同作者、机构署名和发表年份。单个线索不够稳，多个线索一致时才更可信。”

不要把引用数、h-index、近三年引用等指标放进本节。这些数字属于 `数据库指标说明`。如果放在这里，学生容易把资料状态误读成导师评价。

本节的结论只到“这批材料能否暂时放在同一位老师名下阅读”。论文长表怎样扫，交给 `论文集合`；数据库数字为什么不同，交给 `数据库指标说明`；每条来源支撑什么，交给 `来源与参考资料`。

读完检查：读者读完这一节，不需要理解老师做什么研究，但应能说出四句话：官方入口在哪里；身份和论文集合大致稳不稳；主要风险是什么；接下来读 `学术履历` 和 `论文集合` 时先看什么。

容易写偏的地方：

- 把本节写成导师摘要或导师简介，提前总结研究方向。
- 只列字段，不告诉读者每一项怎么用。
- 只列数据源名称，不写覆盖状态和风险。
- 只列一堆链接，不解释它们怎样共同支撑身份。
- 把 GS 或 OpenAlex 搜索结果默认当本人。
- 不写同名和单源风险。
- 把 `T1/T2/T3/T4` 直接写给学生，却没有自然语言解释。
- 只给“身份已锁定”结论，不说明后文怎样使用或限制这些证据。
- 在这里塞入过多引用指标，和后面的 `数据库指标说明` 重复。

### 学术履历

要解决的问题：把导师经历整理成后文可用的阶段索引。学生读完这一节后，应知道老师大致经历了哪些阶段、每个阶段留下了哪些研究线索，以及为什么后面的论文集合要按阶段看。

它承接 `资料与身份核查`。上一节已经说明资料和身份能信到什么程度；这一节开始进入导师本人信息。读者不用把履历当简历背诵，只需要看清三个点：阶段边界在哪里，阶段变化可能带来什么研究线索，哪些地方还要回到来源核查。

正文怎么写：

1. 开头说明本节功能：履历阶段是后文阅读论文集合的索引。
2. 先确定阶段边界：时间、机构、职位，以及公开资料中能看到的方向、平台或合作环境变化。这个判断属于写作检查，不必把“划分依据”作为学生可见表头。
3. 给出履历阶段表。表格不要只列年份，要写清每个阶段暴露出的研究线索，以及后文怎么看。
4. `阶段线索` 只写证据能支撑的短标签，例如对象、平台、方法或问题类型；不要在这里解释术语含义。
5. 表后补一小段读法：阶段变化可能对应研究问题、方法平台、合作者和论文主题的变化；这些只是线索，后面要用 `方向入口` 和论文集合继续核查。
6. 缩写、机构和领域词首次出现时展开；来源缺失或阶段边界不稳时标 `[未找到]` 或 `需人工复核`。

图表怎么用：主表用履历阶段表。建议表头为：

| 阶段 | 时间 | 机构 / 职位 | 阶段线索 | 后文怎么看 |
|:---|:---|:---|:---|:---|

`阶段线索` 要写给学生看，不写成后台判断。它可以包括三类信息：研究对象、方法平台、问题类型。当前阶段可以标得更清楚，但仍然只做索引，不提前完成方向解释。

必要时加一行文本链路：

```text
履历阶段 -> 论文分组 -> 方向入口 -> 后续领域地图
```

这行只在能降低理解负担时使用；如果表格已经够清楚，就不额外加图。

读完检查：读者应能说出老师大致经历了哪些阶段，各阶段出现过哪些研究线索，当前阶段在哪里，以及为什么后面看论文集合时不能把所有年份的论文混成一团。

容易写偏的地方：

- 把履历写成传记、履历美化或导师评价。
- 只列年份、机构和职位，没有说明阶段对论文阅读有什么用。
- 只写阶段边界，没有写各阶段研究线索，学生看完仍不知道为什么要按阶段看论文。
- 在这里提前完成研究方向解释，抢走 `方向入口` 和 `02_领域地图.md` 的职责。
- 把阶段方向写成包装性总结，例如“完整建立某某体系”，但没有来源边界。
- 阶段划分依据不清，导致后面的论文分组看起来像机械按年份切块。
- 缩写不展开，例如 RIKEN、AMO、HHG 直接出现。

### 方向入口

要解决的问题：让学生在进入 `02_领域地图.md` 前，先获得几个最低限度的方向入口词。读者读完这一节后，不需要理解领域全貌，但应知道后面反复出现的几个词大概指什么、这些词来自哪些公开线索、还需要在哪里核查。

它承接 `学术履历`。上一节把导师经历整理成阶段索引；这一节从当前阶段、官方方向和近期论文中提取入口词。这里给的是初步抓手，完整领域坐标留给 `02`，论文内部关系留给 `03`。

正文怎么写：

1. 开头先用短段承接履历：读者已经知道老师经历了哪些阶段，但还缺少当前方向的入口词。
2. 说明入口词来源：官方方向、当前机构或课题组、近期论文题名 / 摘要词、履历当前阶段。
3. 给一个小型入口词卡，列 3-5 个关键词。每个词写中文、英文或缩写、一句话解释和后续位置。
4. 表后用短段说明证据边界：这些词只是进入 `02` 的临时抓手，完整领域位置要到 `02` 建立，论文之间的关系要到 `03` 再看。
5. 如果某个词只来自弱线索，标 `需人工复核`；如果来源缺失，标 `[未找到]`。

图表怎么用：只用一个入口词卡，默认不再加“方向线索表”。方向线索和证据边界用短段落说明，避免把本节写成表格集合。

入口词卡骨架：

| 入口词 | 先这样理解 | 主要线索 | 后续去哪看 |
|:---|:---|:---|:---|
| <中文词>（<English / abbreviation>） | <一句话解释> | <官网 / 近期论文 / 当前阶段> | `02` / `03` |

读完检查：读者应能说出 3-5 个入口词，知道它们分别来自官网、当前阶段还是近期论文，并知道自己还没有真正理解完整领域。

容易写偏的地方：

- 写成完整领域综述。
- 把早期方向、旁支论文和当前方向混成一坨。
- 用两张以上表格拆解线索，让学生还没进入 `02` 就被证据表压住。
- 只堆专业名词，没有给新手一句话解释。
- 把一两篇近期论文直接写成当前主线。
- 没有说明每个方向词的来源和证据边界。

### 论文集合

要解决的问题：完整论文表保留了老师直接相关的研究细节，也给后续核查留下事实入口。学生第一遍不需要逐篇精读；正文要教他先看履历阶段、计数、年份分布、来源风险和当前阶段信号。

它承接 `方向入口`。上一节给出几个方向入口词；这一节回到完整论文事实。按学术履历分组，是为了让学生看到机构、职位或职称变化后，研究对象、方法平台、合作者和论文主题可能怎样变化。后续文档会使用这张表复核方向判断；本节自己不选择目标论文，也不解释论文路线。

正文怎么写：

1. 说明论文集合的覆盖范围和主要风险，数据源口径细节留到 `数据库指标说明`。
2. 说明为什么保留完整列表：它保存导师论文层面的研究细节，后续可用来复核方向判断。
3. 按学术履历阶段组织论文表，阶段边界尽量对应前文履历，而不是机械按年份切块。
4. 说明第一遍读法：先看阶段标题、论文数量、年份分布、来源风险和当前阶段信号。
5. 每个阶段表格前加 1-2 句读法提示：这个阶段大概看什么，避免直接报一串题名。
6. 保留完整论文表，不删除论文，不改写原题，不筛成代表作。
7. 若结构化数据已支持中文辅助标题，可单独设置 `论文标题` 和 `中文标题` 两列；若暂不支持，只能在阶段读法提示或表外短注中提供中文辅助，不手工增删论文表列或行。

结构化支持到位后，论文表可以写成：

| # | 年份 | 论文标题 | 中文标题 | 期刊 / 出处 | 引用 | 来源 |
|:---|:---|:---|:---|:---|:---|:---|
| 1 | 2025 | `<English original title>` | `<中文辅助翻译>` | `<journal / venue>` | `<citation>` | `<DOI / arXiv / database>` |

图表怎么用：阶段读法提示 + 完整论文表。中文标题对新手有价值，但它必须来自结构字段或明确的成品试写，不允许执行 AI 为了可读性手工改坏事实表。为控制复杂度，中文标题只放短翻译，不放解释；论文标题只保留原始题名；不要再加摘要、关键词、研究问题或论文角色列。

读完检查：读者应能说出论文集合的大致规模、阶段分布、当前阶段信号和主要风险。

容易写偏的地方：

- 删除或改写脚本生成表格。
- 写“代表性论文”导致事实集合被误解为筛选集合。
- 每个阶段只报标题，没有解释阶段读法。
- 长表前没有缓冲，学生一眼被压垮。
- 在 `01` 里判断目标论文、论文角色或研究路线，抢走 `03_论文路线.md` 的职责。
- 给中文标题列写解释性长句，使表格比原英文题名还难扫。
- 中文标题没有标清辅助性质，让学生误以为它是数据库或期刊提供的原始元数据。
- 后续文档把早期阶段论文和当前阶段论文混用，没有说明为什么当前申请更关注最新相关阶段。

### 合作与公开信息

要解决的问题：用论文之外的公开线索补充前文事实，并提示可能的冲突。合作对象和公开活动可以反映研究环境，但在 `01` 里先服务信息核查，不写成人脉、平台或实力评价。

它承接 `论文集合`。上一节保留完整论文事实；这一节挑出能补强或质疑这些事实的公开线索，例如高频合作者、合作机构、讲座 / 新闻、项目 / 平台。学生读完后，应知道哪些线索能补强“同一阶段、同一论文圈层、同一公开主题”的判断，哪些只暴露疑点，不能单独重做身份锁定。

正文怎么写：

1. 说明这一节为何放在论文集合之后：先有论文事实，再用论文之外的公开线索检查这些事实是否对得上。
2. 先写核查目标：看合作者、机构、公开主题和项目线索，能补充什么事实，是否暴露机构、年份、主题或论文归属冲突。
3. 用一张公开交叉佐证表，不列人脉清单。
4. 表后写使用边界：公开线索可以补充阶段、论文归属和研究环境判断，不能单独证明导师实力、申请价值或当前主线。
5. 如果出现同名、机构不一致、方向不相干、年份对不上等情况，标 `需人工复核`。
6. 接到 `主体自检`：合作与公开信息看完后，主体事实线已经走完，下一节检查学生能不能带着事实底座进入 `02`。

图表怎么用：使用一张公开交叉佐证表。它回答“这条公开线索能补充什么、可能暴露什么冲突”，不回答“这位导师强不强”。

| 线索类型 | 主要来源 | 能补充什么 | 不能证明什么 | 风险处理 |
|:---|:---|:---|:---|:---|
| 高频合作者 | 论文共作者、课题组页 | 论文集合是否像同一研究圈层 | 不能直接证明导师实力 | 合作者同名或跨领域时标 `需人工复核` |
| 合作机构 | 论文署名、项目页、新闻页 | 履历阶段、实验平台或合作环境是否对得上 | 不能单独推出当前主线 | 机构与履历不一致时回查论文来源 |
| 公开讲座 / 新闻 | 学院新闻、会议页面、讲座海报 | 某段时间公开讨论过什么主题 | 不能替代论文证据 | 来源缺失或宣传性强时降级为弱线索 |
| 项目 / 平台 | 基金页、实验室页、设备平台页 | 当前阶段可能依托什么平台或方向 | 不能评价资源强弱 | 只写可核查事实，不写价值判断 |

表后要有短结论。证据较稳时，可以写“这些公开线索与前文论文集合和履历阶段大体能互相支撑”。证据弱时，直接写“公开信息目前只能提供有限辅助，后续判断仍应以官网身份、履历阶段和论文集合为主”。

读完检查：读者应能说出三件事：哪些公开线索补强了阶段和论文归属，哪些线索只说明研究环境，哪些地方需要人工复核。

容易写偏的地方：

- 写成平台宣传。
- 写成“人脉强”“资源好”“实力强”。
- 用合作对象评价导师。
- 用新闻稿替代论文和官网证据。
- 在这里重新给出身份锁定结论，和 `资料与身份核查` 重复。
- 把讲座、新闻或项目标题直接写成当前主线。
- 发现机构、年份、合作者或方向对不上，却没有标出风险。
- 把所有合作者列成名单，没有说明它们能补充什么或可能暴露什么冲突。

### 主体自检

要解决的问题：把 `01` 主体阅读的出口从“看完了”变成“能复述事实底座，并知道哪里还不稳”。这一节是主体自检，只检查身份、履历、方向入口、论文集合和公开线索的最低输出；它不替代 `00_材料导读.md` 的整套材料自检，也不要求学生已经理解导师的研究领域。

它承接前面的主体小节。学生读到这里时，已经看过身份、履历、方向入口词、论文集合和公开交叉线索。自检的作用是把这些信息收束成一张可操作的检查表：能说出来，就可以进入 `02_领域地图.md`；说不出来，就回到对应小节补读。数据库记录和来源参考放在后两节，作为需要细查时使用的核查区。

正文怎么写：

1. 先说明自检目的：确认自己是否抓住事实底座，不检查专业理解深度。
2. 写清通过标准：学生应能用自己的话说出身份来源、履历阶段、论文集合读法、当前阶段信号和主要风险。
3. 给出自检表，表中每一行都要有“应能说出什么”和“答不上回哪里”。
4. 表后写处理方式：答不上不是失败，而是提示该回到哪一节补事实；不要带着身份和论文归属疑问直接进入领域解释。
5. 接到后续核查区：如果自检通过，可以进入 `02`；如果还想核查数据库记录、引用指标或具体来源，再看 `数据库指标说明` 和 `来源与参考资料`。

图表怎么用：短表“自检项 / 应能说出什么 / 答不上回哪里”。

| 自检项 | 应能说出什么 | 答不上回哪里 |
|:---|:---|:---|
| 身份与资料状态 | 当前身份来自哪些官方或作者档案线索，资料是否能暂时合并阅读 | `资料与身份核查` |
| 履历阶段 | 老师经历了哪些主要阶段，当前或最近相关阶段在哪里 | `学术履历` |
| 方向入口 | 进入 `02` 前先记住哪些最低方向词，这些词还需要在哪里继续核查 | `方向入口` |
| 论文集合 | 论文按哪些阶段组织，为什么不能只看代表作或只看最新几篇 | `论文集合` |
| 公开交叉线索 | 哪些公开线索能补强阶段、论文归属或研究环境，哪些线索只暴露疑点 | `合作与公开信息` |
| 风险意识 | 哪些判断仍是弱证据或需人工复核，后续阅读时不能当成定论 | 对应风险所在小节 |

表后要有短承接。合适的出口是：“如果这些问题大体能回答，就可以进入 `02_领域地图.md`，开始理解当前方向的位置；如果身份、履历阶段或论文归属仍不清楚，先回到对应小节，不要急着用不稳事实解释研究方向。数据库记录、引用指标和具体来源可在后两节继续核查。”

读完检查：读者应能判断自己是否可以进入 `02`，也应知道未来读完 `02/03/04` 回看 `01` 时，应该复核身份、阶段、论文归属、数据口径和来源强弱。

容易写偏的地方：

- 写成考试题。
- 只问“是否理解”，不定义合格输出。
- 不告诉学生答不上回哪一节。
- 把 `02_领域地图.md` 的专业理解要求提前放进来。
- 把数据库记录和来源参考写成进入 `02` 前必须精读的前置条件。
- 只写鼓励性总结，没有形成可操作的回看入口。

### 数据库指标说明

要解决的问题：解释 agent 采用的关键数据库记录和指标应该怎样读。前文 `资料与身份核查` 已经判断这批资料能否暂时合并阅读；本节不重新做身份判断，也不汇总所有来源，只说明数据库类记录、引用指标和论文元数据的口径。若数据库记录暴露出新的同名、机构或论文归属冲突，应回到 `资料与身份核查` 标出风险。

正文怎么写：

1. 说明引用数、h-index、论文数和作者档案都是数据库在某个时间点的记录，只能作为资料线索。
2. 简短解释首次出现的数据源和缩写，例如 Google Scholar、OpenAlex、ORCID、DOI / Crossref、arXiv。
3. 用一张数据口径表说明每类记录可以参考什么、常见风险是什么、本文怎样处理。
4. 表后写本画像采用的口径：论文归属优先看官网身份、作者档案、题名、合作者和机构署名是否互相支撑；引用指标只记录数据库状态。
5. 如果数据库之间论文数、机构、引用或作者档案差异较大，标 `需人工复核`，不要用顺滑文字圆过去。
6. 接到 `来源与参考资料`：这一节讲数据库指标怎么看，下一节列出关键事实分别由哪些来源支撑。

图表怎么用：只用一张数据口径表，不用 Mermaid。表格回答“这个数据项可以帮我判断什么，不能证明什么”。

| 数据项 | 主要来源 | 可以参考什么 | 常见风险 | 本文怎么处理 |
|:---|:---|:---|:---|:---|
| 作者档案记录 | Google Scholar / OpenAlex / ORCID | 数据库怎样记录作者、机构和论文 | 同名、机构滞后、作者页合并或拆分 | 这里只说明数据库口径；身份是否成立以前文核查为准 |
| 引用数 | Google Scholar / OpenAlex | 数据库收录和论文传播记录 | 不同数据库口径不同，近期论文引用少很正常 | 只描述记录状态，不写成影响力评价 |
| h-index | Google Scholar / OpenAlex | 作者档案中的整体引用记录 | 受数据库覆盖、同名混入和学科差异影响 | 可列出，但不用于导师价值判断 |
| DOI / 期刊页 | Crossref / 期刊官网 | 单篇论文题名、年份、期刊、作者列表 | DOI 缺失、元数据延迟、题名差异 | 用来核对论文表，不单独证明作者身份 |
| 预印本 | arXiv 等 | 论文早期版本和开放全文线索 | 版本可能不同于正式发表版 | 标明版本，不替代正式论文元数据 |

读完检查：读者应能说清引用指标能参考什么、不能证明什么；也应知道数据库差异、同名噪声、近期论文收录延迟和 DOI 缺失都需要保守处理。

容易写偏的地方：

- 写“影响力很大”“引用很高所以值得关注”。
- 把 h-index 当成导师评价。
- 不说明 Google Scholar、OpenAlex、ORCID、Crossref 等来源口径差异。
- 只列数字，不说明这些数字能证明什么。
- 因为近期论文引用少，就暗示论文不重要。
- 因为某数据库没有收录，就直接排除论文。

### 来源与参考资料

要解决的问题：让关键事实可回查。它接近参考文献，但比普通参考文献多一个核查功能：每条来源都要说明支撑什么、不能证明什么，避免读者看到链接就误以为所有判断都有同等强度的证据。

正文怎么写：

1. 说明来源与参考资料的用途：身份、履历、论文集合、数据风险都应能追溯。
2. 按来源类型或证据用途列出来源。
3. 每条写清支撑内容和限制。
4. 缺失来源写 `[未找到]`，弱判断写 `需人工复核`。

图表怎么用：来源表，列为“来源 / 类型 / 支撑内容 / 不能证明 / 链接”。如果来源很多，可以按官方身份、作者档案、论文元数据、公开交叉线索、背景资料分组。

表后加一个很短的核查出口。合适的写法是：“如果你只做第一遍阅读，前面的 `主体自检` 已经足够进入 `02`；如果你要复核关键事实，应至少能从这里找到身份、履历、论文归属和数据风险分别由哪些来源支撑。找不到来源的判断，后续阅读时按弱证据处理。”

读完检查：读者应能从正文关键判断回到来源，并判断来源是否足够支撑当前句子。

容易写偏的地方：

- 只放 URL。
- 来源和判断之间没有对应关系。
- 用一个来源支撑它不能支撑的结论。
- 写成论文 bibliography，只列作者、年份和题名，不告诉读者这条来源在本文里支撑什么。

<a id="confirmed-rules"></a>
## 已确认规则

- `01` 是事实底座，不写成传记、领域综述、论文路线或导师评价。
- 文首必须有目录和显式 anchor；正文顺序采用“入口 -> 资料核查 -> 主体明细 -> 主体自检 -> 核查区”。
- `01` 必须支持两次阅读：第一遍粗读事实轮廓，读完 `02/03/04` 后回看核查事实和风险。
- `资料与身份核查` 只给“资料能否暂时合并阅读”的主判断，不抢数据库口径、来源映射、领域解释或论文路线。
- `学术履历` 只做阶段索引和研究线索，不提前解释完整方向；`方向入口` 才给进入 `02` 前的最低入口词。
- 完整论文集合是事实审计资产，不能删除、改写、筛成代表作，也不能在 `01` 中判断目标论文、论文角色或研究路线。
- 论文集合必须按履历阶段组织，并在长表前给第一遍读法提示。
- 中文标题列是试写和技术回收候选；结构化支持到位前，不要求执行 AI 手工增删论文表列或行。
- `合作与公开信息` 只能补强事实或提示冲突，不能重新锁定身份，也不能写成导师实力、人脉、平台或申请价值评价。
- 数据库指标只描述记录和口径，不能写成导师影响力评价；`来源与参考资料` 必须说明每条来源支撑什么、不能证明什么。
- `主体自检` 只检查 `01` 主体事实底座，并告诉学生答不上时回到哪一节；数据库记录和来源参考属于后续核查区。
- 第一次出现的机构缩写、领域缩写和专业术语必须展开并给短解释；复杂 Mermaid 默认不用，表格只在能降低记忆负担时使用。

## 成品试写时验证

这一节只记录还需要通过张鹏举、汪非凡等成品试写判断的开放点；逐节写法已经放在“小节设计”里，不再重复。

## 阶段 1 技术边界

这里不展开技术重构方案，只保留会影响 `01` 成品标准的约束：

- workbench 先不被旧 §1-§9 脚本骨架约束；旧脚本输出只能当事实材料，不能决定最终章节顺序。
