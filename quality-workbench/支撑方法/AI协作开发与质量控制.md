# AI 协作开发与质量控制调研

状态：工作台调研稿。用于解释本项目为什么反复出现“理解正确、落地很差”，并给出后续使用 Codex 重构 S1-S4、D00-D04、skill 和 harness 时的协作规则。

边界：这份文稿不直接修改 S1-S4、D00-D04、真实导师成品、`research-advisor` skill、verifier 或 harness。它先把 AI 协作方式说清楚，等支撑方法和成品标准稳定后，再决定哪些规则进入正式工作流。

## 这份调研要回答的问题

本项目遇到的低效，核心不在写作速度。这里要追问的是：模型经常能复述用户的目标，也能写出看似完整的文档，交付出来的东西仍然像资料拼接、章节堆叠、图表装饰和原则口号。

这份调研只回答三件事。

- 为什么顶级模型在这个项目里没有稳定产出高质量结果。
- 外部团队和工具文档怎样把 AI 放进开发流程，避免输出失控。
- 接下来重构 S1-S4 前，Codex 应该怎样改变协作方式。

判断标准只有三条：能不能减少返工，能不能让用户更早发现方向偏差，能不能让 AI 的每一步输出都留下可审查的依据。

## 外部资料给出的共同信号

外部资料没有给出一个万能流程。它们共同指向同一件事：AI 的能力要通过任务边界、证据、检查、反馈和复盘接到工程系统里。只给模型更多上下文、更多规则、更多模板，通常不能自动提高质量。

| 来源 | 公开资料给出的信号 | 本项目应吸收的做法 |
|:---|:---|:---|
| OpenAI Codex 文档 | Codex 任务需要清楚写出目标、上下文、约束和完成条件；`AGENTS.md`、skills 和 repo 指令适合放长期规则；复杂任务需要拆分并让 Codex 能运行验证。 | 每次开工前写“任务合同”：当前只改哪一层、输入是什么、完成条件是什么、哪些文件禁止动、验证命令是什么。 |

这些资料落到本项目，就是一个判断：AI 协作质量取决于模型，也取决于模型被放在什么任务结构里。当前项目的问题，很多来自任务结构和验证结构。

## 为什么本项目会低效

### 1. 任务层级经常混在一起

这个仓库现在至少有七层对象：M0 总方法、S1-S4 支撑方法、D00-D04 文档标准、真实导师成品、`research-advisor` skill、verifier、harness。用户指出 S4 可视化烂，AI 可能转去改 03 成品；用户指出 03 章节接不上，AI 又可能回头改总方法；用户要求优化支撑方法，AI 又容易顺手改标准或样本。

层级一混，模型就会选择最容易生成的东西：补章节、加表格、写原则、画流程图。它看起来在推进，实际是在不同层之间乱跳。

后续每次开工都要先锁层级：

```text
当前问题属于哪一层？
本次只允许改哪一层？
其他层只记录影响，不直接修改。
```

### 2. 上下文过多，但缺少路由

项目里有 `AGENTS.md`、`QUICKSTART.md`、上下文交接、workbench、多个 skill、历史聊天和真实样本。每一份都有合理内容，但模型在长上下文里会抓到很多相互竞争的线索。

规则数量不能保证执行稳定。当前缺的是路由：遇到一个问题时，应该先判断它属于哪一层、调用哪份方法、产出什么中间判断、何时停下。没有路由时，模型会把“所有看起来相关的规则”都塞进输出，文档就会变厚、变散、变像模板。

### 3. 完成条件太虚

“优化这份方法”“提高质量”“让学生能看懂”都是真目标，但不能直接作为完成条件。模型会把它们翻译成更长的解释、更整齐的结构和更多检查项。

可执行的完成条件应该写成：

```text
读者入口是什么？
读者出口是什么？
本次新增了哪项调研或判断？
哪些旧内容被删除、降级或合并？
学生或执行 AI 可以用哪几个问题检查是否通过？
```

没有这些条件，AI 很容易把“写得完整”误当成“写得有效”。

### 4. 验证只覆盖格式，覆盖不了学术质量

当前 verifier 可以检查结构、来源标记、禁用语和部分 smoke。它有价值，但它不能判断论文路线是否真的读懂，不能判断 02 的领域定位是否被 03 反向修正，也不能判断一张图是否真的减少学生的理解负担。

这会制造一个危险错觉：机器检查通过，AI 就倾向于声称完成。实际上，本项目最难的是研究判断、证据边界、读者路径和章节接力，这些需要人工审稿式检查。

后续要把验证分成两层：

- 机械验证：格式、链接、禁用语、文件状态、diff。
- 人工审查：证据能否支撑判断、章节能否接力、学生能否完成输出、图表是否必要。

### 5. 参考资料没有进入决策链

前几轮支撑方法暴露了一个固定问题：参考资料看起来很多，正文却没有体现“为什么读这些资料、从资料得到什么结论、怎样应用到本项目”。参考文献留在文末，方法正文仍然像 AI 自己写出来的原则拼贴。

以后每条外部资料都要过一张小卡：

```text
资料解决的原问题：
资料给出的可迁移判断：
本项目吸收成什么规则：
这条规则会改变哪份文档或哪一步执行：
不能直接照搬的边界：
```

写不出这张卡，资料就不能进入方法正文。

### 6. 研究型任务被当成普通代码任务处理

代码任务往往有编译、测试、类型检查、截图和日志。导师调研文档没有这么强的自动反馈。`03_论文路线.md` 这类任务尤其难，因为它要求 AI 真读论文集合，判断问题链和角色分工，再把判断写成学生能跟上的路线。

这类任务不能用“先生成，再润色”的方式做。它必须先有证据表、论文角色判断、冲突和弱线索标记，再进入正文写作。否则模型会根据 02 的假设先选材料，再把选出来的材料写成看似自洽的路线。

### 7. 用户反馈没有及时沉淀成执行门槛

用户已经反复指出：章节像抽屉、支撑方法像资料拼凑、图像装饰、AI 味重、03 不能先射箭再画靶。问题在于这些反馈经常停留在聊天里，下一轮模型又靠记忆和临场理解执行。

后续反馈要立刻归档到当前层级的门槛里。例如“03 不能只挑支持 02 的论文”应该进入 M0 和 D03；“可视化先问学生少背哪段关系”应该进入 S4；“支撑方法要写问题、资料、结论、应用”应该进入所有 S1-S4 的统一大纲。

### 8. 单个主 agent 容易被自己的意图污染

长任务里，主 agent 会同时扮演作者、解释者、修订者和审稿者。它知道自己“本来想表达什么”，所以会把页面上没有写清的关系自动补全，把图表看成自己脑中的设计，而不是读者实际看到的材料。上下文越长，这种自证偏差越重：前面的解释、道歉、承诺和旧方案都会混进判断。

本项目不能只靠主 agent 自审。高风险文档要在主 agent 自审后冻结当前稿，交给上下文更干净的 subagent 做回放。Subagent 不负责替作者圆场，也不负责继续扩写；它只报告一个新读者或专项审稿者实际看到了什么、哪里卡住、能不能完成预期输出。

### 9. Subagent 用少了和用滥了都会低效

少用 subagent，主窗口会继续把调研噪声、解释过程、失败修补和自我辩护全塞进同一个上下文，最后看起来理解了用户要求，实际交付还是按旧路线走。滥用 subagent，则会出现另一种低效：每个小改动都开审查，主 agent 等结果时不做事，多个 agent 读同一堆脏上下文，最后得到三份相似的泛泛意见。

充分利用 subagent 的标准不是“开得多”，而是让每个 subagent 带走一类会污染主线、又能独立完成的工作。主 agent 留在主线程里做三件事：锁层级、做最终判断、编辑当前层级。Subagent 做五类事：并行读材料、隔离审查、跑验证、模拟读者，或承担 disjoint 的机械改动。

## 让 Codex 变强的协作规则

下面这套规则先作为工作台协作协议。它不替代 M0，也不替代 S1-S4；它规定 AI 在改这些文件前怎样工作。

### 新窗口默认调度

新窗口接手非小任务时，先用 2 到 3 分钟做调度判断。不要等用户提醒，也不要一上来开一串 agent。

```text
1. 读入口：AGENTS.md、QUICKSTART.md、docs/上下文交接.md。
2. 锁层级：本次只处理 M0 / S1-S4 / D00-D04 / 样本 / skill / verifier / harness / 代码中的一层。
3. 写任务合同：目标、输入、约束、完成条件、验证、隔离审查。
4. 判断主线阻塞：下一步必须由主 agent 本地完成的，留在主线。
5. 判断可并行侧线：读证据、查冲突、模拟读者、图文审查、跑验证，可以交给 subagent。
6. 给每个 subagent 一个单一审查轴和返回格式。
7. 主 agent 在 subagent 运行时继续做不依赖结果的本地工作。
8. 收到结果后只回灌结论，不把 subagent 的原始长过程塞回主线。
```

这张调度卡的目标是让新窗口主动判断“该不该开、开什么、何时等、怎样用结果”。任务合同里的“隔离审查”不能写成空泛的“无”，只能写 `不开 / 需要 / 延后`，并说明触发原因、审查轴和回灌位置。

短指令默认分派：

| 用户短指令 | 主层级 | 默认 subagent | 主 agent 本地职责 |
|:---|:---|:---|:---|
| 继续修 S4 | `S4 可视化与理解` | 图文审查；必要时加小白读者回放 | 先只读判断现稿是否真的回答“什么情况下应该出什么图”，再决定返工小块 |
| 继续修 D03 / 03 论文路线标准 | `D00-D04 / D03` | 小白读者回放；涉及论文证据、目标论文或论文归属时加证据审查 | 锁定 D03，不顺手改 S4、样本、skill 或 verifier |
| 继续修 D04 / 04 学习向导标准 | `D00-D04 / D04` | 小白读者回放；涉及目标论文、课程桥、核心图时加证据审查 | 检查学生是否能从基础课走到论文和下一步行动 |
| 继续修真实样本 | 真实样本 | 小白读者回放；身份、论文、领域、目标论文风险加证据审查；图文密集处加图文审查 | 先确认样本路径和本轮只修哪份或哪类问题，不一次重写五份 |
| 跑验证 / 渲染 / smoke | verifier 或当前层级验证 | 验证型 subagent 可并行 | 主 agent 保持最终解释权，不把 smoke 当学术质量 |
| 查很多来源 / 读很多文件 | 当前锁定层级 | 探索型 subagent 可并行 | 主 agent 负责把探索结论转成资料到判断卡 |

### G0：层级锁定

每次任务开头先写一行层级判断：

```text
本次处理：M0 / S1-S4 / D00-D04 / 成品样本 / skill / verifier / harness
本次不处理：
如果发现其他层有问题，只记录到“影响和后续回收”，不顺手修改。
```

### G1：任务合同

开工前写清楚五件事：

```text
目标：这次要解决哪一个失败。
输入：要读哪些本地文件和外部资料。
约束：哪些文件、目录和判断不能碰。
完成条件：用户或审稿者怎样判断这次通过。
验证：能跑哪些命令，哪些质量只能人工审查。
隔离审查：不开 / 需要 / 延后；触发原因是什么；审查轴是什么；结果回灌到哪里。
```

这一步要短，但不能省。没有任务合同，模型会用“看起来相关”的内容填满回合。

### G2：证据卡和决策卡

涉及调研、方法、论文路线、学习路径和可视化时，先写证据卡，再写正文。证据卡回答来源说了什么，决策卡回答本项目吸收什么。

证据卡不需要变成长期文件，可以先写在当前文稿内部。只有跨导师、跨文件稳定复用的判断，才回收进 skill 或 harness。

### G3：小步设计

改一份支撑方法前，先给出大纲和段落工作，不直接整篇重写。用户确认方向后，再改文件。

对本项目来说，“一次改四份支撑方法”风险很高。更稳的节奏是：

```text
S1 大纲和方法链 -> 用户确认 -> S1 正文 -> 自审和提交
S2 大纲和方法链 -> 用户确认 -> S2 正文 -> 自审和提交
S3、S4 同理
```

### G4：单层编辑

一次只改一个层级。重构 S1 时，不顺手改 D01-D04、样本、skill 或 verifier。发现必须同步修改时，先在文末写“影响和后续回收”，等当前里程碑结束再决定。

这条规则可以减少项目继续长胖，也能避免用户以为在改方法，结果 AI 悄悄把成品标准和执行脚本也改了。

### G5：人工质量关

每份中文方法文档至少过六个问题：

```text
它从哪个真实失败开始？
它调研了什么资料？
它从资料得到什么判断？
它把判断转成什么执行动作？
它怎样检查失败有没有被解决？
它没有覆盖什么，后续回哪里？
```

如果一份支撑方法回答不了这六个问题，就算写得很顺，也不能算通过。

### G5a：隔离 Subagent 审查

Subagent 的作用是切断主 agent 的自证偏差和长上下文污染。它只在有明确质量风险时使用，不把每个小改动都变成多 agent 流水线。

本项目默认使用四种 subagent 工作模式。`AGENTS.md` 里列的是六类常用角色；其中小白读者回放、证据审查和图文审查都属于下面的“审查型”。两处说的是同一套路由，不是两套规则。

| 模式 | 适合任务 | 主 agent 同时做什么 | 不适合 |
|:---|:---|:---|:---|
| 探索型 | 查目标文件、找规则冲突、梳理论文或来源集合 | 本地锁层级、读主文件、设计下一步 | 让 subagent 替主 agent 决定总体路线 |
| 审查型 | 小白读者、证据审查、图文审查、结构审查 | 冻结稿件，准备回灌表 | 还没有草稿或稳定 diff 时空审查 |
| 验证型 | 跑测试、链接、渲染、smoke、长命令输出归纳 | 继续改不依赖验证结果的文本或计划 | 让验证型 agent 判断学术质量 |
| 工作型 | disjoint 文件或模块的机械改动 | 主 agent 负责接口、范围和最终整合 | 多个 agent 并行改同一文件或同一段文档 |

`隔离审查` 只能写三种结论：

```text
不开：只改错字、路径、标题、机械格式；或只是单段措辞调整，且不改变证据、主线、图文和学生输出；或本次已经是单轴只读审查，没有并行分轴价值。
需要：命中高风险触发器，当前已有冻结稿或明确可并行侧线，可以立即开 subagent。
延后：当前还没有冻结稿，但任务完成后会产生高风险稿件；必须写明冻结点、审查轴和回灌位置。
```

必须优先考虑 subagent 的场景：

| 场景 | 推荐 subagent | 触发原因 |
|:---|:---|:---|
| `00-04` 成品、`D00-D04` 标准完成一轮实质修订 | 小白读者回放 | 主 agent 容易把设计意图当成读者实际路径 |
| `03_论文路线.md`、`04_学习向导.md` 或对应标准 | 小白读者回放；必要时加证据审查 | 论文角色、学习桥和学生输出很难靠机械验证证明 |
| `S4` 可视化、任何图表密集段落 | 图文审查；必要时加小白读者回放 | 图容易变成完成任务的装饰 |
| 身份锁定、论文归属、领域定位、目标论文选择 | 证据审查 | 来源强弱和弱线索很容易被主线语气放大 |
| 一个窗口已经长时间推进、发生上下文压缩或多次被用户指出偏差 | 小白读者回放或只读审查 | 主窗口可能带着旧解释继续自洽 |
| 需要读很多文件、论文、来源或旧规范，但最终只想沉淀少量判断 | 探索型 subagent | 把长阅读和噪声移出主线程 |
| 机械验证或渲染会产生长日志，但主 agent 还可以继续本地工作 | 验证型 subagent | 主线程不被命令输出污染，也不空等 |

不必使用 subagent 的场景：

- 只改错字、路径、标题锚点或机械格式。
- 单段局部措辞调整，且不改变证据、主线、图文或学生输出。
- 当前任务已经是只读审查，且没有需要并行分轴判断的问题。
- 环境或工具不可用；此时必须在汇报里说明没有开 subagent 的原因和残余风险。
- 下一步是主线阻塞决策，例如确定当前层级、选择本轮目标、决定是否改文件。这些由主 agent 本地完成。

Subagent 输入要保持干净。主 agent 只给四类信息：

```text
冻结稿或目标文件：
读者身份或审查角色：
预期出口：
禁止事项：
```

不要给完整聊天记录、作者原意、旧版辩解、内部过程稿或主 agent 对质量的自我评价。小白读者 subagent 只看最终文本和预期出口；证据审查 subagent 只看判断、来源和证据边界；图文审查 subagent 只看图文片段、图前图后和学生输出。

Subagent 输出必须能回灌，不写泛泛优缺点。最低输出格式：

```text
我实际读到的主线：
我能完成的输出：
我完成不了的输出：
我在哪里卡住：
哪段文字、表格或图造成误读：
建议先修的最小位置：
```

不同模式的提示模板：

```text
探索型：
只读目标文件。回答这个具体问题：<问题>。不要改文件，不要读 archive/。只返回证据、路径和判断，不返回长摘录。

小白读者：
你只知道<学生背景>。阅读冻结稿<文件或片段>，尝试完成<学生输出>。不要看内部规则，不替作者脑补。报告实际阅读路线、卡住位置和不能完成的输出。

证据审查：
只检查这些判断是否被来源支撑：<判断列表>。按“直接支持 / 交叉支持 / 弱线索 / 不支持 / 需人工复核”分类，给出来源位置和降级建议。

图文审查：
只检查图、表、Mermaid、SVG 或证据卡是否降低理解负担。逐项回答：学生卡点、图是否必要、图前图后是否消费、应保留/降级/删除。

验证型：
运行或检查<命令/目标>，只返回通过项、失败项、关键日志、建议下一步。不要改文件，除非任务明确给了 disjoint 写入范围。
```

主 agent 不要立刻等待每个 subagent。只有下一步被 subagent 结果阻塞时才等；否则继续做本地上下文整理、读主文件、准备编辑或机械验证。等待结果后，主 agent 只合并“可执行结论”，不把 subagent 的长链路复制进正文。

### G6：机械验证

机械验证只证明文件状态和格式底线。文档类里程碑至少运行：

```powershell
git diff --check -- . ':(exclude)archive/**' ':(exclude)output/**/archive/**'
rg -n "<项目禁用表达正则>" <目标文件>
git status --short
```

如果涉及真实导师成品，再运行对应的 phase docs verifier。通过后仍要声明：机械验证不能证明学术判断通过。

### G7：复盘回收

里程碑结束后只做一件回收判断：

```text
这次发现的是单文件表达问题，还是跨导师、跨文件会复现的方法问题？
```

单文件问题留在当前文档。跨文件问题才进入 M0、S1-S4、D00-D04、skill、verifier 或 harness。这个判断能防止每次反馈都长出新规则。

### G7a：Subagent 反馈回灌

主 agent 收到反馈后，先做对照表，再改文件：

| 原设计路线 | Subagent 实际读到 | 偏差原因 | 修法 |
|:---|:---|:---|:---|
| 学生应该拿走的判断或动作 | subagent 实际复述或失败的位置 | 证据、段落、图文、术语或出口问题 | 删、改、拆、降级或回到上游方法 |

更细的回灌卡可以这样写：

```text
subagent 发现：
证据位置：
对应层级：
对应关卡：
主 agent 决策：采纳 / 部分采纳 / 不采纳 / 待验证
本次动作：
是否需要后续回收：
```

如果 subagent 不能完成学生输出，主 agent 不能只润色句子。要回到对应关卡：出口不清回 S2 或 M0 出口关；证据不足回 S1；图不帮理解回 S4 支撑关；段落接不上回 S3 成文关。两轮回放仍失败时，应停止在当前层级继续补丁式修改，改做只读复盘或重新锁定更小范围。

反馈不自动等于规则。只有跨导师、跨文档会复现，且能写成执行动作和停写条件的反馈，才进入 `quality-workbench/`、skill、template、verifier 或 harness。单次表达问题留在当前文件；未采纳的 subagent 意见要简短说明原因，防止下一窗口重复追同一个假缺口。

## 对 Codex、skills 和 harness 的改造建议

### 短期：先改变当前协作，不急着改系统

接下来重构 S1-S4 前，先执行五条规则。

- 每次开工先写任务合同和层级锁定。
- 每份支撑方法都按“真实失败 -> 资料和依据 -> 本项目判断 -> 执行动作 -> 质量关”组织。
- 每次只改一份文件，完成后跑验证、看 diff、检查 `git status`。
- 用户指出偏差后，先判断偏差属于哪一关，再回到对应文件修改，不跳去样本或 harness。
- 高风险文档完成主 agent 自审后，开小白读者、证据审查或图文审查 subagent 做只读回放；主 agent 只拿回放结果和反馈，不把完整脏上下文传给它。

这几条足够改变当前返工模式。现在不宜马上重构 skill、verifier 或模板，因为方法层还没稳定。

### 中期：给 skill 和 workbench 加路由

等 S1-S4 稳定后，再考虑把 `research-advisor` 拆得更清楚。当前 skill 入口把事实采集、蓝图、五份成品、可视化、验证和回收都放在一个大入口里，容易让模型在长任务里抓错重点。

中期可以做三件事。

- 给 skill 增加“当前层级判断”入口，先判断任务属于 M0、S、D、样本、verifier 还是 harness。
- 把支撑方法的参考资料改成“来源 -> 判断 -> 项目规则”的索引，不让参考文献只做摆设。
- 增加人工审稿清单，专门检查证据边界、章节接力、学生输出和图文必要性。它不伪装成自动 verifier。

### 长期：把自进化理解成评估闭环

Apodex 这类公司提到 auto-evolution，可借鉴点是闭环。对本项目来说，自进化应该长成这样的形态：

```text
失败样本 -> 失败原因定位 -> 最小规则修订 -> 金样本回归 -> 新任务复测 -> 决定是否回收进正式 harness
```

长期可以建设一小组稳定评估样本，停止继续堆流程文件。张鹏举样本可以作为黄金样本，但前提是先把它人工修到足够好。之后每次改方法、skill 或 verifier，都回到黄金样本检查：输出是否更像学生能读懂的导师材料，是否少了幻觉，是否减少用户纠偏成本。

## Skills 回收边界

这次调研先回收到项目内 superpowers 类 skills，并补了一次全局中文文档 skill 的通用缺口。项目内 skills 只服务本仓库；全局 skill 只吸收跨项目适用的中文文档质量门槛。

| Skill | 本次处理 | 原因 |
|:---|:---|:---|
| `using-superpowers` | 改成项目技能路由入口 | 防止技能机械触发后把任务带向旧流程。 |
| `brainstorming` | 改成 workbench、方法、标准、skill 和 harness 的设计门槛 | 防止默认创建 `docs/superpowers/specs`，也防止未锁层级就改文件。 |
| `writing-plans` | 改成按项目类型选择计划位置 | 文档方法计划不应被写成代码实现计划，也不应默认新增计划文件。 |
| `verification-before-completion` | 增加文档、skill、导师成品和代码四类检查 | 让“已完成”必须对应新鲜验证，同时保留人工质量边界。 |
| `research-advisor` | 暂不大改 | 等 M0 和 S1-S4 稳定后，再把已验证规则回收到正式项目 skill。 |
| 全局 `human-readable-chinese-docs` | 已补通用门槛 | 增加资料到判断链、章节接力、方法文档从失败到动作的检查；导师项目细节仍留在 workbench 和 `research-advisor`。 |

后续继续改全局 skill 时，仍然要先证明问题跨项目出现，并能写成短规则。只属于导师调研、论文路线、学习路径或 `00-04` 的规则，先留在本仓库。

A0b 的分流结论是：全局中文文档 skill 可以管“资料怎样进入判断”“章节怎样接力”“方法规则怎样从失败走到动作”；它不管某位导师、某篇论文、某份 `00-04` 的具体学术判断。具体学术判断仍由当前项目的 M0、S1-S4、D00-D04、`research-advisor` 和人工审稿承担。

## 对接下来重构 S1-S4 的影响

这份调研直接改变 S1-S4 的写法。

S1 信息检索与筛查要做成“资料怎样获得正文资格”的方法。它要从同名、弱线索、论文归属和领域误判这些失败开始，说明检索资料怎样经过消歧、分级、冲突处理和正文边界，不能只列搜索渠道。

S2 学习与认知原则要做成“学生怎样从看过走到会复述、会自检、会回退”的方法。它要把认知负荷、主动提取和自我解释转成文档出口、学习块和回看路径，不能堆学习科学名词。

S3 学术品味与文稿设计要做成“章节怎样接力推进判断”的方法。它要解决章节孤岛、段落空转、AI 味和判断语气的问题，不能只讲写作原则。

S4 可视化与理解要做成“什么时候图能减少学生要背的关系”的方法。它要从学生卡点选择正文字段、表格、矩阵、证据卡、范围图、结构图或路线图，不能再收集图形家族。

四份支撑方法都要回答同一组问题：

```text
它解决哪种真实失败？
它参考了哪些资料？
它从资料得到什么结论？
它怎样应用到 00-04？
执行 AI 怎样知道自己做错了？
```

如果这一组问题回答不出来，支撑方法就会再次变成资料拼接。
