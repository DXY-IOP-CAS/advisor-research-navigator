# 学习方法论 v8.5

## 一、总体设计

### 一句话本质

**一次只学一个东西。** 讲一小段 → 你实操一小段 → 你复述 → 确认理解 → 下一个。

### 双轨学习

| | 操作型学习 | 调研型学习 |
|---|---|---|
| **目标** | 会用这个工具/技能 | 理解设计决策，形成自己的判断 |
| **初体验** | 跑一次，看输入输出 | 读摘要/README，确认这是什么 |
| **知识点形态** | 概念 + 沙盒实操 + 引导性验证 | 概念 + 对比分析 + 判断问题 |
| **复述重点** | "跑出什么了？为什么？" | "这个设计决策对我们意味着什么？能用吗？" |
| **模块收尾** | 综合应用（闭卷实操） | 收尾盘点（回顾所有判断，写入设计问题） |

判定标准：模块中有没有可运行的东西。有 → 操作型路线；没有 → 调研型路线。一个模块内可混合。

### 五阶段流程

操作型：`全景图 → 初体验 → 定大纲 → 逐点学习 × N → 综合应用`

调研型：`全景图 → 快速定位 → 定大纲 → 逐点学习 × N → 收尾盘点`

阶段名只标识进度，不绑定文件编号。

### 阶段-模式绑定

每个阶段对应不同的 AI Agent 设计模式。进入阶段时 AI 应切换行为模式——不是形式主义标记，而是在结构层面提醒 AI 当前阶段的认知任务不同：

| 阶段 | 对应模式 | 该模式的行为约束 |
|:----:|:-------:|:--------------:|
| 全景图 + 初体验 | Plan-and-Execute | 先全貌后细节，不跳步，不省略结构 |
| 定大纲 | Plan-and-Execute | 拆分学习点并等待用户审核确认——这是 HITL 关键节点 |
| 逐点学习（调研型） | **ReAct** | 搜索→阅读→分析→再搜索，每个 K 点循环执行 |
| 逐点学习（操作型） | Plan-and-Execute | 按沙盒步骤执行，每步确认用户实操结果 |
| 收尾盘点（调研型） | **Reflection + HITL** | 自评性回顾——对照设计问题逐条回答——人工确认闭合 |

模式来源：Anthropic (2024.12) Building Effective Agents + 行业共识 (2026)。不同阶段的模式天然不同，本表正式绑定它们。

### 设计依据

| 设计决策 | 文献支撑 |
|---------|---------|
| "一次只学一个东西" | Sweller (1988)：认知负荷理论——工作记忆容量有限（≈4-7 信息块），一次塞多个导致认知过载 |
| 全景图 + 初体验（先看完整样例再动手） | Sweller & Cooper (1985)：工作样例效应——耗时减半、错误率降至五分之一 |
| "必须复述" | Chi et al. (1989)：自我解释效应——复述环节强制触发自我解释 |
| 综合应用不翻课件（闭卷测试） | Roediger & Karpicke (2006)：检索练习效应——闭卷回忆远优于重复阅读 |

## 二、学习流程

### 2.1 全景图 + 初体验

**全景图**：知道这玩意能干什么。内容：一句话本质 + 目录结构 + 输入输出例子。**呈现形式**：写入 `00_全景图.md`，告知文件名后由用户阅读。禁止在对话中直接展示全景图内容。检查点：能说出它吃进去什么、吐出来什么。

**初体验（操作型）**：用最简单输入跑一次，不解释细节。检查点：对输出形状有基本感觉。

**初体验（调研型）**：读摘要/README，确认这大概是什么、解决什么问题。检查点：能用一句话说出它的核心主张。

### 2.2 定大纲

根据学习内容拆知识点清单。你参与审核：加减知识点、调整顺序。

**两种拆分逻辑**：
- 操作型：**按使用流程拆，不按文档结构拆**。拆完检查——每个知识点有没有对应的实操动作？没有 → 拆错了。
- 调研型：**按"理解→对比→判断"的独立闭环拆**。源文档是汇总报告时，拆分应考虑被汇总对象的独立性——一个 H2 如果代表独立框架的完整思想体系（有自己的核心机制、设计张力、适用场景），即使只有一个 H2，也独立成学习点。

**切块标准**：一个学习点 = 一个概念 + 一个实操动作（或对比/判断动作）。机械判定（任一命中 → 切太粗了，拆）：

| 条件 | 操作型 | 调研型 |
|------|:---:|:---:|
| 课件 H2 超过 N 个 | N=3 | N=5 |
| 实操动作超过 1 个 | 拆 | 拆 |
| 学习点标题含 `+`、`与`、`&` | 拆 | 拆 |

调研型放宽 H2 上限的理由：纯概念对比知识点的内容是表格和列举而非独立子概念，多 H2 不代表认知过载。但 H2 上限只用于判"太粗"——不用于判"够细"。当 H2 各自是独立框架时，即使总数未超 5 也拆为独立学习点。

**大知识点拆分**：满足以下任一条件 → 拆为子步骤（KX → KX-1, KX-2, KX-3）：(a) 课件草稿 H2 仍超限且概念不同；(b) 内容自然分为"背景/核心机制/设计分析"三层；(c) 用户反馈"认知过载"。AI 在写课件前先评估——超限 → 先拆分再写，不允许写完后补拆。

**调研型启动要求**：定大纲阶段结束时，AI 必须写下本模块要回答的设计问题（1-3 个）。收尾盘点时逐条对照。无设计问题的调研模块无方向。

### 2.3 逐点学习

每个知识点执行同一个循环，操作型和调研型不同。

**操作型**：`AI 写入文件 → 你读 → 你复述 + 指不懂 → AI 纠正/补充 → 确认 → "继续"`

**调研型**：

```
AI 回溯原始来源（WebSearch/WebFetch 查原始论文、官方文档、社区原帖）
→ AI 写文件（内容来自原始来源而非本地汇总文档，写清核心机制和设计张力。篇幅以信息完整为准——保证从零基础到深入理解所需信息均已覆盖；同时按认知科学"循序渐进"原则分层递送，防止一次性全量加载造成认知过载）
→ 告知文件名+概要
→ 你读
→ 你在对话中说判断
→ AI 补充/纠正
→ 确认
→ 反馈落盘：AI 先读 `学习方法论-附件.md` 附录E（模板+自检锚点），再提取 1-3 条设计决策或待定问题，追加写入模块文件夹内 `学习笔记.md`。写完立即执行自检锚点#2（每条决策含 `**怎么来的**` 字段且引用用户原话）
→ "继续"
```

**禁止**：一条消息讲多个点、不等复述就推进。

### 2.4 收尾

**操作型（综合应用）**：用一个挂钩实际项目的任务做闭卷测试，不翻课件。完成后评价：优点 + 缺陷 + 可提取什么。

**调研型（收尾盘点）**：以 `学习笔记.md` 为基础，逐条对照模块启动时提出的设计问题。产出：`收尾盘点.md`，包含"已有积累 → 核心判断 → 对设计问题的影响 → 仍开放的问题"。

**模块关闭动作**：收尾盘点完成后，AI 自动更新 `指导/续学指令.md` 指向下一模块；将本次模块的核心判断变更写入续学指令的"已完成模块"节；检查关联指导文件是否有因本次变更而过期的引用并同步修正。无需用户提醒。

**不要在此窗口启动下一模块。** 收尾完成后 AI 只做两件事：(1) 告知用户本模块已就绪、可以新开窗口继续；(2) 给出续学指令中新窗口开场指令的预览。不再询问"是否立即开始下一模块"——每次跨模块必须新开窗口，以避免不同模块的上下文互相污染。

**收尾体验检查**：收尾盘点结束后，AI 在对话中向用户提三个问题（30 秒回答，不写入文件）：

1. 这个模块学完，有什么是流程本身拖慢了学习的？
2. 有什么是上次反馈过、这次仍然没改善的？
3. 如果只能改一件事，改什么？

用户无需全都回答。AI 根据回答诊断根因——可能是方法论的问题，也可能是模块内容本身不适合用户的学习风格。确认需修改的方法论条目后，按指导文件迭代的元规则操作（备份→修改→更新版本记录）。

这三个问题的作用：每次模块收尾给用户一个明确的"可以给反馈"的信号，防止问题累积到爆发才暴露。

**框架回填**：收尾盘点 + 体验检查完成后，AI 基于本模块学到的内容，修改 `.claude/skills/project-pilot/` 中对应的阶段文件。回填规则：

1. **只改本模块对应的阶段**（D04→阶段1、D05→阶段2、D06→阶段3+4、D07→全局铁律）
2. **改的是具体的规则/模板/对话策略**——不是"建议"，是可直接采用的 diff 级修改
3. **在修改处标注来源**（`<!-- 来源：D0X 模块X 决策X -->`），确保 13 份文档学完后能从标注反向追溯每条改动的设计依据
4. **改完后重读 `练习册/project-pilot/综合设计文档_框架设计快照_v0.1.md`**——逐条确认新改动与已有设计决策无冲突。无冲突 → 通过。有矛盾 → 在改动处标注 `⚠️ 待确认：与 D0X 决策N 矛盾`，不强行裁决——留到下次收尾或 D13 汇总时统一处理
5. **改完汇总到模块的 `学习笔记.md`**——记录"本模块回填了哪条、为什么、与已有决策有无冲突"

设计依据：回填是"看一棵树改一棵树"——容易漏掉跨阶段的冲突。改动后重读综合设计文档的检查动作是从"树"飞起来看"林"，不新增流程步骤，只加一个 30 秒的确认。

## 三、AI 执行规则

1. **用户控速**：不得自行推进，用户说"继续"是唯一信号。简单知识点可由用户主动要求合并推进。
2. **零跳步**：不省略中间推理，禁用"显然""易得"。小纠正和补充可在对话中完成，不走文件流程。
3. **文件出入**：首次教学（全景图/定大纲/逐点课件）和重大修正 → `.md` 文件，对话告知文件名和概要。小纠正/快速演示直接在对话中进行。输入写完整可复制字符串，输出不省略中间步骤。调研型课件篇幅充分且循序渐进——保证从零到深入理解所需信息量，同时避免认知过载。必须来自原始来源；操作型含沙盒课件正常密度。
4. **按需验证**：不强制每个知识点末放验证题。复述中已展示正确理解 → 直接推进。复述中发现偏差 → AI 给 1 个针对性追问。
5. **防幻觉**：课件中出现 arXiv 编号/论文标题/作者名/DOI/公式/物理常数/实验数据/第三方引用/工具行为 → 必须先 WebSearch + WebFetch 验证再写入。搜索严格依序执行（规则见[身份与学术标准.md](身份与学术标准.md)的"联网搜索铁律"——硬约束，禁止跳步，连续 2 次失败才可降级）。搜索 URL 附课件末尾"验证来源"小节。
6. **原始来源回溯**：调研型课件涉及外部框架/论文/工具（含本地汇总文档中引用但未展开的）→ 必须先读原始来源再写课件。判据：课件中至少包含一处"从原始来源获得、本地汇总文档中没有的细节"。
7. **反谄媚分析**：调研型课件将外部框架与 project-pilot 对照时，必须同时包含"支持"和"挑战"两列。挑战列回答：这个框架的什么比我们好？什么在质疑我们的核心假设？全表只有支持性证据 → 机械判定为谄媚，必须重写。
8. **追问回路**：用户主动追问（"给我讲讲 LLM 原理""那 Harness 有什么缺陷"）→ AI 全力解答，不受深度限制。解答完成后 → AI 主动确认：**"我们在 [当前主线任务] 的中途。继续主线，还是这个方向再挖深一层？"** 不自动跳回，不自动跑偏。设计依据：五角色模型的 Metadata Paradox——用户临时指令（P4）不应永久覆盖框架 Authority（P1），但应在接到指令时全力满足，满足后回位。

### 学习者规则

1. **复述理解**：读完课件后在对话中用自己的话复述核心内容。
2. **指出不懂**：遇到任何不理解的点立刻说"这里不懂"，不跳过。

### 协作规则

AI 写文件 → 告知文件名+概要 → 你读 → AI 给引导性问题（给观察视角不给结论）→ 你复述+指不懂 → AI 纠正/补充 → 确认 → 继续。AI 负责要求复述，你负责复述——缺一不可。

## 四、沙盒模式

纯文档阅读无法建立"亲手跑过一遍"的实操感。学框架设计或代码机制时，提取与当前学习点直接相关的**最小可运行片段**。

**三条原则**：

1. **极简沙盒**：最多两个文件（`_沙盒_` + `_结果_`），直接放在模块文件夹内与课件同级。文件命名 `NN_N_描述.md`——第一个 NN 是课件创建顺序编号，第二个 N 是组内排序（`_1_` 课件 / `_2_` 沙盒 / `_3_` 结果）。
2. **人触发，人解释**：AI 只生成运行材料。你必须亲手执行（终端回车、对话参与）。跑完后你先解释——AI **禁止**在你复述之前给出结构化分析。AI 的分析在你复述后写入 `_结果_` 文件。
3. **不重复聊天**：复述在对话中完成，不额外创建 `复述.md`。对话记录本身可追溯。

**沙盒分级**：

| 级别 | 何时用 | 形式 | 落盘 |
|------|-------|------|:--:|
| 快速演示 | 小概念、"看一眼就懂" | 对话中展示，30 秒–2 分钟 | 否 |
| 物理沙盒 | 核心机制需要亲手体感 | 两个文件，5–15 分钟 | 是 |

不要沙盒的知识点：课件中标注"本知识点不设沙盒"。

## 五、文件规范

**编号**：按创建先后顺序，取目录下已有最大编号 +1（`01_xxx.md`, `02_xxx.md`, …）。编号和描述松耦合。非课件文件（PDF、翻译 Markdown、源码包）不加数字前缀。

**内部格式**：标题 `#`，H2 之间用 `---` 分隔，只用 H2 和 H3（H4 禁止）。禁止 `>` 块引用、中文数字序列（一、二、三）、`\tag{}`、`\begin{theorem}`、`\begin{proof}`。公式用 `$$...$$`，步骤理由用加粗写在公式下方，手动编号 (1)(2)(3)。最终结果用 `**最终结果：** ...`。

格式细表、已知局限、版本记录、参考文献见：[学习方法论-附件.md](学习方法论-附件.md)
