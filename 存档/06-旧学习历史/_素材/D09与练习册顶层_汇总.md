## 1.1 五种分发形式

来源：`02_K1_五种分发形式全景扫描.md`、`02_K1-1_五种形式快速认识.md`

### 1.1.1 GitHub 模板仓库
- 定义：预配置 GitHub 仓库，用户通过 "Use this template" 按钮或 `git clone` 创建副本
- 核心机制：Git 模板机制，新项目继承全部文件后完全脱离原模板（无自动同步）
- 关键特征：携带脚手架（空目录结构）、全平台兼容、分发和更新都用 git
- 用户拿到的是：一堆文件（README, CLAUDE.md, 空目录）
- 适用场景：项目脚手架——给用户一个"开箱即用"的起点

### 1.1.2 Claude Code Skill
- 定义：符合 Agent Skills 开放标准的 `.md` 文件，安装到 `~/.claude/skills/<name>/SKILL.md`
- 核心机制：按需加载 + 描述匹配；跨 30+ 工具兼容
- 用户拿到的是：纯指令集——没有文件、没有目录结构
- 适用场景：可复用的能力模块

### 1.1.3 Claude Code Plugin
- 定义：打包了 MCP Server + Hooks + Skills 的功能套装，marketplace 一键安装
- 核心机制：Plugin = MCP Server（工具执行环境）+ Hooks（事件触发）+ Skills（提示词模板）
- 用户拿到的是：代码+服务——有后端进程在运行
- 适用场景：复杂功能套装（需后端服务/数据库/外部 API）

### 1.1.4 npm/npx 包
- 定义：经 npm registry 分发的 Node.js 包
- 核心机制：CLI 脚手架工具，框架逻辑体现为可执行脚本和交互式命令行
- 用户拿到的是：可执行脚本
- 关键特征：需 Node.js 环境、能生成高度定制的项目骨架

### 1.1.5 Cookiecutter 模板
- 定义：用 Cookiecutter 工具驱动的高度定制化项目模板
- 核心机制：用户回答预设问题 → 模板替换占位符 → 生成最终项目，可含条件分支逻辑
- 用户拿到的是：可定制项目骨架
- 关键特征：每次创建可走不同分支，但需要额外安装 Cookiecutter

### 1.1.6 两种本质类型
来源：`02_K1-1_五种形式快速认识.md`

| 类型 | 代表 | 给用户的是什么 |
|------|------|--------------|
| **文件载体** | 模板仓库、npm 包、Cookiecutter | 文件+目录结构 |
| **指令载体** | Skill | 纯指令，不携带文件 |
| **服务载体** | Plugin | 指令+代码+后端服务 |

## 1.2 四个判定维度

来源：`03_K1-2_四个判定维度拆解.md`

### 维度一：用户体验（门槛和第一印象）
- 模板仓库：1 步（点 Use this template），门槛：有 GitHub 账号
- Skill：2 步（下载→放到 skills/），门槛：知道 Claude Code
- Plugin：1 步（marketplace 装），门槛：只限 Claude Code
- npm/npx：1 步+等待，门槛：需 Node.js
- Cookiecutter：2 步（装工具→运行），门槛：需装额外工具
- **权重：高**。框架是"做事方式"——低门槛是用户愿意试的唯一理由。

### 维度二：开发成本（维护者的投入）
- 模板仓库/Skill：写 .md 文件，无需写代码、配置 CI、管理包版本
- Plugin：全都要做（打包 MCP+Hooks+Skills）
- npm/npx：开发 CLI、维护打包
- Cookiecutter：写模板+占位符逻辑
- **权重：高**。现阶段单人维护，任何"先写代码才能发布"的机制都是负担。

### 维度三：跨平台（框架的生命力）
- 模板仓库：任何能读 .md 的编辑器/IDE
- Skill：Agent Skills 标准 → 30+ 工具
- Plugin：仅 Claude Code
- npm/npx：仅 Node.js 项目
- Cookiecutter：任何能装 Python 的环境
- **权重：中**。现阶段聚焦 Claude Code，但结构上不应排斥未来的跨平台。选了 Platform-locked 的形式以后再改代价大。

### 维度四：可迭代（纠错和演进）
- 模板仓库：git pull upstream merge → 用户可选择性采纳
- Skill：覆盖文件/版本管理器 → 不覆盖就不变
- Plugin：marketplace 自动更新 → 推了就更新
- npm/npx：npm update → 要么更要么不更
- Cookiecutter：重新生成 → 破坏定制内容
- **权重：高**。方法论迭代频繁，用户必须先"审"再"采纳"。

### 综合评分
来源：`03_K1-2_四个判定维度拆解.md`

| 形式 | 用户体验 | 开发成本 | 跨平台 | 可迭代 | 总分 |
|------|---------|---------|-------|-------|:----:|
| 模板仓库 | 5/5 | 5/5 | 5/5 | 4/5 | 19/20 |
| Skill | 4/5 | 5/5 | 5/5 | 5/5 | 19/20 |
| Plugin | 5/5 | 2/5 | 1/5 | 3/5 | 11/20 |
| npm/npx | 3/5 | 3/5 | 2/5 | 3/5 | 11/20 |
| Cookiecutter | 2/5 | 4/5 | 5/5 | 3/5 | 14/20 |

**模板仓库和 Skill 分数并列最高，且不是替代关系**。

## 1.3 为什么不是纯 Plugin 纯 Skill

来源：`04_K2_为什么不是纯Plugin纯Skill.md`

### 不选纯 Plugin 的三条理由
1. **平台锁定**：Plugin 只能在 Claude Code 用。框架的规则和模板写在 .md 里，任何能读 .md 的 IDE 都该能用——Plugin 自断跨平台
2. **成本不成比例**：Plugin 要求打包 MCP Server（后端进程）。框架核心 = 规则+模板，不需要后端服务。为分发 .md 文件搭 MCP Server 是本末倒置
3. **更新太激进**：marketplace 推更新就自动装。框架迭代频繁，用户需要"看看再决定要不要"——不是"推了就覆盖"

### 不选纯 Skill（全局安装）的三条理由
1. **description 太宽泛**：框架触发条件"用户表达真实项目意图"——几乎所有对话都有可能触发，频繁误激活浪费上下文
2. **无法携带脚手架**：SKILL.md 是纯指令不携带文件。框架需要 CLAUDE.md + 空目录结构 + 模板文件——Skill 给不了
3. **用户自定义和框架更新冲突**：全局 Skill 更新=覆盖文件。用户做过本地修改被抹掉。git merge 模式（模板仓库）允许选择性采纳

核心矛盾：**Skill 设计哲学是"能力模块"（给 AI 加的指令集），框架需求是"项目容器"（给项目搭的架子）**。

### 组合方案的三层架构
来源：`04_K2_为什么不是纯Plugin纯Skill.md`

| 层次 | 载体 | 职责 |
|------|------|------|
| **容器层** | GitHub 模板仓库 | CLAUDE.md、目录结构、模板文件、Hooks 配置 |
| **内容层** | `.claude/skills/项目执行/SKILL.md` | 阶段判断、流程引导、质量门规则 |
| **脚手架** | 空目录（产出/需求/调研/计划/） | 项目产物组织引导 |

三种需求分别满足：
- 跨平台：CLAUDE.md 和目录结构在任何工具中有意义
- 按需加载：SKILL.md 只在 Claude Code 中自动发现，其他工具忽略
- 更新可控：git merge upstream 模式，用户审阅

### D09 对个人使用的核心价值
1. **Skill 的 description 不能太宽也不能太窄**——太宽每句话都触发，太窄永远不触发
2. **目录结构设计意图**——`产出/` 放阶段产物，`需求/` `调研/` `计划/` 是空目录引导用户
3. **CLAUDE.md 的定位**——始终加载的宪法级约束，不与 Skill 互斥

### D09 对个人没用的部分
五种形式对比、Plugin vs Skill 论证、用户使用流程、更新机制——这些是面向 C 端分发的，个人使用不需要。

### D10 核心（增量验证与用户参与）
三个认知偏差现象：
- **Instant Gratification Bias**（14 名参与者 156 次偏差）：开发者接受 AI 输出是因为"它能跑"，不管它是不是对的
- **Fixation（固执于初始方案）**（回退率 43.4%）：AI 给的第一个方案不一定对，不改方向会越跑越偏
- **Suggester Preference（盲目信任）**（102 次）：用户相信"AI 的数学构造优于自己的判断"

三条防止策略：
1. **增量验证**——每写一小段就检查，原子任务粒度 1-2 轮对话
2. **Shift Left（验证前移）**——写→查→写→查→… 而不是写写写→最后查
3. **用户参与分两种模式**：学习型（阅读→复述→提问→确认）vs 审核型（对照清单逐条确认）

最小理解门槛三问：做了什么？为什么这样做而不是那样做？怎么验证做对了？

### 学习反馈记录（F1/F2/F3）
- **F1（重度）**：框架机械强制层空白。14 条规则中 9 条纯提示词软约束。三次修复失败历史（CLAUDE.md 写规则→不遵守；加 SKILL.md 细化→仍绕过；加 PreToolUse Hook 提醒型→无效无阻断）。根因：三层审核的 Hook 层空白、AI 自检层纯提示词。
- **F2（已确认方向）**：优先级重排——个人使用优先，C 端分发推后。D09 分发形式决策的核心价值是"最终用户"，但当前只需要为一人使用。
- **F3（待处理）**：Agent 自进化方向需要 D11 时大范围补充调研。D11 当前覆盖的是"框架治理"（RFC 废弃机制、规则上限、KISS），不覆盖 Agent 自进化技术（Reflexion、Self-Refine、AgentOptimizer 等）。

## 2.4 软约束失效问题

来源：`软约束失效问题_业界方案.md`

### 问题确认
"提示词只是建议"不是项目特有——是整个 LLM Agent 行业正在解决的开放问题。

### 业界方案收敛：确定性控制面
多个独立团队结论高度一致——禁止在模型内部做规则执行，在模型外部建一个代码层拦截器。

当前项目可落地的机械强制点：
| 强制点 | 检查内容 | 可阻断？ |
|--------|---------|:-------:|
| Write 前 | 文件路径是否符合约定命名 | 可阻断 |
| Write 后 | 文件内容是否含 `## 验证来源` | 可警告 |
| 非 Write 工具前 | 回复中是否含教学内容但未先写文件 | 待验证 |
| 阶段切换前 | 前阶段产出文件是否存在 | 可阻断 |

### 自进化的正确定位
Self-Harness 的思路——"执行→记录失败痕迹→分析失败模式→提出规则修正→回归测试验证→用户确认→应用"——正是你说的"外循环：人决策+AI 执行"。

### 对当前的意义
不要再问"方法论怎么写对"——核心矛盾不在于规则设计。方向：
1. 先把可阻断的 Hook 建起来（3-4条）
2. 保证"不跳过调研阶段"（你最看重调研）
3. 保证"不跳过用户确认"
4. 在有约束的壳里试跑一次真实需求
5. 跑的过程中记录违规→变成新 Hook 或规则修正

## 4.4 对导师调研工具的总建议

1. **形态**：GitHub 模板仓库 + 内置 Skill（复制 project-pilot 的交付策略）
2. **分层**：CLAUDE.md（宪法约束）+ `.claude/skills/导师调研/SKILL.md`（阶段流程）+ 空目录（产出/ 结构引导）
3. **四阶段**：需求锚定（确定学者和范围）→ 全面调研（论文搜索+筛选+精读）→ 计划制定（讲稿大纲+优先级）→ 执行审核（逐章生成+增量验证）
4. **机械强制**：Hook 层必须现实——引用验证、文件存在检查、阶段门控。不能纯靠提示词
5. **自迭代**：借鉴 Self-Harness 五步循环——每次使用记录哪些搜索策略无效 → 用户确认后更新搜索规则
6. **不做什么**：不 Plugin 化（平台锁定）、不需要后端服务（纯 .md 文件）、不引入代码框架依赖
