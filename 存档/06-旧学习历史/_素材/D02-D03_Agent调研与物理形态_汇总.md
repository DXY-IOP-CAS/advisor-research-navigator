# D02: 通用 Agent 全面调研报告

## 块 1: D02 全景与定大纲

来源: 00_全景图.md / 01_定大纲.md

### 调研了什么

D01 快速扫描了 4 个框架（BMAD、AgentFactory、SDD、Claude Code 基础），感觉"有缺口但没证据"。D02 在此基础上系统调研 10 个框架，用对比矩阵来验证空白是否存在。

### 学到了什么

D01→D02 的变化: 框架数从 4 个扩到 10 个，方法从"快速找差异"变成"逐一深剖 + 对比矩阵"。结论从"好像没人做完整的"升级到"严格证明没人同时满足 5 个条件"。D01 已学的 BMAD、AgentFactory、SDD 在 D02 中跳过不重学，仅在可借元素环节引用。

### 对工具设计的可借鉴点

- 跨框架对比矩阵方法是验证"市场空白"的有效工具——不是凭感觉说"别人没有"，是用数据证明"别人在哪些维度有缺失"
- D02 启动时设定的 3 个设计问题值得借鉴: (1) 三层架构是否需要调整? (2) 哪些元素是"不拿来框架就不成立"? (3) 差异化定位还站得住吗?
- "定大纲"阶段明确设计问题，收尾时逐条对照——这种闭环方法可直接用于任何调研型工作

### 结论/决策

D02 学习路线: K1（3 个新增核心框架）→ K2（轻量新增 + 对比矩阵）→ K3（综合判断）。D01 已学框架跳过。

## 块 2: K1 — 3 个新增核心框架概览

来源: 02_1_K1_新增核心框架.md

### 调研了什么

三个新增核心框架: AI for Auto-Research（学术研究四阶段）、Plan-and-Execute（行业标准 Agent 架构）、Claude Code 生态（6 层能力模型）。为每个框架做了"一句话本质"、核心结构分析、与 project-pilot 的对照表。

### 学到了什么

三框架对照总表显示: Auto-Research 有"四阶段意识"但纯学术导向; Plan-Execute 通用但缺需求诱导和调研; Claude Code 生态提供能力栈但不是流程。没有一个同时覆盖"需求诱导 → 调研 → 计划 → 执行审核"四个阶段 + 领域无关 + 轻量分发 + 自我迭代。

### 对工具设计的可借鉴点

- 三框架对照表比单框分析更能暴露空白——定位不是"我们比别人好"，是"别人缺的东西恰好我们做"
- 7 大 Agent 设计模式（Reflection、ReAct、Plan-and-Execute、Tool Use、Multi-Agent、Memory Management、Human-in-the-Loop）应该作为"战术工具箱"而非替换框架
- Claude Code 6 层能力模型（CLAUDE.md → Skills → Hooks → Subagents → Agent Teams → Plugins）证明了不需要从零造平台——框架是编排已有能力的方法论

### 结论/决策

D01 的"感觉有缺口"经系统验证成立。三个框架各有侧重，但空白确认存在。

## 块 3: K1-1 — AI for Auto-Research 深挖

来源: 04_1_K1-1_AI_for_Auto-Research.md

### 调研了什么

arXiv 2605.18661（2026 年 5 月）, 20 位作者的综述论文。覆盖学术研究完整生命周期: 四阶段八步骤（Creation: S1 想法生成/S2 文献综述/S3 编程实验/S4 图表 → Writing: S5 论文写作 → Validation: S6 同行评审/S7 反驳修订 → Dissemination: S8 论文→多格式）。五方法论文族（Prompt Engineering → RAG → Training-free Agentic → Training-based → Hybrid）。

### 学到了什么

五大核心发现/设计原则:
1. 产出生成超越验证能力（generation outpaces verification）——验证基础设施全域滞后
2. 人治协作是最可靠部署模式（human-governed collaboration）——判断/解释/问责由人保留
3. 开放任务中能力边界急剧下降——不是所有任务都适合全自动
4. 有效系统收敛于分层架构——编排方法论价值不亚于模型能力
5. AI 使用是治理问题不是检测问题

Practitioner Playbook 关键建议:
- 阶段间验证门: 假设错误会跨阶段传播，除非被显式检查
- LLM-as-Judge 不可靠: 自动评分与真实影响力负相关（rho=-0.29）
- 想法→实现有退化: 工作流需为想法-执行落差预留缓冲
- 迭代检索: 文献综述用 plan→retrieve→read→update→synthesize 循环而非单次遍历

Auto-Research 成本: $15 生成完整论文，低风险场景够用但未达顶会标准。

### 对工具设计的可借鉴点

对照表中 project-pilot vs Auto-Research 的差异:
- 切分逻辑: 学术生命周期（创造→传播） vs 项目执行流程（锚定→审核）
- 起点: 已有研究想法 vs 需求诱导对话（前置）
- 文献调研位置: 想法生成之后 vs 独立阶段（可能改变需求）
- 验证: 论文写完后 vs 每阶段之间嵌入质量门
- 人治模式: "最可靠模式"但非默认 vs 协议层强制 Human-in-the-Loop

### 结论/决策

五大发现每一条都独立印证了 D01 的设计方向:
- 原则 1 → 验证门不是 nice-to-have
- 原则 2 → Human-in-the-Loop 作为协议层方向正确
- 原则 3 → 领域无关不等于全自动
- 原则 4 → 三层架构方向正确
- 原则 5 → 经验层需要溯源追踪

四条挑战:
- 挑战 1: 阶段 2（调研）不应是固定深度——需要规模自适应
- 挑战 2: $15 端到端生成证明轻量全自动在某些场景能用——"人确认"不应一刀切，需风险分级
- 挑战 3: Phase 4 Dissemination 暴露了 project-pilot 的阶段覆盖盲区——产出怎么传播?
- 挑战 4: 两种切法（学术生命周期 vs 项目执行流程）各有适用场景，没有天然正确

## 块 4: K1-2 — Plan-and-Execute 深挖

来源: 05_1_K1-2_Plan-and-Execute.md

### 调研了什么

从 ReAct（推理+行动交替）的先天缺陷出发（慢且贵、短视、错误级联），引出 Plan-and-Execute 的核心思想: 把规划与执行拆成两个人。三种渐进演化形态: Basic（串行）→ ReWOO（变量引用消除观察开销）→ LLMCompiler（DAG 并行，快 3.6 倍）。

四个主要实现框架: LangGraph（有向图可中断）、CrewAI（角色分配并行）、AutoGen（对话式+代码执行）、Semantic Kernel（企业函数调用链）。

7 大 Agent 设计模式: Reflection、ReAct、Plan-and-Execute、Tool Use、Multi-Agent Collaboration、Memory Management、Human-in-the-Loop。

LangChain/LangGraph 的过度抽象问题: 3 行代码变 30 行，API 两周 breaking change，出 bug 调试的是框架代码不是业务代码。

### 学到了什么

Plan-and-Execute 的四个盲区:
1. 假设任务已经明确——没有需求诱导
2. 没有"调研"概念——计划从模型内部知识直接跳出来
3. 计划质量 = 模型一跳推理质量——没有方法论支撑（无 WBS/MoSCoW/SWOT）
4. 人确认是可选的，非强制

核心教训: Plan-and-Execute 是一个设计思想，不是一个代码库。三句话可以描述: "先出完整计划、人确认、再执行。每完成一步检查是否偏离方向。如果新信息改变前提，重新规划。"

### 对工具设计的可借鉴点

7 大设计模式按"去掉这个 framework 还存在吗"判据分层:
- 协议层（不可变）: Plan-and-Execute（四阶段骨架）、Human-in-the-Loop（阶段间质量门）、Reflection（阶段 4 审核本质）
- 模板层（可替换）: ReAct（适合调研阶段但不适合所有场景）、Tool Use（取决于项目类型）、Multi-Agent（复杂项目有用但非必需）、Sequential Workflow（执行层优化选项）

四条挑战:
- 挑战 1: Plan-and-Execute 没有需求锚定和调研也能产出可用结果——质疑了"必须"假设
- 挑战 2: Plan-and-Execute 的极简性是竞争力——三句话就能描述，project-pilot 的认知负担更大
- 挑战 3: DAG 并行 3.6x 加速——project-pilot 是否需要内建并行调度?
- 挑战 4: 7 大模式是社区自下而上涌现而非自上而下设计——质疑"方法论先行"路径

### 结论/决策

- project-pilot 定位可能需要从"通用框架"收窄为"高复杂度、多人确认、方法论依赖型项目"
- 用户体验必须优先保证: 没学过方法论的人能否 5 分钟内理解四阶段?
- "方法论先行"路径本身需要验证——先做最小版本（如只做需求锚定）找人试用
- 绝不引入 LangChain/LangGraph 作为依赖

## 块 5: K1-3 — Claude Code 生态深挖

来源: 06_1_K1-3_Claude_Code_生态.md

### 调研了什么

Claude Code 五层能力栈:
- 层 1 CLAUDE.md: 每次对话启动时自动加载的项目级指令，写"是什么"不写"怎么做"，200 行以内
- 层 2 Skills: YAML frontmatter + Markdown 指令，放在 .claude/skills/<技能名>/ 下。关键特性: 自动调用（description 触发）、手动限定（disable-model-invocation）、隐藏知识库（user-invocable: false）、工具预授权（allowed-tools）、子 Agent 执行（context: fork）、动态内容注入（反引号 command 反引号）、参数传递（$ARGUMENTS）
- 层 3 Hooks: 事件驱动脚本，确定性质量门——不依赖 LLM 判断
- 层 4 Subagents: 对话内独立 Agent 实例，有自己的上下文窗口
- 层 5 Plugins: 将 Skills + Hooks + MCP Servers + Agents 打包分发

### 学到了什么

project-pilot 的载体映射:
- 协议层（Constitution）→ CLAUDE.md（始终加载，不可跳过）
- 模板层（Templates）→ Skills（按需加载，每个阶段独立 Skill）
- 经验层（Experience）→ CLAUDE.md 引用 + 外部文件

四阶段 Skill 拆分方案:
- 阶段 1: anchor-requirements（需求诱导模板 + 假设确认清单）
- 阶段 2: deep-research（搜索策略 + 来源验证 + 对比矩阵模板）
- 阶段 3: make-plan（WBS 拆解 + MoSCoW 优先级模板）
- 阶段 4: review-and-close（检查清单 + 对照原始需求验证）

质量门 Hook 方案: 四个阶段过渡各对应一个检查事件。

### 对工具设计的可借鉴点

- project-pilot 不需要造平台——最小实现 = CLAUDE.md + 4 Skills + 4 Hook 规则
- Skills 遵循 Agent Skills 开放标准（agentskills.io），其他工具（Cursor、Gemini CLI、Codex）也可用
- Hooks 做自动化检查，人只看自动化解决不了的——自动化 + 人检例外
- 与 Claude Code 官方最佳实践的张力: 官方要求"state what to do, not how or why"，但 project-pilot 的方法论价值在于解释"为什么"

### 结论/决策

- 载体确认: CLAUDE.md（协议）+ Skills（模板）+ Hooks（质量门）= 最小完整实现
- 定位从"框架"下调为"一套标准写法"——约定被编码为一套 CLAUDE.md + Skills
- 质量门从"全人检"调为"自动化 + 人检例外"
- 放开"Claude Code 独占"假设
- 四个挑战: 可能只是一份精心编写的 CLAUDE.md; Skills 可跨工具; Hooks 是质量门竞争者; 方法论 vs 极简张力

## 块 6: K2 — 轻量新增框架 + 对比矩阵

来源: 03_1_K2_轻量新增与对比矩阵.md / 07_1_K2_轻量新增与对比矩阵.md

### 调研了什么

4 个轻量框架:

(1) AgentsWay（arXiv 2510.23664, 2025 年 10 月）: 首个为 AI Agent 团队设计的软件开发方法论。五个 Agent 角色（Planning → Prompting → Coding → Testing → Fine-tuning），Human as Product Owner。三个设计原则: Specialization、Iterative Learning、Privacy-by-Design。

(2) Driving AI Agents with Specifications（Pat McGuinness, 2025 年 8 月）: 先写 Spec 再动手。三份 Markdown 文档（PRD → Design Doc → Plan）一起喂给 AI，AI 按 Plan 分步执行，完成后对照 PRD 验收。

(3) 中文社区方案: 知乎 2026 AI Agent 技术全景（12 大框架解析）+ Eino ADK（字节跳动，从 0 到 1 搭 Agent）+ 国内学术工具（知网/万方 AI 助手）。中文社区方法论层空白比英文社区更大。

(4) 内容创作方向: 有工具无方法论。Remotion（视频）、Jasper/Copy.ai（写作辅助）、AI 课程生成——都是"单次生成"，无完整流程。方法论空白确认存在。

三张核心对比表:

表 1 流程阶段覆盖: project-pilot 是唯一"需求锚定 + 全面调研"都打勾的框架。BMAD 有需求锚定没调研，Auto-Research 有调研没需求锚定。

表 2 非代码场景适用性: 多数框架只服务一个领域。project-pilot 填补"通用方法论"格子。

表 3 存在形式: project-pilot 以文档为主，和 BMAD 同形态但更轻（不需要 YAML 工作流引擎）。

### 学到了什么

- AgentsWay 验证"人定方向 AI 执行"的学术认可度，但不提供领域无关性
- Spec 文档的双重作用（执行指令 + 验收标准）支撑"中间产物即验收标准"的设计
- 中文社区空白不是劣势，是更大的填补空间
- 视频工作流目前底层技术未成熟，不急着用方法论解决

### 对工具设计的可借鉴点

- "单 Agent + 按阶段换 Skills"比 AgentsWay 的"多 Agent 各管一摊"更好——避免上下文碎片
- Spec-first 法的三个缺陷: (a) Spec 本身写错; (b) 粒度难把握; (c) 盲目执行拒绝质疑 Spec。应对机制: 阶段间回退——调研可推翻需求，执行可反馈计划
- 对比矩阵中"project-pilot 全部打勾"的视觉效果可能是谄媚——需要警惕

### 结论/决策

- 矩阵结果: 需求锚定 + 全面调研的独特性在 10 框架对比后仍然成立
- 极简挑战: Pat McGuinness 三份 Markdown 就够、Plan-and-Execute 三句话就够——project-pilot 需证明额外复杂度是值得的
- 定位风险: 独特性可能等于"解决了没人觉得是问题的问题"

## 块 7: K3 — 综合产出

来源: 08_1_K3_综合产出.md

### 调研了什么

14 条可借设计元素的系统归类和采纳状态:

来自 BMAD（D01 已确认）:
1. 需求诱导顺序（先场景共识再约束收集）→ 进协议层
2. 逐节 QA 验证 → 放大到阶段级 QA
3. Stakeholder Round Table → 延后
4. 三轨道分级 → 待规模自适应

来自 AgentFactory（D01 已确认）:
5. 标准化阶段产出文档 → 进协议层
6. 工作区隔离 → 进模板层

来自 Auto-Research（D02 新增）:
7. 阶段间验证门 → 进协议层
8. 风险分级 → 待与规模自适应合并
9. 溯源追踪 → 进经验层方案

来自 SDD（D01 已确认）:
10. "Constitution"不可变层 → 已实现为协议层
11. 需求锚定 + 阶段深度自适应 → 已采纳

来自 Claude Code 生态（D02 新增）:
12. CLAUDE.md = 协议层载体 → 进协议层
13. Skills/Hooks = 模板层/质量门载体 → 进模板层

来自对比空白（D02 观察）:
14. 单 Agent + 按阶段换 Skills → 待定

5 个定位声明在 10 框架对比后的状态:
1. 唯一覆盖需求锚定+全面调研——站得住
2. 领域无关——站得住
3. 轻量分发（文档为主）——站得住
4. Human-in-the-Loop 强制——站得住
5. 四阶段可独立升级——部分站得住（DAG 并行是否打破串行? 待定）

### 学到了什么

最关键的警示: 5 个声明全部站得住可能是因为 D01 已经让我们知道了要"找什么"——D02 的对比自然验证了 D01。更诚实的说法: 10 框架中没有直接竞争者，不一定因为我们独一无二，也可能因为"没有人觉得这四个阶段必须放在一起"。

### 对工具设计的可借鉴点

下一步行动修正: 不是"学完 13 份文档再做 MVP"，而是 D02 收尾后就开始 MVP——先用阶段 1（需求锚定）在自己需求上试用。13 份文档边做边学。

### 结论/决策

- 14 条中 7 条已确认采纳，7 条待定
- 优先级最高的下一个设计工作: 规模自适应（什么时候走 Quick/Standard，什么时候人确认，什么时候自动化）
- MVP 验证是验证"独特性等于市场需求"的唯一出路
- 学习节奏调整: 不再学完所有文档再做——D02 收尾后立刻用阶段 1 模板跑一个真实需求

## 块 8: D02 收尾盘点

来源: 学习笔记.md / 收尾盘点.md

### 调研了什么

3 个设计问题的回答:

(1) 三层架构需要调整吗? → 不需要改结构，但需要换比喻。"协议/模板/经验"改为"骨架/肌肉/记忆"更直观。

(2) 哪些不可变? 哪些可选? → 不可变: 四阶段流程、阶段间质量门、Human-in-the-Loop、Plan-and-Execute 思想。可调整: 7 大设计模式、具体工具、规模自适应参数、Agent 模式、经验层格式。

(3) 差异化定位站得住吗? → 数据上站得住。但独特性可能等于"大部分人没觉得这是问题"——MVP 验证才能确认。

### 学到了什么

D02 对 D01 的修正:
- "每阶段人确认"一刀切 → 风险分级（Hook 自动化 + 人看拦不住的事）
- 阶段 2 固定深度 → 规模自适应
- "通用框架" → 更诚实地描述为"高复杂度、多利益方、方法论依赖型项目"
- Claude Code 独占 → Skills 开放标准跨工具
- 学完再建 → 边学边验（先做阶段 1 MVP）

D02 新增: 反谄媚分析机制、反馈落盘机制、单 Agent + 按阶段换 Skills、规模自适应需求、学习方法论 v8.1。

### 对工具设计的可借鉴点

仍开放的问题:

## 块 10: K1 — Agent Skills 标准与格式规范

来源: 02_1_K1_Agent_Skills标准与格式规范.md

### 调研了什么

Agent Skills 是 Anthropic 于 2025 年 10 月 16 日发布、12 月 18 日开源为开放标准的跨平台 AI Agent 能力打包格式。Anthropic 的隐喻: 给 Agent 写 Skill 就像给新员工写入职指南。策略复制了 MCP 的剧本——先做内部实现，再开源为标准让生态采纳。

目前已被 30+ 工具原生支持: Claude Code、Cursor、GitHub Copilot、OpenAI Codex、Gemini CLI、OpenCode、OpenHands、JetBrains Junie、Spring AI、Mistral Vibe 等。

格式规范:
- 目录结构: skill-name/SKILL.md + 可选 scripts/references/assets
- SKILL.md frontmatter 字段: name（必须，与目录名一致，小写字母/数字/连字符）、description（必须，1-1024 字符）、license、compatibility、metadata、allowed-tools（实验性字段）

渐进式披露（Progressive Disclosure）三级:
- 1 启动时: name + description（~100 tokens）
- 2 任务匹配时: 完整 SKILL.md 正文（建议 < 5000 tokens）
- 3 需要时: scripts/references/assets 中的文件（无上限）

文件引用只保持一层深度——防止无限追溯链。

### 学到了什么

三个设计张力:
- 跨平台承诺 vs 实验性功能: allowed-tools 跨平台支持不均
- 简洁 vs 复杂 Skill 表达力: 500 行 SKILL.md 上限 + 一层引用深度——四阶段流程可能写不下
- Anthropic 主导的开源标准 vs 真正的中立标准: 竞争生态（GPT Store、Gems、Declarative Agents）各自推不同格式

### 对工具设计的可借鉴点

对 project-pilot 的直接约束:
1. name 与目录名一致 → 多 Skill 方案必须用英文小写连字符命名
2. description 是触发开关 → 描述写得好不好决定 Skill 是否被正确加载
3. allowed-tools 不可依赖 → 跨平台场景可能失效，需要备用机制
4. 一层引用深度 → 不能做模板引用模板的链式结构

### 结论/决策

- project-pilot 的本质提炼: "AI 入职指南"——不是让 AI 变强，是让 AI 按标准做事
- 英文 Skills + 中文阅读副本分离: 英文版遵循 Agent Skills 规范（供使用），中文版不用于安装（供阅读）
- 跨平台策略: "Claude Code 跑通，然后移植"——内核不变，只需用不同"语言"重组
- Skill 编排问题是行业痛点——承认，暂不急于设计解决方案

## 块 11: K2 — 三种方案对比与 B+C 推荐

来源: 03_1_K2_三种方案对比与B+C推荐.md

### 调研了什么

三种物理形态方案:

方案 A 纯 Skill（单枪匹马）: 一个 SKILL.md 包含所有阶段指令 + 模板文件。优点: 极简、一个文件夹。缺点: 可能超 500 行限制、四个阶段混在一起 AI 可能跳过。

方案 B 多 Skill 协作（各阶段独立）: 主 SKILL.md 编排四阶段流向 + 每个阶段独立子 SKILL.md。优点: 阶段隔离、利用渐进式披露。缺点: 结构复杂、Skill 间需要状态传递、规范无内置编排机制。

方案 C GitHub 模板仓库 + CLAUDE.md + Skill 混合: 用户 clone 开箱即用。CLAUDE.md 始终加载，Skill 按需加载，空目录等待填充。

本地文档推荐 B+C 混合: 理由——跨平台、渐进加载、自文档化、可迭代、模板仓库一键启动。

### 学到了什么

反谄媚分析: 从原始来源提取了 4 个反对 B+C 的证据:
1. Agent Skills 没有编排机制——Skill 间的调用靠手写 Markdown 约定，不是平台机制
2. name = 目录名的硬约束导致中文目录名无法满足规范
3. 渐进式披露被反向利用——多 Skill 连续加载不是优化场景
4. 竞争生态可能导致格式碎片化

关键发现: 在不知道要长成什么样之前，方案 A（纯 Skill）重构成本最低。

### 对工具设计的可借鉴点

| 维度 | 方案 A（纯 Skill） | 方案 B+C（推荐） |
|------|:-:|:-:|
| 前期投入 | 低 | 高 |
| 重构成本 | 低（一个文件夹） | 高（目录和命名绑定） |
| 规模上限 | 500 行封顶 | 基本无上限 |
| 编排可靠性 | 不适用（单 Skill） | 靠 Markdown 手写约定 |
| 跨平台可迁移性 | 最好 | 取决于编排逻辑 |

### 结论/决策

B+C 方案在"全功能"和"自文档化"上优势明显，但建立在框架有足够大承载复杂度的前提下。核心判断: B+C 的编排复杂度可能过早引入不必要的负担——特别是 Agent Skills 规范本身没有编排机制。如果框架还在生长中，方案 A 更安全。

## 块 12: K3 — 能力映射与 QMC 先行策略

来源: 04_1_K3_能力映射与QMC先行策略.md

### 调研了什么

Claude Code 五层能力映射（用户决策简化版——跨平台前期不考虑，仅 Claude Code 生态内最优化）:

五层结构: 用户指令 → CLAUDE.md（始终在上下文，项目级规则）→ Agent Skills（阶段流程剧本）→ Hooks（自动化事件响应）→ Subagents（专业子 Agent）→ Skills(子)（子 Agent 专属能力）。

核心发现: project-pilot 的设计核心只有两层——CLAUDE.md + Skill。Hooks、Subagents、子 Skills 是 Claude Code 原生机制增强，少了仍然可以工作，但不是框架骨架。

QMC 先行策略评估:
- 支持理由: Claude Code 生态锚定、真实项目约束防止过度设计、Agent Skills 格式简单
- 反对理由: QMC 特征偏单一（排版密集型）、过去经验可能固化为惯例而非方法论、编排问题未解决
- 折中: 双线并进——路线 A 主（开始写 QMC，发现框架需求时记录到"框架待办"）+ 路线 B 辅（每完成一批 QMC 内容，回顾框架待办，提取通用模式，沉淀到模板）

### 学到了什么

核心层只有两层，意味着方案 A（纯 Skill 一个文件）就能跑通核心功能。B+C 的复杂度是为规模化和复用准备的，不是跑通功能的条件。

QMC 先行的关键是"定期回顾抽象"的节奏: 不是先跑完 QMC 再一次性提炼（那样风险大），而是每批 QMC 内容完成后就回顾一次。

### 对工具设计的可借鉴点

用户确认的两个 K2 决策:
1. 跨平台不考虑前期——Claude Code 先跑通再移植
2. Skill 编排问题是行业痛点——目前不急于设计解决方案

这两个决策将能力映射从"30+ 平台"收敛为"Claude Code 生态内最优化"，大幅简化了设计空间。

### 结论/决策

- 五层中核心 = CLAUDE.md + Skill（2 层架构）
- QMC 先行策略风险可控，前提是"边跑 QMC 边每批回顾抽象"
- 完成全部调研文档（D01-D13）后再动手设计框架——当前阶段纯学习不实践
- 如果框架过度适配 QMC 特征（排版格式、学者沟通等偏物理方向需求），回到"真空中设计"模式
