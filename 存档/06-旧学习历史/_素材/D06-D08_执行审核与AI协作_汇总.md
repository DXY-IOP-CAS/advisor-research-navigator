## 一、WBS / MoSCoW

### 1. WBS（工作分解结构）

**来源文件**：`02_K1_WBS_MoSCoW.md`

**定义**：面向可交付物（deliverable-oriented）的层级工作分解。不是按"谁做"或"什么时候做"分，而是按输出物分。每个输出物即一个 work package（工作包）。

两条核心规则：
- **100% Rule**：WBS 必须覆盖项目范围的 100% 工作，下层子节点工时总和 = 上层工时总和
- **8/80 Rule**：每个工作包 8-80 小时（启发式，非强制）

**你的理解**：WBS 是确认工具而非发现工具——它在"输出物结构已知"时极其有效，但对探索性项目（写论文、研究新方法）来说，分解结构本身在执行中会变化，WBS 无法处理这种动态。最致命的假设是"项目的分解结构在执行开始前是已知的"。

**对工具设计的可借鉴点**：
- WBS Dictionary 格式（编号、输入条件、输出物定义、验收标准、工时估算、负责人）可作为工具中"任务定义"的模板
- 100% Rule 可转化为完整性检查：工具可自动验证子任务工时之和是否等于父任务工时
- 但工具需支持**渐进式 WBS**（迭代内 100%，跨迭代增量补充），而非一次性完全分解

---

### 2. MoSCoW 优先级法

**来源文件**：`02_K1-2_MoSCoW.md`

**定义**：Must Have / Should Have / Could Have / Won't Have 四类优先级。由 Dai Clegg（Oracle, 1994）为 RAD 场景设计，后被 DSDM 采纳。DSDM 补充了 **60% 规则**——Must Have 不超过可用资源的 60%。

**你的理解**：MoSCoW 本质上是**多方博弈管理工具**——它的力量来自公开宣告"这个我放弃了"的透明度，而非四档分类本身。在单人 + AI 的弹性资源场景下，60% 规则的约束力自动消失（因为没有固定的资源边界）。MoSCoW 在单人场景中核心价值折半。

**对工具设计的可借鉴点**：
- 优先级标签（Must/Should/Could/Won't）可作为任务的属性标注语法引入
- 但 60% 规则在弹性资源环境不适用——需要为"没有固定资源边界"的场景设计替代取舍机制
- MoSCoW 不回答"Should Have 内部谁优先"——需要第二层排序（数字评分或 pairwise comparison）

---

### 3. 阶段3骨架的完整定义

**来源文件**：`学习笔记.md`（决策 4）、`收尾盘点.md`

**定义**：阶段3骨架 = WBS（分解）+ MoSCoW（优先级）+ 依赖关系编排（执行顺序）。三条维度互补而非互斥。用户精炼表述："WBS列清单，MoSCoW标轻重，依赖关系排顺序。"

依赖关系编排是 WBS + MoSCoW 之后缺少的第三维度——WBS 只做了平面分解，无法表达"第 N 章是第 N+1 章的前置条件"这类依赖。

**对工具设计的可借鉴点**：
- 任务管理应同时支持：静态分解（WBS）、优先级标注（MoSCoW）、前后置依赖（Task DAG）
- 依赖关系编排的具体实现可参考 Chorus 的 Task DAG 格式（有向无环图，表达先/后/并行）
- 依赖关系在项目中可能变化——工具需支持依赖的动态调整

## 二、迭代节奏

### 4. 敏捷 vs 瀑布

**来源文件**：`03_K2-1_迭代节奏.md`

**定义**：两者不是非此即彼的选择，而是"什么时候做决策"的模式选择。瀑布 = 一次性计划然后执行；敏捷 = 先做一小部分，看结果再决定下一步。大多数项目落在中间光谱。

**三个判断维度**：需求稳定性（高倾瀑布/低倾敏捷）、可交付物结构（已知倾瀑布/探索倾敏捷）、回退成本（高倾瀑布/低倾敏捷）。

**你的理解**：迭代节奏应渐进适配 WBS——不是先完全分解再执行，而是"迭代 1 只分解第 1 章工作包，执行完后根据经验调整第 2 章分解"。WBS 在迭代内强制，跨迭代增量补充。物理/数学类讲义推荐按章迭代（每章写完后做完整 Check，结论反馈到下一章 Plan）。

**对工具设计的可借鉴点**：
- 工具应支持"章级瀑布，章节间迭代"的混合节奏——每章内部按瀑布推进（Plan -> Do -> Check），章节之间按敏捷迭代（上一章输出修正下一章方向）
- 工作量分解不应一次性完成，而应是增量式——工具应支持迭代内 WBS 完整性检查而非项目级一次性检查

## 三、AI 规划工具

### 5. BMAD Method / Chorus

**来源文件**：`03_K2-2_AI规划工具.md`

**定义**：BMAD = Breakthrough Method for Agile AI-Driven Development（Brian Madison 开源，约 12 个专用 Agent 角色）。Chorus = 开源 AI-Human 协作框架（chorus-ai.dev），PM Agent 生成 Proposal + Task DAG。

**你的理解**：现有 AI 规划工具都做了同一层——把"需求"翻译成"任务清单"。都没做下一层——**任务的渐进式更新和反馈回路**。规划是一次性生成的，但阶段3要求 WBS 渐进式、优先级在执行中调整、依赖关系可能变动。没有工具支持"计划在执行中逐步修正"。

**对工具设计的可借鉴点**：
- Chorus 的 Task DAG 比 BMAD 的 Sprint Story 更能表达依赖关系，可参考其 DAG 格式
- PM Agent 生成 + 人类审核的模式（双路径验证）是正确方向
- 但 BMAD/Chorus 都面向多 Agent 软件开发，Agent 角色体系不适用于写作场景
- 工具应自建轻量级规划能力，不依赖外部规划工具

## 四、PDCA / ATDD

### 6. PDCA（Deming 环）

**来源文件**：`04_K3_PDCA_ATDD.md`

**定义**：Plan -> Do -> Check -> Act 四步循环。由 Shewhart 提出、Deming 推广。关键在"循环"二字——Check 的输出是下一轮 Plan 的输入，不是结束。

**你的理解**：用户描述的工作方式"写第1章 -> 你读 -> 指出逻辑不通 -> 修正 -> 写第2章"本质上就是 PDCA 循环——迭代边界为每章。PDCA 不是项目管理工具，而是**任何改进活动的最小闭环**。

**对工具设计的可借鉴点**：
- 工具中的每个任务/章节执行应内嵌 PDCA 循环，而非线性终检
- Act 阶段的输出应自动成为下一轮 Plan 的输入——工具应跟踪"从上一轮学到了什么"
- 循环的粒度可配置（按章/按节/按知识块），默认推荐为"按章"

---

### 7. ATDD（验收测试驱动开发）

**来源文件**：`04_K3_PDCA_ATDD.md`

**定义**：先写验收标准，再做工作。先想"怎么算写对了"再开始写。验收标准是事前定的，对标真实需求，而非事后编的倾向于低标准。

**你的理解**：ATDD 的效力边界取决于验收标准的**本质深度**而非形式存在。用户用 Remotion 实践验证："AI 做出来的分镜看起来满足标准，但实质是一坨不能落入生产的狗屎。"验收标准只定表层指标（"有动画"）则 AI 自检必然通过——但验收标准定深层指标（"分镜有叙事表达力"）则 AI 自检不了，需要人类判断。

**对工具设计的可借鉴点**：
- 验收标准必须区分**形式标准**（机器可检：格式/符号表/引用完整）和**本质标准**（人类判断：叙述连贯性/表达力）
- 工具应默认提供每个任务的验收标准模板（包含形式标准和本质标准两列）
- 阶段1的成功标准应翻译为阶段3 WBS 的工作包级别验收标准——保持维度一致，但给执行中的调整留空间（不要写太具体绑死灵活性）
- 折中方案：阶段1定义原则和维度，阶段3写出每个工作包的具体验收标准

## 五、审核机制

### 8. 三级审核清单（DO-CONFIRM / READ-DO / HOOK）

**来源文件**：`05_K4_审核机制.md`

**定义**：来自 Atul Gawande《清单宣言》（2009）。三层级嵌套使用：
- **DO-CONFIRM**：自己做完后自己对照清单复查（人类判断：表达力、叙述逻辑）
- **READ-DO**：另一个人/AI 对照清单逐项检查并报告（AI 可执行：符号表一致、公式数量匹配）
- **HOOK**：提交时自动触发（全自动化：格式规范、引用完整性）

**你的理解**：三层不是选择题——是嵌套使用。用户的 Remotion 经历验证了"三个层级缺一个都会出问题"（DO-CONFIRM 没做、HOOK 没做、READ-DO 做了但标准太浅）。三级审核天然对应两种标准的检查方式——HOOK 和 READ-DO 处理形式标准（AI 执行），DO-CONFIRM 处理本质标准（人类判断）。

**对工具设计的可借鉴点**：
- 审核机制应分三层实现：自动检查（HOOK）、AI 自检（READ-DO）、人类审查（DO-CONFIRM）
- 每层的检查项应明确标注属于哪一层级——不可混用
- READ-DO 的输出应结构化（每项通过/失败/不确定），而非开放式描述
- HOOK 应整合到工具的提交/写入流程中，自动触发

---

### 9. 双路径验证

**来源文件**：`05_K4_审核机制.md`

**定义**：AI 自检（第一轮）+ 人类审查（第二轮）+ 合并结论。AI 自检报告含形式标准项（通过/失败/不确定）和本质标准项（标注"此条需要人类确认"）。人类的不是从头查——是**审 AI 的自检报告**。

**你的理解**：AI 自检和人类审查各自有盲区，每轮只做自己擅长的事，互相不能替代。AI 看不到本质标准（表达力、叙述逻辑），人类会疲劳漏检。不设置 AI 自检层 -> 人类被格式检查填满；不设置人类审查层 -> 形式过关但实质不过关。

**对工具设计的可借鉴点**：
- 双路径验证应作为工具中审核步骤的标准流程
- AI 自检报告格式：每个检查项独立行，标注"通过/失败/不确定"，本质标准项加"需要人类确认"标签
- 人类审查的 UI/接口应呈现 AI 的自检报告而非原始交付物——降低人类审查的认知负荷

---

### 10. 阶段4审核 = 检查门 + 学习门

**来源文件**：`学习笔记.md`（决策 10）

**定义**：阶段4的审核不仅有"检查当前交付物质量"的职能，还应承载"让用户通过审核过程理解项目原理，积累领域经验，以便下次定义更精确的标准"的学习职能。

**你的理解**：用户提出的"在执行过程中增进理解，理解后才能知道遇到类似问题时能用什么思路解决"——这是对阶段4审核职能的根本性扩展。用户参与审核不只是"把关当前批次质量"，更是通过审核过程建立"参与审核 -> 理解原理 -> 能定更好的标准"的能力闭环。

**对工具设计的可借鉴点**：
- 工具中的审核步骤不只要给"通过/不通过"结论——还应输出"学到了什么，下次可以改进什么标准"
- 审核模板增加"这次审核让我意识到"字段，鼓励用户在审核中做认知积累
- 长期来看，用户的审核经验应能反哺到框架/工具的上游（阶段1的标准定义）

## 六、文件化工作流

### 11. 文件化 > 对话化

**来源文件**：`02_K1_文件化工作流.md`

**定义**：2025-2026 年所有主流 AI 编码平台（Claude Code、Cursor、GitHub Copilot）的坚固共识：重要的上下文和规则必须存储在文件中而非对话中。对话会随上下文增长衰减或被截断，文件不会。

**分层共识**：宪法级（始终加载）-> 流程级（任务触发）-> 事件级（事件触发/Hooks）-> 路径级（按代码位置）。

**你的理解**：项目已有 CLAUDE.md + Skills 的文件化方向符合行业共识，但存在差距：缺少 `@imports` 或 `.claude/rules/` 这类规则分片机制（所有规则要么在 CLAUDE.md 始终加载，要么在 SKILL.md 整体加载，粒度不够细）；Hooks 未与规则系统深度整合；没有路径级规则；上下文预算从未系统审计。

**对工具设计的可借鉴点**：
- 规则存储必须文件化（禁止仅靠对话记忆）
- 规则应分层存放：宪法级（核心约束）、流程级（阶段流程）、事件级（操作触发）、路径级（按目录/场景触发）
- 各层应有独立的加载策略（宪法级始终加载，流程级按需加载，事件级条件触发）
- 上下文预算应被审计——工具应统计当前已加载规则的总 token 消耗

## 七、Context Engineering

### 12. 注意力预算与三层上下文结构

**来源文件**：`03_K2_Context_Engineering.md`

**定义**：LLM 的上下文不是无限池子——随 token 增长，模型呈现"边际收益递减"。Context Engineering 的目标是找到"最小的高信号 token 集合"。

Anthropic 三层上下文结构：
- **System Prompt**：宪法级规则，简洁明确
- **Tools**：最小可行集
- **Examples / Few-shot**：精选多样化典型样例

长程任务的三大技术：Compaction（压缩后重启新对话）、Structured Note-Taking（关键状态写入外部文件）、Sub-Agent Architectures（子 Agent 聚焦任务，每个保持干净上下文窗口）。

**你的理解**：注意力稀释的根本原因是 Softmax 的数学特性——所有权重之和永远等于 1，token 越多每个 token 分到的注意力越少。Lost in the Middle（U 形曲线：开头和结尾性能好，中间差）让中间位置的信息沉没。规则必须分层、必须精简——不是所有规则都始终加载。

**对工具设计的可借鉴点**：
- 工具应内置"最小高信号 token 集合"的意识——每次调用只加载当前阶段相关的上下文
- 支持 Just-in-time retrieval（Agent 只维护轻量标识符，运行时动态加载）
- 长任务超出上下文限制时，应自动触发压缩并重启对话
- 跨阶段/跨会话的决策传递应只传递"已审批摘要"，不传递全部上下文

### 13. 五角色上下文包模型（arXiv 2604.04258）

**来源文件**：`03_K2_Context_Engineering.md`、`04_背景知识_LLM工作原理与三层工程范式.md`

**定义**：每条上下文信息分配一个优先级角色：
- P1 **Authority**（权威）：需求文档、设计约束、审批签名
- P2 **Exemplar**（范例）：模板、历史报告、成功同类作品
- P3 **Constraint**（约束）：字数限制、API 版本要求
- P4 **Rubric**（评估标准）：DoD、质量门条件
- P5 **Metadata**（元数据）：项目名、截止日期、用户 prompt

**关键发现：Metadata Paradox**——用户的 prompt 本身永远是最低优先级的元素（Metadata）。当用户的临时指令与 Authority 文档冲突时，Authority 胜出。这不是"忽略用户"，而是保障系统性一致。

**你的理解**：最颠覆认知的发现——"我对 AI 说的话是最重要的指令"这个直觉是错的。当用户说"先别调研直接写代码"时，框架的 Authority 规则（"阶段退出条件不满足禁止推进"）应覆盖这个临时指令。框架从未显式定义过优先级，当 CLAUDE.md 和用户 prompt 冲突时没有裁决规则。

**对工具设计的可借鉴点**：
- 工具必须显式定义规则优先级——CLAUDE.md > SKILL.md > 用户临时指令
- 高优先级规则覆盖低优先级规则——当冲突时告警或自动裁决
- Metadata Paradox 的工程含义：用户的临时请求不应轻易覆盖框架的 Authority 规则
- 规则检查清单每条都可标注其角色（Authority/Exemplar/Constraint/Rubric/Metadata）以明确优先级

### 14. Prompt Engineering -> Context Engineering -> Harness Engineering 三层范式

**来源文件**：`04_背景知识_LLM工作原理与三层工程范式.md`

**定义**：三者不是同一条线上程度加深，而是抽象层级的跃迁：
- **Prompt Engineering**（2022-2024）：怎么问 AI 才能得到最好的回答
- **Context Engineering**（2025）：该让 AI 看到什么信息才能做出正确决策
- **Harness Engineering**（2026）：怎么设计 AI 的整个运行环境，让它稳定、可靠地自主工作

**类比**：写一封完美的邮件（Prompt）-> 给邮件附上所有需要的文件（Context）-> 设计整间办公室的布局、规则、流程（Harness）。

**你的理解**：不能把后两者归为 Prompt Engineering——Context Engineering 涉及动态组装上下文（RAG 检索、文档拼接、工具定义），单一 prompt 做不到。Harness Engineering 涉及 AI 外部的运行环境（linter、Generator/Evaluator 分离、CI/CD 反馈回环），改变的不是 AI 看到什么，而是 AI 所处的环境有什么约束和回馈。

**对工具设计的可借鉴点**：
- 工具设计应在三层都有投入——只做 Prompt 层不够
- Context 层要管好注意力预算——工具应设计上下文的"组装-加载-卸载"策略
- Harness 层要管好运行环境——工具应设计反馈回路、护栏、可观测性

## 八、Harness Engineering

### 15. Agent = Model + Harness

**来源文件**：`05_K3_Harness_Engineering.md`、`06_扩展_Harness_Engineering深入.md`（D08）

**定义**：由 Mitchell Hashimoto（Terraform 创始人，2026.2）提出：Model 处理推理（正在快速商品化），Harness 是模型之外的一切（真正拉开差距的地方）。核心原则："每次发现 AI Agent 犯了一个错误，你就在它的运行环境中设计一个解决方案，让它永远不能再犯同样的错误。"（Ratchet 原则）

**你的理解**：项目一直在做 Harness（CLAUDE.md、Skills、四阶段流程都是在设计 AI 的运行环境），但从未意识到自己在做 Harness Engineering，也就没有用 Harness 原语指导设计。"Agent 搞砸了不是因为模型不够聪明——是因为上下文腐烂、工具描述歧义、缺少门控检查、会话之间状态丢失。"

**对工具设计的可借鉴点**：
- 五大核心支柱的覆盖度是工具设计的检查框架：Tool Orchestration / Guardrails / Feedback Loops / Observability / Human-in-the-Loop
- 项目 14 条规则中 9 条是软约束（提示词级"请遵守"），缺少**机械强制**（linter/CI 自动拦截/Hook 拒绝执行）
- 机械强制 vs 提示词约束的差距是工具设计中最值得投入的方向

---

### 16. 前馈控制 vs 反馈控制

**来源文件**：`06_扩展_Harness_Engineering深入.md`（D08）

**定义**：来自 Fiddler AI（2026.6）：
- **前馈控制（Feedforward）**：在 Agent 行动之前引导它——系统提示词、Skill 文件、工具允许/阻止列表、CLAUDE.md。减少可能犯错的空间。
- **反馈控制（Feedback）**：在 Agent 行动之后评估它——Linter、测试、LLM-as-Judge、CI 门控。捕捉前馈控制漏掉的错误。

两层都需要的根本原因：模型是概率性的，无法通过纯指令穷举所有失败模式。

**你的理解**：项目护栏层全是前馈（提示词约束）、反馈全依赖 HITL（用户审查）——这是当前最大的结构弱点。反馈控制目前完全依赖人的判断力，缺少自动化信号。

**对工具设计的可借鉴点**：
- 工具在前馈层：设计清晰的上下文组装策略、工具权限定义、规则文件加载机制
- 工具在反馈层：设计自动化质量信号（格式检查、完整性验证、冲突检测）不依赖人
- 反馈控制不一定是测试套件——对非代码项目，可以是结构检查、公式一致性验证、引用完整性检查

---

### 17. Control Surface（控制表面）与强制执行硬度

**来源文件**：`06_扩展_Harness_Engineering深入.md`（D08）

**定义**：控制不是集中式面板而是分布式的"可挂杠杆的地方"。强制执行有三个硬度等级：硬（sandbox/CI 门控）-> 中（Hooks/typed tool schema）-> 软（自然语言指令）。

**你的理解**：D07 发现 14 条规则中 9 条是软约束——真正的问题不是"我有没有规则"，而是"规则有没有机械强制力"。工具设计应问：哪些规则应升级到"中硬度"？哪些硬约束不值得投入？

**对工具设计的可借鉴点**：
- 每一条规则都应有明确的硬度等级标签（软/中/硬）
- 设计时应优先将高频失败模式的约束升级到中或硬
- 中硬度的 Hooks 是投入产出比最高的——不需要改动基础设施，但能提供程序化的自动检查

## 九、7 大 AI Agent 设计模式

### 18. 7 大模式汇总与框架对照

**来源文件**：`06_K4_7大设计模式.md`

**定义**：7 大模式是 Agent 行为的可复用架构模板。不是互斥的——每个阶段可用不同模式组合。

| 模式 | 机制 | 对应框架位置 | 状态 |
|:----|:----|:------------|:----:|
| Reflection | 生成->自评->修正循环 | 调研阶段的搜索循环 | 可改进（缺收敛阈值） |
| ReAct | 推理与行动交替（思考->行动->观察） | 阶段2（全面调研） | 良好（核心使用） |
| Plan-and-Execute | 分解目标为子任务后逐步执行 | 阶段3->阶段4 | 良好 |
| Tool Use | 工具作为认知扩展 | 搜索链、MCP 工具 | 可改进（未治理） |
| Multi-Agent | 专门化 Agent 各司其职 | 阶段2的两路线并行 | 雏形 |
| Memory | 外部存储维持跨会话状态 | 版本记录/续学指令 | 可改进 |
| Human-in-the-Loop | 高风险节点插入人工审核 | 阶段切换用户确认 | 良好 |

**你的理解**：项目的模式使用是"隐式"而非"显式"——已经在用 ReAct、Plan-and-Execute、HITL，但从不在设计文档中标注。Reflection 缺口最严重——质量控制完全依赖外部的阶段审核，缺少每个阶段内部的自评->修正机制。

**对工具设计的可借鉴点**：
- 每个阶段/步骤应显式标注当前使用的设计模式——方便调整时参考理论框架
- 缺少 Reflection 模式——工具应为每个阶段设计自评循环（生成->检查->修正->迭代上限）
- 7 大模式可转化为工具的功能模块清单：哪些已实现、哪些可改进、哪些缺失

## 十、约束规则与关键发现

### 19. 五类约束规则

**来源文件**：`07_K5_约束规则与关键发现.md`

**定义**：14 条规则分为五类：
1. **防幻觉规则**（3 条：搜索验证/来源追溯/URL 标注）——2 条提示词规则 + 1 条可机械强制
2. **文件化规则**（3 条：先写文件/状态文件存在性/版本记录只追加）——2 条提示词 + 1 条可机械强制
3. **阶段推进规则**（3 条：阶段切换用户确认/退出条件/跳过风险告知）——全依赖 HITL
4. **质量标准规则**（3 条：模板结构/缺少小节退回/EARS 标注）——全依赖人工判断
5. **上下文管理规则**（2 条：SKILL.md < 200 行/阶段指令独立文件）——1 条可自动化 + 1 条架构设计

**你的理解**：14 条规则中 9 条是提示词规则（软约束），5 条可机械强制的内容检查完全未实现。每条规则理论上都有对应的检查方式（文件存在性、内容模式匹配、行数限制），只是当前没有实现。这解释了"规则看着都很好但 AI 就是不好好遵守"的根因。

**对工具设计的可借鉴点**：
- 规则的检查机制不应是提示词级"请遵守"——应为每条规则设计对应的自动化检查
- 优先级实现检查的三条：来源标注（文件末尾是否有 URL）/ 阶段产出文件存在性 / 文件行数告警
- 规则状态统计表是工具中"规则健康度"的检查原型——统计软/硬约束比例

## 十一、Chorus 查重

### 20. Chorus 深度分析

**来源文件**：`02_K1_Chorus.md`

**定义**：Chorus（chorus-ai.dev, AGPL-3.0）——最接近的四阶段框架项目。工作流：Idea -> Proposal -> [Document + Task DAG] -> Execute -> Verify -> Done。PM Agent 通过结构化 Q&A 澄清需求，生成 Proposal（含文档草案和任务 DAG）。

**巧妙之处**：Reversed Conversation（AI 提案人类验证——反转了人指令 AI 执行的权力关系）、任务 DAG（支持依赖图、并行路径、循环检测）、权限矩阵（5 资源 x 3 操作的精细控制）、双路径验证（Dev 自检 + Admin 独立审查，每条证据结构化）。

**你的理解**：Chorus 和本框架是"同一母题的两种具体化"——都在回答"AI 和人怎么结构化协作"，但路线不同：Chorus 走工具路线（软件产品管流程），本框架走方法论路线（规则定义流程）。互补而非竞争——可串联使用。

**对工具设计的可借鉴点**：
- **Reversed Conversation 模式**：当用户输入高度模糊时，先让 AI 生成需求理解草案供确认，而非直接进入完整流程
- 双路径验证的结构化证据设计——验收清单增加"通过/失败证据"字段
- Chorus 无调研阶段验证了本框架的差异化定位——这是关键优势

### 23. 方法论层 vs 基础设施层

**来源文件**：`04_K3_结论与定位.md`

**定义**：
- **方法论层（Agent Methodology）**：回答"编排之前应该完成什么阶段、每个阶段产出什么"。典型句子："先做需求锚定确认用户要什么，调研一下有无现有解决方案，再做计划。"
- **基础设施层（Agent Harness）**：回答"怎么编排 Agent、用什么工具"。典型句子："这个工作流用 3 个 Agent 并行，第 1 个搜索，第 2 个摘要，第 3 个写报告。"

**类比**：Excel（基础设施层）和会计准则（方法论层）——你做账用 Excel，会计准则告诉你该怎么记账。同样，Chorus/Conductor（基础设施层）和本框架（方法论层）——框架定义"在编排之前应该完成什么阶段"，Chorus/Conductor 实际执行这些阶段的编排。

**你的理解**：79+ 个工具教你怎么编排 Agent，0 个工具告诉你编排之前应该完成什么阶段。赛道已确认，方法论层是空白。但"方法论层空白"不意味着别人做不了——轻量方法论的优势是低门槛，劣势是缺乏机械强制力。

**对工具设计的可借鉴点**：
- 工具定位应明确在方法论层而非基础设施层——不造轮子（Harness 已有 79+），写驾驶手册
- 工具应与基础设施层工具可集成/可串联——不是取代关系
- 工具设计的核心价值是"告诉 Agent 和用户该完成什么阶段"，而非"提供编排引擎"

---

### 24. 通用方法论种子定位

**来源文件**：`05_补充调研_行业独立调研框架.md`、`学习笔记.md`（决策 6、7）

**定义**：6 个领域特化框架（RPI/AutoR/AutoResearchClaw/ResearStudio/QUIT/multi-agent-research-synthesis）各自在自己领域里独立实现了"调研前置、人审兜底"。每个都是自己领域的完整大树。本框架的定位是"通用种子"——不是大树的未完成版，而是"可以根据不同土壤长出不同类型大树"的起点。

**3 个核心需求**（导师分析 / 高效学习流 / 视频工作流）就是三片不同土壤，用于测试种子的通用性。

**你的理解**：这一定位有两层含义：（1）参考和学习领域特化框架是必要的——不能闭门造车（可以从它们提取 FAR 验证、SmartPause、产物驱动信息传递等设计精华）；（2）通用种子的价值来自"可以根据需要发展出符合自己的工具"——不是为了通用而通用，而是因为用户有多个不同类型需求共享底层逻辑。

**对工具设计的可借鉴点**：
- 工具不应追求"一次性覆盖所有场景"——应设计为可扩展的通用骨架，领域细节由用户按需填充
- 6 个领域框架的设计精华可直接提取（不需从零设计）：
  - RPI：FAR/FACTS 验证标准、每阶段新建会话
  - AutoR：已审批摘要作为唯一跨阶段记忆、恢复与回滚
  - AutoResearchClaw：SmartPause（只在不确定性高时问人）
  - ResearStudio：两种协作模式的灵活切换
  - QUIT：产物驱动的跨阶段信息传递
  - multi-agent-research-synthesis：三级验证升级链
- 三大核心需求应作为首批实践验证项目，每个需求在框架四阶段中走一遍

## 十四、Harness Engineering 深入

### 25. Ratchet 原则

**来源文件**：`06_扩展_Harness_Engineering深入.md`（D08）

**定义**：由 Mitchell Hashimoto 提出——"Every mistake becomes a permanent rule." 操作流程：Agent 犯错 -> 追溯根因到 Harness 的某个层 -> 在该层编码预防规则 -> 该错误不能再发生。

**你的理解**：每一条规则背后都应该能追溯到一条真实的历史失败。14 条规则中有多少条可以追溯到具体的失败？如果一条规则写出来只是"我觉得应该这样"，它很可能在执行中被绕过。Ratchet 原则是"规则是活的、有根的、有历史的"。

**对工具设计的可借鉴点**：
- 工具中的每一条规则/约束应绑定其"来源失败记录"（哪天/什么场景/什么错误触发的）
- 新规则的提出应要求提供这个"来源失败"——否则不入规则库
- Ratchet 的外循环实现：AI 记录摩擦点 -> AI 提议根因 -> 人判断是否合理 -> 人确认后执行修改+备份+版本记录。**决策权在人手里**

---

### 26. 三层嵌套循环

**来源文件**：`06_扩展_Harness_Engineering深入.md`（D08）、`07_扩展_Loop_Engineering深入.md`（D08）

**定义**：
- **内层循环（Agentic Loop）**：Agent 在代码中运行——think -> act -> observe 的核心循环。时间尺度：秒到分钟。
- **中层循环（Supervisory Engineering）**：人在会话中操控 Agent——读 diff、改方向、批准/拒绝。时间尺度：分钟到小时。
- **外层循环（Harness Engineering / Loop Engineering）**：你在会话之间改变 Harness 的表面——规则文件、Hooks、工具定义——使得内层和中层跑得更好。时间尺度：天到周。

**Ratchet 在此的位置**：内层循环出错 -> 中层循环发现 -> 外层循环编码修复。

**你的理解**：本框架的外循环必须是"人决策 + AI 执行"——禁止 AI 自嗨式自我迭代。用户明确强调："我很怕 AI 不知道什么是好什么是坏，如果让 AI 自己自嗨的话，可能陷入越来越偏离实际需求的状态。"这与 D01 对 AgentFactory 全自动 Self-Evolve 的拒绝一脉相承。闭环的决策权在人手里，不在 AI 手里。

**对工具设计的可借鉴点**：
- 工具的外循环架构应是：AI 记录摩擦点 -> AI 提议根因 -> 人判断 -> 人确认后执行
- 禁止 AI 自主修改框架/方法论规则——这是宪法级约束
- 内外循环的分工：内循环 Agent 自主（think-act-observe），中层人类操控（审批/纠正），外层人类决策（规则修改）

---

### 27. 三层模型最终版

**来源文件**：`学习笔记.md`（决策 11）

**定义**：
1. **框架层（project-pilot）**：决定"完成什么阶段、产出什么"。外循环——每次项目后，人决策 -> AI 执行指导文件更新。
2. **Harness 层（Claude Code + 阶段2选型的工具）**：提供项目的完整 Agent 运行环境。内循环的支撑基础设施。
3. **项目层（具体产出物）**：内循环——Agent 执行 -> 反馈 -> 验证 -> 修正。Ratchet 编码进项目规则。

三层不是单向流水线——框架从项目中学（外循环）、项目内部自我改进（内循环）、Claude Code 提供 Harness 能力但不决定方法。

**你的理解**：原来 D01 的三层架构是"框架 -> Claude Code -> 项目"单向流水线，现在更新为"每层都有独立反馈回路"。阶段2不再只是"调查工具"——本质上是为该项目的 Harness 层做组件选型调研（哪些现成工具可直接用、哪些可组合、哪些需要自己设计）。

**对工具设计的可借鉴点**：
- 工具的设计结构应体现这三层的分层和独立反馈回路
- 阶段2的执行指令应补充"Harness 组件选型"维度——选型评估框架使用 Ratchet 能力/前馈控制成熟度/反馈控制完备性
- 每层的反馈回路应独立运作——框架层不干涉内循环细节，项目层不越级修改框架规则

## 十五、Loop Engineering 深入

### 28. Loop 的五个组件 + 一个外部记忆

**来源文件**：`07_扩展_Loop_Engineering深入.md`（D08）

**定义**：Addy Osmani《Loop Engineering》（2026.6.7）：
- **自动化**：按计划发现工作+分诊——循环的心跳。没有自动化，循环只是一次单独运行。
- **工作树**：隔离并行 Agent 之间的文件冲突。
- **Skills**：把项目知识写成文件，不每次重新解释。无 Skill 时每次从零推导项目。
- **连接器**：让循环触碰真实工具——Issue 追踪器、数据库、API、Slack。
- **子 Agent**：写的人不能给自己的作业打分。分离制造者和检查者。
- **外部状态**：循环之间保持记忆，在会话上下文之外持久化。

**你的理解**：Claude Code 和 Codex 都有全部五件套——名称不同，能力相同。一旦注意到形状是一样的，就不再争论哪个工具更好——而是设计一个在任何工具里都能跑的循环。

**对工具设计的可借鉴点**：
- 工具的自动化支撑：按间隔/按条件触发的循环机制
- 子 Agent 独立验证：执行者和检查者必须分离
- 外部状态持久化：Agent 会忘，文件不会——所有状态走文件系统

### 29. Loop 四个硬标准

**来源文件**：`07_扩展_Loop_Engineering深入.md`（D08）

**定义**：不是"把 Agent 按计划多跑几次"——那只是更昂贵的失败方式：
1. **反馈信号**：环境必须回应 Agent 的行动（测试失败、类型检查报错）
2. **自我纠正**：Agent 诊断失败原因再调整——Loop 和 Retry 的区别（Retry 重复，Loop 转向）
3. **独立验证**：写的人不能给自己的作业打分——确定性验证优于 LLM-as-Judge
4. **持久状态**：循环之间的记忆在文件系统上，不在上下文窗口中

Addy Osmani 的实践数据：每次重试前强制 30 秒反思提示词——减少 67% 卡住的 Agent。

**四个终止条件**：验证通过 / 迭代上限 / 预算上限 / 无进展检测（最关键——PID 控制器的导数项，看错误的变化率而非绝对值）。

**你的理解**：最核心的缺失是 Ratchet 的编码动作本身还没有自动化——框架没有机械能力去"读反馈、提取教训、修改模板"。项目已有"版本记录零省略"原则，但从版本记录到"下一次自动受益"的编码过程全依赖人工。

**对工具设计的可借鉴点**：
- 循环应包含四个硬标准的实现检查——工具可评估自己的循环是否"真循环"而非"伪循环"
- 无进展检测是最关键的终止条件——工具应为长时间运行的任务设置无进展上限
- 循环的反馈信号应在工具层面提供——不依赖外部人工判断
- 工具应实现"失败诊断+转向"而非简单重试

## 设计哲学层面的总纲建议

1. **不做基础设施层工具**——79+ 项目已在做编排/运行时/框架。工具的价值在方法论层（告诉用户和 Agent 该完成什么阶段），不在"又一个编排引擎"。

2. **规则的硬度比规则的数量重要**——14 条软约束不如 3 条机械强制的规则。工具应优先将高频失败模式的约束升级到"中硬度"（Hooks/程序化检查）。

3. **人决策 + AI 执行作为外循环铁律**——工具可以记录摩擦点、提议根因、执行修改，但框架的规则更新必须经过人的判断确认。

4. **三层审核不取代、不混用**——自动检查（格式/引用完整性）、AI 自检（形式标准每项通过/失败）、人类审查（本质标准判断）。层层的输出是下一层的输入。

5. **种子的定位优于大树的定位**——通用骨架 + 领域适配能力 > 一次性覆盖所有场景的工具。三大核心需求就是三片土壤，用于验证和迭代工具。
