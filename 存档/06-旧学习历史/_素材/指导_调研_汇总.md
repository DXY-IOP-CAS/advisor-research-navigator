## 1. 模块 1-8 各模块核心结论

### 模块1：需求锚定方法（来源：模块1_需求锚定方法.md）

**核心发现**
- 需求锚定不是简单的"对话+确认"，应有标准化内部子步骤：需求诱导（Elicitation）→ 需求分析（Analysis）→ 需求规格（Specification）→ 需求验证（Validation）
- EARS 标注法（Mavin et al., 2009）：五种标准句型（Ubiquitous/Event-driven/Unwanted behavior/State-driven/Optional），每条需求必须符合其一，防止模糊表达
- Context Engineering 五角色模型（arXiv 2604.04258, 2026.4）：Authority + Exemplar + Constraint + Rubric + Metadata，可用来验证框架完整性
- AI 在需求诱导阶段最有价值，但最终确认必须由人类完成
- 半结构化访谈 + 五问法：先问动机（为什么）→ 再问行为（怎么做）→ 最后问需求（什么）；连续追问"为什么"直到触及根本

**框架设计决策**
- 需求规格书模板骨架包含：项目动机、目标受众（用户故事格式）、成功标准（可检查指标）、硬性约束（EARS格式）、妥协空间、参考范例、版本记录
- 四轮对话策略：动机挖掘 → 受众与标准锚定 → 约束与边界明确 → 成功标准定义 → 最终确认轮

**对导师调研工具的参考价值**
- 导师匹配本质上是一个需求锚定问题：学生"想找导师"可能是模糊表达，需要像四轮对话一样先挖掘真实动机（为什么想找这个方向的导师？以后想做什么？）
- EARS 标准句型可以用于定义导师选择条件（"WHEN 该导师近3年有顶刊论文，且课题组有计算资源，THEN 列入候选名单"）
- 需求规格书的格式可以迁移为"导师匹配规格书"

### 模块2：全面调研方法（来源：模块2_全面调研方法.md）

**核心发现**
- 系统文献综述（SLR）四步法：Scoping（范围界定）→ Searching（搜索）→ Screening（筛选）→ Reporting（报告）
- 两路线调研策略：路线A领域调研（别人做到什么程度了）和路线B工具/资源调研（有什么现成的可以借力），并行执行后在调研报告中交汇
- 调研质量四维检查：覆盖率（≥3个数据源）、时效性（优先近6个月成果）、正确性（间接引用追溯原始来源）、完整性（正反两面+未覆盖方面）
- 现有AI调研工具（Elicit、Consensus、Semantic Scholar、SciSpace、Research Rabbit）各有限制，需组合使用
- Deep Research 工具：OpenAI Deep Research、Google Deep Research、Perplexity Deep Research、Claude Code Skills 的 deep-research

**框架设计决策**
- 调研报告模板包含：搜索范围策略、领域现状（直接相关/部分相关/启发性）、工具评估表、关键发现、未覆盖方面、验证来源
- 质量检查清单作为阶段退出的硬性门控

**对导师调研工具的参考价值**
- 导师调研就是路线A（导师本人研究领域现状+论文水平+课题组声誉）和路线B（哪些工具/平台可以辅助搜索导师）的交汇
- SLR 四步法可以直接用作导师搜索策略的骨架：先确定搜索范围（学科、地域、时间段）→ 并行搜索多个数据源（Google Scholar、知网、arXiv、ResearchGate）→ 筛选（读主页+论文标题）→ 报告（生成候选列表）
- 调研质量检查清单可迁移：是否搜索了≥3个数据库？是否覆盖了学术+同行推荐+课题组官网三类来源？是否验证了关键信息的一致性？
- 正反两面发现尤其重要：不仅要找导师的优点（论文、资源、声望），还要标记潜在风险（近期产出少、学生反馈差、即将退休）

### 模块3：计划执行审核（来源：模块3_计划执行审核.md）

**核心发现**
- WBS（工作分解结构）+ MoSCoW 优先级法（Must/Should/Could/Won't）是计划阶段的核心方法论——先分解再排优先级
- PDCA 循环（Plan → Do → Check → Act）> 线性执行——执行不是一次性的"写完了检查"，而是"写一部分→检查→调整→再写"的循环
- ATDD（验收测试驱动开发）：阶段1定义的成功标准，必须在阶段4被逐条对照验收，不是写了就完事
- 双路径验证（来自 Chorus）：AI 自检 + 人类独立审查，每条验收标准有结构化的通过/失败证据
- 审核三层级：DO-CONFIRM（人类自查）< READ-DO（AI逐项检查）< HOOK（自动化检查）
- 回顾机制（Retrospective）：每次迭代后回答"什么做得好/什么做得不好/下次试什么"，驱动框架自我进化

**框架设计决策**
- 执行计划模板包含：总体路线图、任务分解（WBS，含输入/输出/验收标准/优先级/依赖）、里程碑、风险与应对
- 收工审核报告模板包含：成功标准对照表、质量门检查、已知局限、后续建议
- 框架本身也需要回顾——每次项目完成后追加回顾记录

**对导师调研工具的参考价值**
- 找导师本身可以视为一个项目，需要 WBS 分解：确定研究方向→列出候选导师→逐个调研→写套磁信→跟进→面试→决策
- MoSCoW 优先级用于导师排序：Must have（研究方向匹配+有经费）、Should have（年轻有干劲+好说话）、Could have（名气大）、Won't have（常年不带学生）
- 双路径验证适用于导师信息核验：AI 收集的信息（引用数、论文列表）需要人工验证（联系师兄师姐、读论文确认方向是否匹配）
- 回顾机制：每次套磁后记录"什么策略有效/什么无效"，优化下一轮

### 模块4：AI协作规范（来源：模块4_AI协作规范.md）

**核心发现**
- Harness Engineering 已是一个独立领域（2026年涌现），有专属的 awesome 列表、术语体系、设计原语
- 文件化 > 对话化：2026年最坚固的共识——任何"靠记忆遵守"的规则最终都会被遗忘
- Context Engineering 五角色模型（Authority/Exemplar/Constraint/Rubric/Metadata）提供了一个完美的规则完整性验证方法
- 7 大 Agent 设计模式（Reflection/ReAct/Plan and Execute/Tool Use/Multi-Agent/Memory/Human-in-the-Loop）不是互斥的，每个阶段可用不同模式组合
- Anthropic 官方指南（2025.9）：分层上下文、规则闭环、上下文预算意识、Be specific not polite
- Addy Osmani 的工作流：先写 Spec 再写代码、小步快跑、上下文边界明确、人工审核关键节点

**Harness Engineering 7 原语**：
| 原语 | 对应框架 |
|------|---------|
| Session Lifecycle | CLAUDE.md + 续学指令 |
| Task State Machine | 四阶段状态机 |
| Context Injection | Skill 按需加载 + 模板引用 |
| Human Approval Gates | 阶段退出条件 + 用户确认 |
| Proof of Work | 每个阶段的文档产出 |
| Observability | 版本记录 + 踩坑清单 |
| Failure Recovery | 版本回退 + 规则修正 |

**框架设计决策**
- 规则分层存放：SKILL.md（跨阶段铁律）→ 阶段执行指令（阶段行为规则）→ 模板文件（产出格式）→ 质量门规则（审核标准）
- 防幻觉规则：论文引用/公式/数据必须 WebSearch 验证再写入；间接引用追溯原始来源；外部信息标注来源 URL
- 文件化规则：所有产出先写文件再开口；阶段状态通过文件存在性判断；版本记录只追加不概括
- 上下文管理：SKILL.md < 200 行，详细规则挂在独立文件中

**对导师调研工具的参考价值**
- Harness Engineering 思维框架可以用于设计"导师调研助手"的状态机：搜索阶段→筛选阶段→深度调研阶段→联系阶段→决策阶段
- 文件化规则意味着调研过程中每一步的产出（搜索记录、论文笔记、邮件草稿）都保存为文件，不依赖对话记忆
- Human-in-the-Loop 模式是关键：AI 可以搜索和汇总导师信息，但最终评估和决策必须由学生完成
- Context Injection 原语：每次进入新的调研阶段时，只加载当前阶段所需的背景知识

### 模块5：查重确认（来源：模块5_查重确认.md）

**核心发现**
- 最接近的项目是 **Chorus**（chorus-aidlc/chorus），AGPL-3.0 开源，Next.js 15 + TypeScript + PostgreSQL + Redis + MCP SDK
- Chorus 工作流：Idea → Proposal → [Document + Task DAG] → Execute → Verify → Done
- 与我们的差异：
  - 需求锚定：Chorus 用 PM Agent 结构化 Q&A；我们用多轮动机挖掘 + EARS 标注 + 需求规格书
  - **Chorus 无独立调研阶段** → 这是我们的核心差异化
  - Chorus 需要 Docker/AWS 部署；我们 git clone 即用
  - Chorus 面向软件开发；我们跨领域
- Chorus 的巧妙设计：Reversed Conversation 模式（AI提案，人类验证）、任务 DAG（依赖图+并行路径+循环检测）、权限矩阵（5资源×3操作）、双路径验证
- 其他相关项目：Conductor（微软开源，YAML 工作流定义）、Manus（闭源，四阶段流程）、awesome-agent-harness 列表
- **查重结论：赛道正在形成，但现有方案都是基础设施层，没有"轻量方法论框架"**

**框架设计决策**
- 我们的框架与 Chorus 互补不竞争：Chorus 是"跑道和塔台"（服务器、权限、编排），我们是"飞行手册"（四个阶段怎么做、每个阶段产出什么、质量怎么检验）
- 未来可以兼容多个 Harness：Chorus 做编排后端，我们的框架做方法论前端

**对导师调研工具的参考价值**
- 导师调研工具应该走"轻量方法论框架"路线，不是重基础设施
- Chorus 的 Reversed Conversation 模式值得借鉴：AI 先提出导师候选方案，人类验证和调整，而不是人类盲目搜索
- 双路径验证：AI 收集信息 + 人类验证（联系师兄师姐核实），避免"看起来对但实际上有偏差"

### 模块6：分发形式决策（来源：模块6_分发形式决策.md）

**核心发现**
- Agent Skills 开放标准（agentskills.io）被 30+ 工具原生支持（Claude Code、Cursor、GitHub Copilot、VS Code、OpenAI Codex、Gemini CLI 等）
- 对比了 GitHub 模板仓库、Claude Code Skill、Plugin、npm/npx 包、Cookiecutter 模板五种分发形式
- Plugin 只在 Claude Code 可用，且要求打包 MCP Server/Hooks/Subagents——对规则+模板形态太重
- 纯全局 Skill 每次对话都会加载 description，太宽泛时会频繁误触发
- 模板仓库 + 内置 Skill 的方案最优：clone 即用、跨平台、框架更新不影响已有项目

- 分发形式：**GitHub 模板仓库**
- 核心引擎：`.claude/skills/项目执行/`（Agent Skills 标准）
- 基础约束：CLAUDE.md（始终加载）
- 更新机制：git upstream merge
- 第一版不需要 Plugin/MCP，框架成熟后可选项打包

### 模块7：增量验证与用户持续参与（来源：模块7_增量验证与用户参与.md）

**核心发现**
- AI 让"看起来能跑"的门槛极低，但"真正能跑"的门槛没变——增量验证是唯一防止被表面质量欺骗的方法
- 认知偏差研究（arXiv 2601.08045, 2026.1）：14名参与者的实验中，"即时满足"偏差出现156次；"Suggester Preference"（盲目信任AI）出现102次
- Fixation 偏差（固执于初始方案）回退率最高——43.4%（76次中33次最终被回退）
- TDD（测试驱动开发）在 AI 时代不是可选的优化，而是防止偏差的必须工具
- Shift Left（验证前移）：从"写→写→写→最后检查"变为"写→检查→写→检查→..."
- 用户参与分两种模式：学习型（用户理解内容）和审核型（用户只需质量过关）

**关键数据**
- 承担重型开发任务的参与者平均发生 112 次偏差行为，远高于简单任务
- 12 种偏差类别中有 12 种被开发者认为"经常"或"总是"发生
- 但行为数据揭示只有 2 种偏差与实际的回退操作有统计显著关联——开发者经常意识不到偏差在伤害他们的工作

**框架设计决策**
- 阶段4新增增量验证规则：原子任务粒度（1-2轮对话）、每完成一个原子任务报告产出+验收标准+验证结果+下一步、禁止一次生成超过3个原子任务
- 用户参与规则：每完成2-3个原子任务暂停邀请参与、每个模块完成后确认"最小理解三问"（功能理解/设计理解/质量理解）、用户明确表示不理解时禁止推进
- 质量控制规则：新产出完成后立即验证（非"之后再说"）、失败修正最多3轮、验收标准必须在模块开始前定义好

**对导师调研工具的参考价值**
- 认知偏差研究对导师调研同样适用：学生可能"看起来匹配"就高兴地发邮件了，没有验证关键信息（近3年是否活跃带学生、经费是否充足、个人性格是否合得来）
- 增量验证策略可以迁移为"逐层筛选"：先粗筛（方向匹配）→ 再中筛（论文质量+课题组资源）→ 最后精筛（联系在读学生确认）——每层有自己的验收标准
- 最小理解三问迁移到导师调研：候选导师是做什么方向的？（功能理解）为什么这个方向适合我而不是另一个？（设计理解）怎么确认他近年还在活跃做这个？（质量理解）
- Shift Left 策略：发现问题越早越好——不要在发出套磁信后才意识到方向不匹配

### 模块8：框架演进与防臃肿（来源：模块8_框架演进与防臃肿.md）

**核心发现**
- RFC 的 "Obsoleted by" 机制是框架臃肿最优雅的解决方案——新版本取代旧版本，旧版本存档但不活跃；不是追加，是替换
- Changelog 必须有 "Deprecated" 和 "Removed" 两个类别——没有它们，规则只会积累不会删除
- KISS 原则的"举证责任倒置"：每条新规则必须有真实踩坑记录为证据，不是"我觉得应该加"
- Linux 内核编码风格文档（~700行）的启示：规则分两类（硬约束/推荐）、规则<20条易遵守>40条人开始跳过、每条规则附带"为什么"
- 规则分级 + 升降级机制（EXP → REC → CORE）让好规则自然沉淀、坏规则自然淘汰

**框架设计决策**
- 规则三级：`[CORE]`（删掉无法运转）、`[REC]`（提高质量可跳过）、`[EXP]`（新增待验证）
- 升降级流程：`[EXP]` 连续3次迭代有效→升级为 `[REC]` → 连续2次MAJOR升级有效→升级为 `[CORE]`；`[REC]` 连续3次未触发→降级→再未触发→移除
- 30 条规则硬上限 + 一进一出机制：接近上限时加一条新规则就必须废弃一条旧规则
- 版本格式：MAJOR.MINOR.PATCH + Changelog（Added/Changed/Deprecated/Removed/Fixed）
- 新旧版本关系：旧版本头部标注 "Obsoleted by vX.Y.Z"
- 区分了"规则内容"（新版本取代旧版本，防臃肿）和"为什么改"（追加模式保留，防遗忘）

**对导师调研工具的参考价值**
- 导师调研工具同样面临"功能膨胀"风险：加新功能很容易（再加一个数据源、再加一种匹配算法），但需要防臃肿机制
- 可以设置功能分级：核心（搜索+筛选+联系模板）、推荐（论文深度分析+导师排名）、实验（AI导师推荐）
- 30条功能上限 + 一进一出：确保工具始终轻量

## 2. 框架设计规范汇总

来源：汇总_框架设计规范.md

**整体定位**
一句话：一个以 Agent Skills 格式发布的、引导 AI 通过"需求锚定→全面调研→计划制定→执行审核"四阶段完成任何项目的轻量方法论框架。

不是：独立 AI Agent 平台 / 基础设施服务（不像 Chorus）/ 只面向软件开发（不像 BMAD）
是：行为规则 + 模板 + 约束 / GitHub 模板仓库分发 / 跨领域 / 随使用迭代演进

查重结论：Chorus 最接近但缺少独立调研阶段且是重基础设施。我们的框架填补"方法论层"空白。

**四个阶段概要**

| 阶段 | 输入 | 流程概要 | 产出 |
|------|------|---------|------|
| 1. 需求锚定 | 用户模糊意图 | 动机挖掘(五问法)→受众锚定(用户故事)→边界明确(EARS)→成功标准定义→确认 | 需求规格书.md |
| 2. 全面调研 | 需求规格书 | 路线A领域调研(≥3源)→路线B工具调研→综合分析→质量检查 | 调研报告.md |
| 3. 计划制定 | 需求+调研 | WBS分解→MoSCoW优先级→里程碑→风险→确认 | 执行计划.md |
| 4. 执行审核 | 执行计划 | PDCA循环→增量验证→用户参与→双路径审核→回顾 | 实际产出+审核报告 |

**跨阶段约束**
防幻觉（WebSearch验证）、文件化（写文件再开口）、阶段门控（文件存在性判断）、版本记录（只追加）、用户确认（切换阶段须逐条确认）、来源标注（URL附在文件末）、上下文管理（SKILL.md<200行）

**自我进化机制**
每次项目完成后追加迭代反馈（什么帮了忙/什么没用/什么缺失/本次修正）。防臃肿机制（规则三级+30条硬上限+一进一出+SemVer版本格式）。

**差异化定位**
与 BMAD（纯软件）、AgentFactory（需Python环境）、Chorus（重基础设施）、SDD（假设需求已知）、AI for Auto-Research（纯学术）的对比中，我们的框架独特地满足：需求对话前置 + 调研内置 + 领域无关 + git clone 即用。

**框架自诊断**
每个阶段切换时 AI 必须：确认产出文件存在且非空 → 确认产出格式符合模板 → 确认用户确认记录存在 → 如任一不满足暂停并报告异常。

## 3. 最终审视：查漏补缺

来源：最终审视_查漏补缺.md

**审视日期**：2026-06-13

**已覆盖**：8个模块全部完成（需求锚定/调研方法/计划执行审核/AI协作规范/查重/分发/增量验证/防臃肿）。

**新发现（审视时新出现的关键发展）**

1. **Agentic Engineering 正式成为独立学科**（arXiv 2606.05608, 2026.6.4）：四阶段演进路线图 Stage I(2023-2025)工具增强 → Stage II(2025-2027)单任务自主 → Stage III(2026-2029)多Agent团队 → Stage IV(2028+)自进化生态。我们的框架处于 Stage II→III 过渡期。

2. **EvoClaw 基准测试的警告**：Agent 在孤立任务上 >80% 准确率，但在持续任务中骤降至至多 38%。四个根本挑战：上下文漂移、级联错误传播、无技术债务意识、验证不完整。这些直接验证了模块7增量验证和文件化工作流的设计。

3. **Anthropic 2026 Agentic Coding Trends Report**：使用 Agentic 工具的工程师每任务时间减少，但总体工作时间增加——花更多时间在审查/验证/理解 AI 产出上。验证了框架"用户参与"要求的行业必要性。

**审视中发现的遗漏**

1. **缺少框架自身的测试策略**：设计了框架帮助用户做项目，但没有设计框架自身的测试方法。需要补充：自我诊断规则、用户反馈收集机制、框架行为异常识别信号。

2. **缺少错误恢复路径**：如果 AI 在执行过程中跑偏了（跳过调研直接写内容），框架如何检测和纠正？需要补充 Hooks 机制做确定性检查，第一版可先用"用户手动检查"。

3. **缺少多项目并行时的状态管理**：已确认当前设计天然支持（状态基于当前工作目录下的文件），但需在 CLAUDE.md 中明确说明"本项目是什么、目前处于哪个阶段"。

4. **缺少框架的"Hello World"或最小可行案例**：新用户需要一个 5 分钟快速体验案例。建议 README.md 中加微型项目模拟，或用 QMC 讲义的真实案例记录。

**框架设计规范的最终修订**
- 新增框架自诊断规则
- 新增 Agentic Engineering 定位声明
- 修订阶段4执行指令，补充 EvoClaw 四个挑战的对应策略

**结论**：框架设计已经足够清晰，可以进入实施阶段。

## 4. 通用Agent调研

### 4.1 通用Agent调研与设想.md

来源：通用Agent调研与设想.md

**核心想法复述**
一个遵循四阶段方法（需求→调研→规划→落地审核）的通用项目执行框架。核心哲学："渔具不是鱼"——以文档+规则+工作流形式增强现有 AI Agent 能力，不是独立新平台。

**初步调研发现的现有框架**

| 框架 | 相似度 | 核心价值 |
|------|--------|---------|
| BMAD Method | 70% | 四阶段结构几乎一样，但纯软件工程导向 |
| AgentFactory (arXiv 2603.18000) | 50% | Meta-Agent + 子Agent池机制 |
| SDD (Spec-Driven Development) | 60% | "需求先行"理念一致 |
| AI Workflow (arXiv 2605.10907) | 80% | 明确把需求收集作为工作流第一阶段 |
| Chorus | 高 | Agent Harness 基础设施 |

**市场空白确认**：没有一个现有框架同时满足：从需求对话出发 + 覆盖文献调研阶段 + 适用于非代码项目 + 轻量文档形式分发 + 可随使用迭代演进。

**现有积累盘点**
- 学习方法论.md (v3.7)：教学场景下的Agent行为规范，可泛化
- 论文下载与中文翻译SOP.md (v1.0)：翻译场景下的完整SOP模板
- 搜索回退链：原生WebSearch → Serper → Tavily
- 教学六步法：全景图→最小示例→源码追踪→完整数据流→变体实验→自测

### 4.2 通用Agent全面调研报告.md

来源：通用Agent全面调研报告.md

**10个框架深度剖析**

1. **BMAD Method**：48,400+ Stars，三种规模轨道（Quick/Standard/Enterprise），9个Agent角色，五大提示词维度，制品驱动交接。局限：纯软件工程，无调研阶段。

2. **AgentFactory**（arXiv 2603.18000）：北京大学团队，子Agent池机制，Meta-Agent调度。局限：需要Python环境，面向代码生成。

3. **AI for Auto-Research**（arXiv 2605.18661）：首个覆盖完整学术研究生命周期的AI辅助综述，四阶段八步骤，五大核心发现。关键发现："artifact generation consistently outpaces verification"——产出生成速度远超验证能力。

4. **SDD**（Spec-Driven Development）：Requirements-First vs Design-First，代表工具Kiro(AWS)、Zencoder、GitHub Spec Kit、Superpowers。

5. **Plan-and-Execute Agent Pattern**：LangGraph/CrewAI/AutoGen/Semantic Kernel 等主流框架的通用设计模式。7大Agent设计模式（2026共识）。

6. **Claude Code 生态体系**：六大功能层级（CLAUDE.md → Skills → Hooks → Subagents → Agent Teams → Plugins），社区已有17个角色Agent的实践。

7. **AgentsWay**（ResearchGate, 2025.10）：首个为AI Agent团队设计的软件开发方法论，"Human as Product Owner"。

8. **Driving AI Agents with Specifications**（Pat McGuinness, 2025.8）：先写Spec再让Agent执行，Spec同时作为交付标准。

9. **中文社区方案**：Eino ADK（字节跳动CloudWeGo），国内以LangChain和自研为主，缺乏面向非代码场景的方法论。

10. **内容创作方向**：无完整方法论框架；Jasper AI、Copy.ai等是单次生成工具。

**跨框架对比结论**
流程覆盖上，所有框架都缺少"需求锚定+调研"的组合；非代码场景覆盖上，仅我们和 Claude Code 生态支持学术写作/翻译；存在形式上，我们是唯一以轻量文档/规则集存在的。

**定位陈述**：
> 我们不是在做一个新的Agent平台。我们是在做一个"如何用好现有Agent平台"的方法论框架。

五个独特卖点：需求诱导前置、调研内置、领域无关、轻量分发、自我迭代。

### 4.3 通用框架物理形态调研.md

来源：通用框架物理形态调研.md

**Agent Skills 开放标准**
- agentskills.io，最初由 Anthropic 提出后开源
- 被 30+ 工具原生支持：Claude Code、Cursor、GitHub Copilot、VS Code、OpenAI Codex、Gemini CLI 等
- 本质：一个文件夹内含 SKILL.md 入口文件 + 可选辅助文件
- 加载机制：启动时只加载 name+description（极小开销）→ 任务匹配时才读取完整 SKILL.md
- 安装位置：`~/.claude/skills/`（全局）或 `.claude/skills/`（项目级）

**三种物理形态对比**

| 方案 | 结构 | 优点 | 缺点 |
|------|------|------|------|
| A: 纯Skill | 单SKILL.md+模板 | 简单 | SKILL.md过长，阶段混在一起 |
| B: 多Skill协作 | 主Skill+子Skill | 阶段隔离，可独立加载 | 结构复杂，需状态传递 |
| C: 模板仓库+Skill | CLAUDE.md+项目目录 | 开箱即用 | 职责划分需精心设计 |

**推荐**：B + C 混合（GitHub 模板仓库，核心引擎是多协作Skills）。当前策略是"边做 QMC 讲义边提炼规则"，不是先在真空中写完美目录结构。

**与 Claude Code 能力层级的对应**
- 基础约束 → CLAUDE.md（宪法级规则）
- 阶段流程 → Skill/SKILL.md（触发条件和执行指令）
- 质量检查 → Hooks（确定性检查）
- 子任务执行 → Subagents（并行调研/审核）
- 可复用能力 → 子Skills（搜索链、格式检查等）

## 5. 对导师调研工具的综合参考价值

将所有模块的启发整合，导师调研工具的设计原则：

1. **需求锚定先行**：学生说"想找导师"时，先通过结构化对话挖掘真实需求（研究方向兴趣、地域偏好、职业目标、经济条件），产出"导师匹配规格书"
2. **两路线调研**：路线A（领域调研：搜索各校该方向的导师/论文/课题组）+ 路线B（工具调研：哪些平台/数据库可以搜索导师信息，如 Google Scholar、ResearchGate、知网、导师评价网）
3. **基于 WBS 的任务分解**：确定方向→搜集候选→深度调研→筛选→套磁→面试→决策，每个任务有验收标准
4. **双路径验证**：AI 收集的信息（论文数、引用、H-index）需要人工验证（联系在读学生、读论文确认方向）
5. **增量验证**：逐层筛选，每层有自己的检查标准（粗筛看方向→中筛看论文+资源→精筛看性格+氛围）
6. **防臃肿机制**：功能三级（核心：搜索+筛选+联系模板；推荐：论文分析+排名；实验：AI推荐），30条功能上限
7. **轻量分发**：GitHub 模板仓库 或独立 Skill，clone 即用，不依赖外部服务器
8. **认知偏差防范**：学生可能"看起来匹配"就高兴发邮件，必须设置验证节点确认关键信息
9. **文件化工作流**：每一步的搜索记录、论文笔记、邮件草稿都保存为文件，支持跨会话追踪
10. **最小理解三问**：每个候选导师完成后，学生应能回答"他做什么方向？/为什么适合我？/怎么验证他还活跃？"
