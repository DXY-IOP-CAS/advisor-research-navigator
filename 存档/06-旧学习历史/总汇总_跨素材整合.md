## Block A：项目背景与需求确认

### A1 工具目标与起因

**来源**：需求确认.md Block 1

每年硕士/博士生选导师是刚需。研一 3-4 月面临报志愿套磁，需快速判断与导师研究方向匹配度。四个痛点：

| 痛点 | 描述 |
|:----|:-----|
| 官网信息粗糙 | 导师官网只有论文列表和标题，无科普级解读 |
| 有论文看不懂 | 论文基于研究生以上知识，学生仅本科基础 |
| 资源散乱无索引 | 论文/教材/视频/B站/博客各处散落，无统一路径索引 |
| 调研极易绕圈 | 不知道"该读什么、按什么顺序"，信息海漫游 |

工具按**四段层层嵌套**运作。前一段输出为后一段提供认知上下文，范围逐段收窄，理解逐段加深：

| 段 | 做什么 | 产出 |
|:---|:------|:-----|
| 第1段 | 导师基础信息调研（地毯式收集多维信息，多源交叉比对） | 导师基础画像，明确所在领域范围 |
| 第2段 | 相关领域发展调研（追溯子领域发展脉络，理清技术路径） | 认知地图，理解该领域在大专业中的位置 |
| 第3段 | 导师论文串联与定位（以领域认知框架审视论文序列） | 论文时间线 + 前沿定位 |
| 第4段 | 从零到前沿学习讲义（从高数+线代起步逆向设计学习路线） | 单线递进式讲义 |

嵌套逻辑：**大领域 -> 小领域 -> 导师定位 -> 学习路径**。工具同时承担三职能：调研（收集）、综述（串联）、教学（设计路线），三职能递进。

### A2 用户画像：大字典与小字典

**来源**：需求确认.md §1.2

**用户实际场景**：大二/大三物理相关专业本科生，准备申请硕博选导师。

**工具假设起点水平**：高等数学和线性代数。所有高阶知识（复变、数理方法、高量、群论、量子多体等）本质上从高数+线代逐步推广。从根推导保证零跳步。

**大字典（教材体系定位）**：

| 层面 | 课程 | 定位 |
|:----|:-----|:----|
| 数学基础 | 高数、线代、概率论、复变、数理方法 | 推导起点 |
| 物理基础 | 普物、四大力学、固体物理 | 物理直觉来源 |
| 高阶延伸 | 高量、群论、量子多体、量子场论、规范场论 | 领域边界——定义触及范围 |

教材从"课本"变成"字典"：AI 时代学生不必逐页通读，工具提供参考坐标，用户按需读取字典条目。

**小字典（工具产出）**：第4段产出即个性化小字典——从大字典中提取第1-3段确认真正需要的知识点，串联成从高数+线代到导师前沿的逻辑链。

### A3 差异化定位

**来源**：需求确认.md §1.3 + D08 §十三

**工具不是**：
- 不是信息卡片（Google Scholar 自己做）
- 不是单论文翻译或知识点孤岛（paper-to-course 已做）
- 不是教科书式全面覆盖（Annual Review 缺乏导师匹配度）

**工具是**：
- 把导师 N 年工作反推回一套大学基础到前沿的递进式推导网络
- 知识的串联方法（基础->领域->前沿是一条线，节点互为因果）
- 调研+综述+教学三职能兼备
- 路线规划和资料索引（给路径不给答案）
- 通用模板引擎：对任何导师适用

**工具不做**：
- 不写具体教材内容（讲义是路径+资源指向）
- 不替代学生主动学习
- 不点评导师/推荐方向
- 不捏造论文/教材链接（所有引用必有 URL）
- 不把"从大二到大四"的学习做完

**不闭门造车原则**：每阶段经过调研再决策，站在巨人肩膀上。用户反复强调："流程是手段，质量是目的"。

## Block B：工具设计参考（D01-D09 关键结论）

### B1 需求锚定方法

**来源**：D04 全部 + D02/D08 相关

**核心方法**：需求工程四阶段（Sommerville 2011, IEEE 29148）——诱导 -> 分析 -> 规格 -> 验证。四阶段不能跳步。

**四轮对话策略**（D04 K5）：

| 轮次 | 内容 | 方法论来源 |
|:-----|:-----|:----------|
| 第一轮：动机挖掘 | 追问"为什么做"到不能追问为止 | 半结构化访谈 + 五问法 |
| 第二轮：受众与标准锚定 | "作为<角色>，我想<能力>，以便<价值>" | 用户故事 Connextra 模板 |
| 第三轮：约束与边界明确 | 硬约束、禁止清单、妥协空间 | EARS 标注法 |
| 第四轮：成功标准定义 | 3-5条可检查指标 | Context Engineering Rubric 角色 |
| 确认轮 | AI 汇总为需求规格书，用户逐条确认 | RE 验证阶段 |

**EARS 标注法**（Mavin et al., 2009 IEEE RE Conf）：五种句型（Ubiquitous/Event-driven/Unwanted behavior/State-driven/Optional），每条需求必须符合其一，防止模糊。起源于 Rolls-Royce 航发控制系统适航分析。

**五问法**（Toyota Production System）：连续问五次"为什么"直到触及根本原因。关键：每次基于事实证据，根因标志是归因到系统/设计而非人。

**需求规格书模板骨架**（D04 K5）：
1. 项目动机
2. 目标受众（用户故事格式）
3. 成功标准（可检查清单）
4. 硬性约束（EARS 格式）
5. 妥协空间
6. 参考范例（好/坏例子）
7. 版本记录

**设计张力**：四轮+确认共五轮，小微任务可能过重。弹性判据未解决——这是 D10 开放问题 P06。

### B2 全面调研方法

**来源**：D05 全部

**调研本质 = 不确定性消除**（D05 K1, v2 从第一性原理推导）：

| 不确定性类型 | 定义 | 标志问题 |
|:------------|:-----|:---------|
| 概念 (Concept) | 不知道基础概念/术语/知识结构 | "这事到底是什么？" |
| 方案 (Scheme) | 不知道已有方法/方案/工具及其优劣 | "别人用什么方法？" |
| 质量 (Quality) | 不知道信息/工具是否可靠 | "这东西靠谱吗？" |
| 可行性 (Feasibility) | 不知道实际条件下是否可行 | "我能不能做到？" |
| 参考 (Reference) | 不知道可复用的资源/素材/模板 | "有什么现成可用？" |

五分类是**分析骨架**（设计时指导覆盖、执行后检查遗漏），不是操作流程。

**项目类型 x 不确定性矩阵**（D05 K2）：六种项目类型（讲义教程/辅助学习/视频工作流/代码工具/调研综述/技能学习）各有 H/M/L 分布。矩阵是适配器不是分类器——同一项目在不同人手中分布不同。

**外部方法论审视**（D05 K3，三个成熟框架对 K1/K2 的检验）：
- **Taylor 信息需求四层次**（1968）：Visceral（模糊）-> Conscious（能绕圈说）-> Formalized（清楚描述）-> Compromised（向系统能力妥协）。五分类回答"关于什么"，Taylor 回答"多清楚"——正交维度，缺一不可。工具选择限制搜索结果（Compromised 问题）。
- **IDEO 设计研究三段法**（HCD 2015）：Inspiration -> Ideation -> Implementation。概念/方案调研属 Inspiration 模式，可行性/质量调研属 Ideation 模式——两种模式放在一起会互相干扰。
- **Double Diamond 发散收敛模型**（British Design Council 2004）：阶段2至少一轮发散->收敛循环，不是均匀推进。

**四层退出条件**（D05 K4）：

| 机制 | 来源 | 适用场景 |
|:----|:-----|:---------|
| 饱和（Saturation） | Glaser & Strauss 1967 | 连续3-5个来源重复已知内容 |
| 时间盒（Timebox） | XP Spike; Scrum Sprint | 所有维度——备选兜底 |
| 验收标准（DoD） | Scrum Definition of Done | 预定义可验证条件全部满足 |
| 雪花球收敛 | Wohlin 2014 | 方案/工具关系追踪 |

多层退出决策树：DoD -> 饱和 -> 时间盒 -> 雪花球收敛。

### B3 执行与审核

**来源**：D06 全部

**阶段3骨架** = WBS（分解）+ MoSCoW（优先级）+ 依赖关系编排（执行顺序）。三条维度互补非互斥。

**WBS（工作分解结构）**：面向可交付物的层级分解。核心规则——100% Rule（覆盖全部范围）、8/80 Rule（每包 8-80 小时）。致命假设：分解结构在执行前已知。对此框架需**渐进式 WBS**（迭代内 100%，跨迭代增量补充）。

**MoSCoW 优先级法**（Oracle, 1994）：Must/Should/Could/Won't。DSDM 补充 60% 规则（Must Have 不超过资源 60%）。单人+AI 弹性资源场景下 60% 规则约束力自动消失——需为弹性资源设计替代取舍机制。

**PDCA 循环**（Deming 环）：Plan -> Do -> Check -> Act。关键在"循环"——Check 输出是下一轮 Plan 输入。迭代边界推荐为每章。

**ATDD（验收测试驱动开发）**：先写验收标准再做工作。验收标准分形式标准（机器可检）和本质标准（人类判断）。AI 只检得了形式标准。

**三级审核**（D06 K4, 源自 Atul Gawande《清单宣言》2009）：
- **DO-CONFIRM**：自己做完自己对照复查（本质标准，人类判断）
- **READ-DO**：AI 对照清单逐项检查报告（形式标准，AI 执行）
- **HOOK**：提交时自动触发（自动化：格式规范、引用完整性）

三层嵌套使用，不可取代不可混用。每层输出是下一层输入。

**双路径验证**（D06 K4, 源自 Chorus）：AI 自检（第一轮）+ 人类审查（第二轮）+ 合并结论。AI 报告含形式标准项（通过/失败/不确定）和本质标准项（标注"需人类确认"）。人类审 AI 自检报告，非从头查。

**阶段4审核 = 检查门 + 学习门**：用户通过审核过程理解项目原理，积累领域经验，形成"参与审核 -> 理解原理 -> 能定更好标准"的能力闭环。

### B4 AI 协作规范

**来源**：D07 全部

**文件化 > 对话化**（D07 K1）：2025-2026 主流 AI 编码平台坚固共识——重要上下文和规则必须存储在文件中。对话随上下文增长衰减或被截断，文件不会。分层共识：宪法级（始终加载）-> 流程级（任务触发）-> 事件级（Hooks）-> 路径级（按代码位置）。

**Context Engineering 三层范式跃迁**（D07 K2）：
1. **Prompt Engineering**（2022-2024）：怎么问 AI 得到最好回答
2. **Context Engineering**（2025）：该让 AI 看到什么信息才能做出正确决策
3. **Harness Engineering**（2026）：怎么设计 AI 整个运行环境，稳定可靠自主工作

**五角色上下文包**（Calboreanu, arXiv 2604.04258, 2026）：
- **Authority**（权威）：需求文档、设计约束——P1 覆盖一切
- **Exemplar**（范例）：模板、历史报告——P2
- **Constraint**（约束）：字数限制、API 版本——P3
- **Rubric**（评估标准）：DoD、质量门条件——P4
- **Metadata**（元数据）：项目名、用户 prompt——P5

关键发现：**Metadata Paradox**——用户 prompt 永远是最低优先级元素（Metadata）。当用户临时指令与 Authority 文档冲突时，Authority 胜出。框架需要显式定义规则优先级：CLAUDE.md > SKILL.md > 用户临时指令。

**Harness Engineering**（Mitchell Hashimoto, 2026.2）：Agent = Model + Harness。Model 处理推理（商品化中），Harness 是模型之外一切（拉开差距的地方）。

**Ratchet 原则**：每次发现 AI Agent 犯错，在运行环境中设计解决方案让其不能再犯同样错误。操作流程：犯错 -> 追溯根因到 Harness 某层 -> 编码预防规则 -> 错误不能再发生。

**前馈控制 vs 反馈控制**（Fiddler AI, 2026.6）：
- 前馈（Feedforward）：Agent 行动前引导——系统提示词、Skill 文件、工具允许/阻止列表
- 反馈（Feedback）：Agent 行动后评估——Linter、测试、LLM-as-Judge、CI 门控

项目护栏层全前馈（提示词约束）、反馈全依赖 HITL——当前最大结构弱点。

**7 大 AI Agent 设计模式**（D07 K4）：Reflection / ReAct / Plan-and-Execute / Tool Use / Multi-Agent / Memory / Human-in-the-Loop。非互斥——每阶段可用不同模式组合。项目缺乏 Reflection 模式（自评->修正循环）。

**外循环铁律**（D08 §十四, 宪法级）：外循环必须是"人决策 + AI 执行"。禁止 AI 自嗨式自我迭代。三层嵌套循环：内层 Agentic Loop（think-act-observe）-> 中层 Supervisory（读 diff 改方向审批）-> 外层 Loop Engineering（改规则文件）。Ratchet 位置：内层出错 -> 中层发现 -> 外层编码修复。

### B5 框架设计规范

**来源**：D02-D03 对比 + D08 查重 + 指导/调研/汇总

**四阶段框架定位**：以 Agent Skills 格式发布的、引导 AI 通过"需求锚定 -> 全面调研 -> 计划制定 -> 执行审核"四阶段完成任何项目的轻量方法论框架。

**五个独特卖点**（10 框架对比后确认）：
1. 需求诱导前置（对话式挖掘动机）
2. 调研内置为独立阶段（最大差异化）
3. 领域无关（非纯软件/纯学术）
4. 轻量分发（文档为主，git clone 即用）
5. 自我迭代（外循环 Ratchet）

**10 框架对比结论**（D02 K2 三张矩阵 + D02 K3 14 条可借元素）：
- 唯一"需求锚定 + 全面调研"都打勾
- 唯一覆盖非代码场景
- 唯一以轻量文档/规则集存在
- Chorus 最接近但缺独立调研阶段且重基础设施
- Manus 四阶段相似但缺调研且闭源

**三层架构**（D01 收尾 + D08 更新）：
- **框架层（project-pilot）**：决定"完成什么阶段、产出什么"。外循环。
- **Harness 层（Claude Code + 阶段2选型工具）**：提供完整 Agent 运行环境。内循环支撑。
- **项目层（具体产出物）**：内循环本身。

三层不是单向流水线——框架从项目中学，项目内部自我改进，Harness 提供能力但不决定方法。

**框架设计意图（10 条核心决策）**：

| # | 决策 | 来源 |
|:-:|:----|:----|
| 1 | 四阶段流程 | Manus、BMAD 独立验证 |
| 2 | 调研独立为阶段2 | 最大差异化定位 |
| 3 | 单 Agent + 按阶段换 Skills | BMAD、AgentFactory |
| 4 | Reversed Conversation 可选入口 | Chorus |
| 5 | 外循环 = 人决策+AI执行 | AgentFactory + Loop Engineering（宪法级）|
| 6 | 阶段2 = Harness 组件选型调研 | Harness Engineering |
| 7 | 三层审核：Hook -> AI自检 -> 人类审查 | Chorus 双路径验证 |
| 8 | 阶段间用"已审批摘要" | Conductor、AutoR |
| 9 | 框架不取代 Harness | D08 定位讨论 |
| 10 | 阶段间可回退但粒度不同 | Loop Engineering 状态恢复 |

**规则分级与防臃肿**（D08 §十五 + 模块8）：
- 三级：`[CORE]`（删掉无法运转） / `[REC]`（提高质量可跳过） / `[EXP]`（新增待验证）
- 升降级：EXP 连续 3 次迭代有效 -> REC -> 连续 2 次 MAJOR 升级有效 -> CORE
- 30 条硬上限 + 一进一出
- RFC 机制：新规则以 RFC 提案发起，评审通过后加入；旧规则被新规则取代时标注 `Obsoleted by`，保持可追溯
- SemVer + Changelog（Added/Changed/Deprecated/Removed/Fixed）

### B6 分发形式

**来源**：D09 §1.1-1.3 + D03 K2

**五种形式对比**：

| 形式 | 用户体验 | 开发成本 | 跨平台 | 可迭代 | 总分 |
|:----|:-------:|:-------:|:-----:|:-----:|:---:|
| 模板仓库 | 5/5 | 5/5 | 5/5 | 4/5 | 19/20 |
| Skill | 4/5 | 5/5 | 5/5 | 5/5 | 19/20 |
| Plugin | 5/5 | 2/5 | 1/5 | 3/5 | 11/20 |
| npm/npx | 3/5 | 3/5 | 2/5 | 3/5 | 11/20 |
| Cookiecutter | 2/5 | 4/5 | 5/5 | 3/5 | 14/20 |

**最终方案**：GitHub 模板仓库 + 内置 Skill（组合方案，非替代关系）。

**不选纯 Plugin 的三条理由**：
1. 平台锁定（仅 Claude Code，框架规则写在 .md 里，任何 IDE 都该能用）
2. 成本不成比例（Plugin 要求 MCP Server 后端进程，框架核心=规则+模板）
3. 更新太激进（marketplace 自动推送覆盖，框架需要用户先审再采纳）

**不选纯 Skill（全局安装）的三条理由**：
1. description 太宽泛导致频繁误激活
2. 无法携带脚手架（CLAUDE.md + 空目录结构）
3. 用户自定义和框架更新冲突（git merge 模式允许选择性采纳）

**组合方案三层架构**：

| 层次 | 载体 | 职责 |
|:----|:-----|:-----|
| 容器层 | GitHub 模板仓库 | CLAUDE.md、目录结构、模板文件、Hooks |
| 内容层 | `.claude/skills/项目执行/SKILL.md` | 阶段判断、流程引导、质量门规则 |
| 脚手架 | 空目录（产出/需求/调研/计划/） | 项目产物组织引导 |

### B7 软约束失效与解决方案

**来源**：D09 §2.4 + 三方对齐报告

### C1 量子蒙特卡洛讲义叙事结构

**来源**：指导/写作/读者画像与叙事基调.md + 15章大纲.md

**双受众设计**：
- 零基础小白（物理系本科核心课程，严禁默认研究生高阶知识）
- 业内专业人士（概念准确、逻辑严密、PRL/PRX 级别）

写作底线：只凭这份文档就能从零精通。

**起承转合五步叙事流**（废除了死板的章节标题模板）：
1. **[起] 物理危机的降临**：紧接上一章主角模型，展示原有公式如何失效
2. **[承] 铸造新武器**：物理隐喻先于数学公式，每 3-5 行代数插入一句物理目的解释
3. **[转] 主角模型实战检验**：把新公式应用到主角模型上手动计算，展示可验证数值结果
4. **[合] 新阴云的汇聚**：利用计算结果指出更极端条件下破绽，引出下章悬念
5. **[预判卡点]**：在核心卡点文档对应位置，用"你可能会想"自然消解

**全书统一主角演进**：经典阶段 N=2 Ising 模型 -> 量子阶段 N=2 Hubbard 模型（t=1.0, U=4.0）。

**四幕剧结构**：

| 幕 | 核心任务 | 章节 |
|:--|:---------|:----:|
| 第一幕：被冻结的电子（经典 Ising） | Metropolis 算法经典自旋采样，解决高维求和 | 第 1-4 章 |
| 第二幕：觉醒的电子（量子 Hubbard） | Trotter 分解解决不对易性，量子->经典映射 | 第 5-7 章 |
| 第三幕：经典回归（HS 变换与 DQMC） | HS 变换引入辅助场，完成闭环 | 第 8-11 章 |
| 第四幕：李自翔三重破局 | 符号灾难 -> MTR/ASF/非平衡动力学 | 第 12-15 章 |

**完整推导链**：Trotter（第6章）-> HS（第7章）-> Grassmann（第8章）-> DQMC（第9章），跳过一步读者断裂。

**三条破局路线**：

| 路线 | 时间 | 核心策略 | 关键论文 |
|:----|:-----|:---------|:---------|
| MTR 对称性 | 2015-2019 | 对称性保护**根本消除**符号问题 | PRL 117, 267002 (2016) |
| ASF 渐近无符号 | 2020-2023 | 狄拉克费米子中**被动等待**符号消失 | arXiv:2211.00663 (2022) |
| 虚时非平衡 | 2021-至今 | 一般体系中**主动规避**符号问题爆发 | Science Advances 12, eadz4856 (2026) |

### C2 审稿四类问题

**来源**：指导/审稿/审稿提示词.md

**角色设定**：PRL/PRX 级别学术同行评审专家，精通强关联计算物理 + 顶尖物理教学法（费曼/萨斯坎德式风格）。

**四类问题体系**：

| # | 病名 | 现象 | 修改指令 |
|:-|:----|:-----|:--------|
| 1 | 物理直觉与动机缺失 | 抛出复杂公式无"为什么做" | 先铺垫物理直觉 |
| 2 | 数学跳步或堆砌 | "显然可见""整理可得"掩盖推导；或机械罗列公式无中间解释 | 补充缺失逻辑；冗长公式中插入"喘息句" |
| 3 | 主线断裂与沙盒脱节 | 开头不承上章危机；沙盒模型被孤立成习题 | 指出断层位置，建议过渡或计算实例 |
| 4 | 认知超载与符号密度 | 连续三段涌现 3+ 未经解释的新符号 | 重写：用隐喻或数值消化一个再引入下一个 |
| 5 | AI 幻觉 | 公式细节错误（Trotter 误差阶、Grassmann 符号、数值验证） | 严厉指出虚假推导 |

**核心理念**：抛弃打分制和清单核对，作为真实读者感受文字流动。

**输出格式**：同行评议意见。总体感受 -> 具体修改批注（定位 + 诊断 + 重写建议）-> 总结（指出必须修改的硬伤，无硬伤放行）。

### C3 学习方法论

**来源**：指导/学习方法论.md + 指导/学习方法论-附件.md + 指导/续学指令.md

**一句话本质**：一次只学一个东西。讲一小段 -> 用户实操 -> 用户复述 -> 确认理解 -> 下一个。

**四条文献支撑**：认知负荷理论（Sweller 1988）、工作样例效应（Sweller & Cooper 1985）、自我解释效应（Chi et al. 1989）、检索练习效应（Roediger & Karpicke 2006）。

**双轨并行**：操作型（全景 -> 初体验 -> 定大纲 -> 逐点 -> 综合应用）vs 调研型（全景 -> 快速定位 -> 定大纲 -> 逐点 -> 收尾盘点）。

**阶段-模式绑定**：

| 阶段 | 绑定模式 |
|:----|:---------|
| 全景图 + 初体验 | Plan-and-Execute |
| 定大纲 | Plan-and-Execute + HITL |
| 逐点学习（调研型） | ReAct |
| 逐点学习（操作型） | Plan-and-Execute |
| 收尾盘点 | Reflection + HITL |

**AI 执行 8 条规则**：
1. 用户控速（"继续"是唯一信号）
2. 零跳步（禁止"显然""易得"）
3. 文件出入（首次教学 -> .md 文件）
4. 按需验证（正确理解直接推进，偏差给 1 个追问）
5. 防幻觉（先搜再写，URL 溯源）
6. 原始来源回溯（先读原始来源再写课件）
7. 反谄媚分析（调研型课件必须含"支持"和"挑战"两列）
8. 追问回路（全力解答 -> 主动确认是否回到主线）

**输出门协议（P1 宪法级）**：所有教学/讲解/对比/分析内容必须写入 .md 文件后再回复。回复第一行必须是 `已写入：[文件路径]`。
