## §1 为什么做这个种子

### 1.1 两个不满，一个正解

你最初对两种常见做法都不满意：

- **生搬硬套**——市场不缺看起来强大的工具，但"强大 ≠ 适配"。别人的工具不知道你实际要做什么。<!-- 来源：初衷/01_为什么要建种子.md "反生搬硬套"节 -->
- **闭门造车**——不调研就动手，等于把已经存在的轮子再发明一遍，且更差。技术日新月异，今天重写的东西下个月就有人做了更好的。<!-- 来源：同上 "反闭门造车"节 -->

正解不是"中间路线"，是**种子 + 大量调研 + 反复迭代**。种子不预设具体形态，它是一套"先搞清楚要什么、再大量调研、站在巨人肩膀、扬长避短"的流程。每一轮反馈都让长出的工具更贴近需求。<!-- 来源：同上 "正解"节 -->

### 1.2 两个不可跳步的动作

你反复强调两个动作，任何流程设计都要保留它们作为不可跳步的环节：

1. **确认需求**——动手前必须搞清楚"为什么做、做什么、做成怎样算成功"
2. **调研**——动手前必须看市面上有什么、到了什么程度

这两个共识在 D05 被独立验证（调研作为独立阶段经过双重验证，D08 决策 2），在 D08 被查重确认——在 79+ 基础设施工具和 6 个领域特化框架中，我们是唯一把"调研"作为独立通用阶段的框架。<!-- 来源：D08 收尾盘点 §四 决策2 + D01-D08_回顾速查 D08 核心结论 -->

### 1.3 架构演进脉络

从 D01 到 D09，框架认识经历了四个阶段：

| 阶段 | 核心问题 | 代表性结论 |
|:----|:--------|:----------|
| D01-D03 | 框架长什么样？ | 三层架构（协议→模板→经验）、单 Agent + 按阶段换 Skills |
| D04-D06 | 每个阶段里有什么？ | 阶段1=五轮对话、阶段2=不确定性消除、阶段3=WBS+MoSCoW+编排、阶段4=三级审核 |
| D07 | 规则质量如何？ | 14 条规则中 9 条软约束、5 条可机械强制但全未实现 |
| D08 | 有人做过吗？ | 查重结论：我们的四阶段流程没有被实现过 |

<!-- 来源：D01-D08_回顾速查 "主线：框架设计从模糊到清晰的演进"节 -->

### 1.4 D08-D09 确认的定位

D08 查重确认：在 79+ 基础设施工具、6 个领域特化框架中，四阶段流程（需求锚定→全面调研→计划制定→执行审核）**没有被实现过**。我们的独特不在"调研重要"，在"调研可跨领域"。<!-- 来源：D08 K3 结论 + D08 决策7 -->

## §2 种子是什么 / 不是什么

### 2.1 种子不是文档

文档的价值在"读完之后人懂了"。种子的价值在"用了几次之后工具变好了"。文档是死的，种子是活的——每次使用中的反馈都会让长出的工具更贴近用户。<!-- 来源：初衷/02_种子的本质.md "种子不是文档"节 -->

### 2.2 种子不是框架

框架给固定结构（四阶段、若干规则），靠"人改框架"来成长。种子给结构 + 生长机制——结构提供骨架，机制让工具在使用中自然逼近用户需求，不需要人定期改种子本身。<!-- 来源：同上 "种子不是框架"节 -->

### 2.3 种子不是 Agent

种子不替代 Agent、不指挥 Agent。种子**驱动** Agent 的行为——告诉 Agent"在这个项目里应该按什么流程做事"。Agent 是手和脚，种子是大脑里的工作习惯。<!-- 来源：同上 "种子不是 Agent"节 -->

### 2.4 种子是——三层结构的活机制

```
种子 ──克隆──> 工具项目（pilot-test 是第一个实例）
                │
                ├─> CLAUDE.md（宪法，约束 Agent 行为）
                ├─> SKILL.md（流程，引导 Agent 做事）
                ├─> Hooks（机械层，强制规则不可绕过）
                ├─> 产出/（工具产生的实际成果）
                └─> 初衷/（种子的基因，克隆时必带）
```

<!-- 来源：初衷/02_种子的本质.md "种子 vs 工具的关系"节 -->

克隆 = 把 `pilot-test/` 整个复制到新文件夹。新文件夹里 Agent 一启动就自动加载这些文件，按种子的流程做事。<!-- 来源：初衷/04_克隆与生长说明.md "一、克隆流程"节 -->

### 2.5 为什么是"种子"不是"模板"

模板给死答案（5 步照做），种子给活机制（"先问清楚再动手"的本能）。模板不能进化（除非人改），种子可以（用户每次反馈让工具变好）。<!-- 来源：初衷/02_种子的本质.md "为什么是'种子'而不是'模板'"节 -->

### 2.6 框架定位的最终表述

D08 查重后定位锁定为"**通用方法论种子——领域特化框架的学习者与抽象者**"（D08 决策 7）。三层模型最终版为"框架层→Harness层→项目层"（D08 决策 11）。框架说"该跑什么阶段"，Harness（Claude Code/Conductor）说"怎么编排怎么执行"——两者不冲突，可以串联。<!-- 来源：D08 收尾盘点 §四 决策7+11 + 框架设计意图.md "设计意向性文件" #9 -->

## §3 已落地的核心决策（现状）

### 3.1 三层架构正式命名

| 层 | 旧称（废弃） | 正式名称 | 承载文件 | 约束类型 |
|:---|:----------|:-------|:-------|:-------|
| 宪法层 | 协议层/骨架 | 框架层 | CLAUDE.md | 提示词约束 |
| 种子层 | 模板层/肌肉 | Harness层 | SKILL.md + 执行指令 | 提示词 + 行为引导 |
| 执行层 | 经验层/记忆 | 项目层 | Hooks（settings.json） | 确定性代码 |

<!-- 来源：综合设计文档 v0.2 "二、整体架构"节 + 框架优化计划书 v0.2 "2. 三层架构"节 -->

**配置层关系**：三层不是等价的。宪法层始终加载，种子层按需触发，执行层条件触发。缺了任一层，框架仍能跑——但约束强度降一级。<!-- 来源：D07 收尾盘点 问题1 "规则分层" + 综合设计 v0.2 "四、跨阶段规则" -->

### 3.2 四阶段骨架（不可变）

四阶段流程是所有后续设计的基础骨架。去掉了任何一个，框架就退化为 Plan-and-Execute 或其他已有模式。<!-- 来源：综合设计 v0.2 "四阶段骨架"节 + D02 收尾 问题2 -->

```
阶段1: 需求锚定 → 阶段2: 全面调研 → 阶段3: 计划制定 → 阶段4: 执行审核
     ↖                                              ↙
          反馈回路（具体协议待 D10 设计）
```

**阶段切换规则**：产出文件存在 + 质量门通过 + 用户逐条确认。禁止跳步。<!-- 来源：综合设计 v0.2 "四阶段推进规则" -->

### 3.3 阶段1 设计现状

**核心原则**：禁止直接跳到解决方案。先说清"为什么做、做什么、做成了怎么算成功"。第一条消息必须含路线图。<!-- 来源：综合设计 v0.2 §三 阶段1 -->

**五轮对话策略**（已落地在 SKILL.md 中）：<!-- 来源：D04 收尾盘点 "核心产出" + 综合设计 v0.2 阶段1 五轮表 -->

| 轮次 | 问什么 | 方法论支撑 |
|:---:|:-----|:--------|
| 0 | 已掌握什么？缺什么？ | D04 决策 6（已有基础） |
| 1 | 为什么做？不做会怎样？ | 五问法（动机层） |
| 2 | 谁看？看了能做什么？ | 用户故事（角色-功能-价值） |
| 3 | 什么限制？什么绝对不能？ | 第一性原理（结构约束层） |
| 4 | 整理 + 逐条确认 | HITL 关键节点 |

**未实现的**：流程弹性判定条件（什么任务走完整流程、什么压缩）仍为开放问题 P06。

### 3.4 阶段2 设计现状

**核心原则**：调研的本质是消除"项目执行前的不确定性"，不是照搬 SLR。<!-- 来源：D05 收尾 问题1 -->

**两路线串行依赖**（已确认不可逆）：路线A（领域调研）先启动 → 产出能力清单 → 路线B（工具调研）基于清单搜索。不能并行——不知道领域就无法知道需要什么工具。<!-- 来源：D05 收尾 问题2 + SKILL.md 阶段2 调研顺序 -->

**五分类覆盖检查**（非操作步骤，是收尾检查）：概念/方案/质量/可行性/参考。调研时自然交叉，收尾时对照五类检查是否有遗漏。<!-- 来源：综合设计 v0.2 §三 阶段2 "五分类法" -->

**退出条件**：四机制决策树（DoD → 饱和 → 时间盒 → 雪花球收敛），但选择规则尚未设计。<!-- 来源：D05 收尾 问题3 -->

**未实现的**：五分类检查清单、退出机制选择规则、迭代搜索操作步骤。

### 3.5 阶段3 设计现状

**核心原则**：骨架 = WBS（分解）+ MoSCoW（优先级标注）+ 依赖关系编排（执行顺序）。三者互补，非互斥。<!-- 来源：D06 收尾 问题1 + 综合设计 v0.2 §三 阶段3 -->

**WBS**：强制步骤，100% Rule。确认"有什么要做、有没有漏"，不是发现工具。<!-- 来源：D06 收尾 问题1 -->

**MoSCoW 注**：在单人+AI 弹性资源环境下 60% 强制取舍规则失去约束力。需替代取舍机制——未设计。<!-- 来源：D06 收尾 问题1 + 综合设计 v0.2 §三 阶段3 -->

**依赖编排**：标注前置任务 ID，区分串行/并行/条件分支。表达格式未定（Chorus Task DAG 还是轻量前置标记？）。<!-- 来源：综合设计 v0.2 §三 阶段3 -->

### 3.6 阶段4 设计现状

**核心原则**：区分形式标准（AI 可检）和本质标准（人类判断）。两者正交，不是替代或增强关系。<!-- 来源：D06 收尾 问题2 + 综合设计 v0.2 §三 阶段4 -->

**三级审核**：<!-- 来源：综合设计 v0.2 §三 阶段4 三级审核表 + D06 收尾 问题3 -->

| 层级 | 检查什么 | 执行者 | 标准类型 |
|:---:|:-------|:-----:|:-------:|
| HOOK | 格式/引用/文件存在 | 自动脚本 | 形式标准 |
| READ-DO | 逐条对照验收标准 | AI 自检 | 形式标准 |
| DO-CONFIRM | 叙述/表达力/逻辑 | 用户 | 本质标准 |

**学习门修正方向**（D06 决策 10）：阶段4 不仅是检查门，也是学习门——用户通过审核自然增进理解。具体设计待 D10/D11 输入。<!-- 来源：D06 收尾 问题3 "用户反馈"节 -->

### 3.7 外循环原则（宪法级）

D08 决策 10 确认：框架外循环必须是"人决策 + AI 执行"，禁止 AI 自嗨式自我迭代。AI 记录摩擦点和提议，改规则/改模板的决策权在人。<!-- 来源：D08 收尾盘点 §四 决策10 + 初衷/03_使用原则.md R3 -->

体现为 CLAUDE.md R3（外循环：人决策 + AI 执行）和 SKILL.md P5。已写入 Hook 阻断原则——任何"为了更好用所以我改了 X"的自动行为视为违规。<!-- 来源：CLAUDE.md R3 + SKILL.md P5 -->

### 3.8 隔离边界（宪法级）

pilot-test 通过 H2 Hook 实现物理隔离（PreToolUse: Write 前检查路径是否含 `..` 上跳）。子目录的 Hook 优先级更高——不污染根项目，根项目规则也不干扰沙盒测试。<!-- 来源：转接文档.md §6 "Hook 加载机制关键事实" + 新窗口使用说明.md 验证1 -->

### 3.9 搜索验证铁律（宪法级）

涉及论文/公式/数据/外部引用必须搜索验证再写。间接引用追溯原始来源。每条外部信息标 URL，末尾"验证来源"小节列出。工具选择按意图分类：学术文献→Exa、技术发现→Exa、深度多源→Tavily、事实核验→Serper。<!-- 来源：全局 CLAUDE.md "联网搜索"节 + pilot-test CLAUDE.md R1+R9 -->

## §4 当前真实痛点

### 4.1 痛点地图

基于反馈日志（两轮）和待定项 F1-F3，当前最痛的三个问题：

#### F1：机械强制层几乎空白（最痛）

**用户感知**："所有约束都是软约束"、"违反无阻断无感知"、"规则都是礼貌的请求不是安全"。

**根因**：14 条规则中 9 条软约束（提示词），5 条可机械强制但全未实现。CLAUDE.md + SKILL.md = 纯文字提示词 = "礼貌的请求不是安全"。唯一能真正强制的 Hook 层是后来才建的，且目前只实现了 4 条 Hook（H1 验证来源检查、H2 隔离边界、H3 阶段门控警告、H4 SKILL.md 行数检查），这些 Hook 的阻断能力仍有疑问（见 §6.3）。<!-- 来源：框架使用反馈日志.md 第一轮+第二轮 + 软约束失效问题_业界方案.md + D07 K5 -->

#### F2：框架几乎从不主动触发

**用户感知**：需要时手动叫"用 pilot"——框架没有在用户最需要的时候自动出现。

**根因**：触发策略最初设计为精确匹配，D09 反馈后改为"主动激活 + 快速退路"。但 SKILL.md 中触发条件仍基于关键词匹配（"我想做 X"、"帮我做 Y"），没有做到"宁可误触不可漏触"。用户原话："你让他尽可能的触发，我是实时根据内容反馈的。"<!-- 来源：框架使用反馈日志.md 第一轮反馈 + SKILL.md frontmatter "宁可误触不可漏触" -->

#### F3："五脏俱全的废物"

**用户感知**：内容看起来完整但质量很差——所有结构都在但实质不到位。

**根因**：D07 发现，规则的可检查性不足导致 AI 生成的产出"看起来对但实际不行"。验收标准停留在形式层面（格式/结构符合），没有机械手段验证实质质量。本质标准只能由人判断，但当前框架没有给用户有效的检查清单工具。<!-- 来源：框架使用反馈日志.md 第一轮反馈 + D07 K5 "规则状态统计" + D06 决策7 "ATDD效力边界" -->

### 4.2 反馈日志中的信号

**第一轮反馈**（2026-06-26）：<!-- 来源：框架使用反馈日志.md 2026-06-26 第一轮反馈 -->

### 6.3 Hook 状况评估

Hook 是在 pilot-test 中才真正搭建的机械强制层。当前 4 条 Hook + 1 条 SessionStart 注入，实际能力如下：<!-- 来源：settings.json 实际内容 + 新窗口使用说明.md §四 "预期不工作的地方" -->

| Hook | 触发时机 | 阻断方式 | 已知问题 |
|:----|:-------:|:-------:|:--------|
| H0 SessionStart | SessionStart | 注入 reminder | 可选实验——与 CLAUDE.md 文字重复可能产生噪音 |
| H1 验证来源 | PostToolUse: Write | stderr 警告 | 仅当文件含外部引用时触发，非泛用检查 |
| H2 隔离边界 | PreToolUse: Write | exit 2 阻断 | 依赖 `..` 检测——若 `${CLAUDE_PROJECT_DIR}` 未传则跳过；不测绝对路径越界 |
| H3 阶段门控 | PreToolUse: Write | stderr 警告 | 故意设计为不阻断（R3 人决策） |
| H4 行数限制 | PostToolUse: Write | stderr 警告 | 仅监控 SKILL.md，不覆盖 CLAUDE.md |

**关键缺陷**：
1. Hook 全部是 PowerShell 脚本——Windows Restricted 执行策略可能阻断所有 `.ps1` 运行（需用户手动 `Set-ExecutionPolicy RemoteSigned`）<!-- 来源：新窗口使用说明.md §四-3 -->
2. H3 不阻断——设计上故意如此（R3），但实际意味着阶段门控仍是软约束
3. 没有 PreToolUse 对非 Write 工具的干预——例如"AI 在没有 `产出/` 产物时直接调用 WebSearch 做调研"无法阻断
4. 没有 Feedback Loop Hook——违反规则后自动记录到文件的功能未实现

### 6.4 CLAUDE.md 超长问题

CLAUDE.md 当前约 160 行，接近"理想 < 200 行"上限（转接文档 §8 的规范来源）。§5（分层与谨慎度）约 50 行——如果移出可大幅精简。但转接文档的起草原则也提示"不写新规则，从全局 CLAUDE.md 已有规则裁剪"——目前已在做。

## §7 后续可能方向

### 7.1 方向一：沙盒验证 + 修复

这是当前的直接下一步。按新窗口使用说明的 6 项验证清单跑一遍，确认 Hook 的工作状态：

1. H2 隔离阻断 → 确认物理安全

5. 流程深度判定触发 → 确认种子流程可用

### 7.2 方向二：跑真实用例

沙盒验证通过后，在 pilot-test 中跑第一个真实用例（如"个人读书卡片工具"或"导师分析"），完整走 4 阶段。目标是：

- 暴露"五脏俱全的废物"——有没有 Hook 都能发现框架效果差的地方

- 验证阶段间信息传递的设计是否够用
- 判断是否需要回灌根项目同名文件

**用户已确认 F2**——先个人使用，不分发。所以跑真实用例的意义是让工具为自己工作，不是为了做样板。<!-- 来源：新窗口使用说明.md §六 "验证通过后的下一步" + 转接文档.md §11 F2 -->

### 7.3 方向三：Hook 机械强制层补强

2. **非 Write 工具干预**——目前 PreToolUse 只检查 Write/Edit。AI 在无产出文件时直接读网页、调用 MCP 无法阻断。<!-- 来源：框架优化计划书 v0.2 §6 "风险与备选" -->
3. **H3 从警告升级为可配置阻断**——如果用户主动启用"严格模式"，阶段门控可改为 exit 1。当前设计为 R3（人决策）所以是软警告。<!-- 来源：框架优化计划书 v0.2 阶段3 提议 + 新窗口使用说明.md §四-2 -->
