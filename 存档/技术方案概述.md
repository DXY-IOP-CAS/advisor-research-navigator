## 1. 工具定位（已确认）

**一句话**：给定一个导师，输出四份结构化文档——从调研到教学，帮助学生快速理解该导师的研究方向。

**三不原则**：
- ❌ 不做导师口碑/评分——不评价人品、管理风格、毕业难度，避开所有争议漩涡
- ❌ 不做学生→导师匹配——不与 PhdFit/PhDuo 等匹配平台竞争
- ❌ 不做专业内容创作——写路径索引和学习路线，不写教材正文

**三大职能**：调研（收集信息）→ 综述（分析串联）→ 教学（设计递进路线），三职能递进，最终产出学习讲义。

## 2. 学者身份标识体系（ID 与适用场景）

调研发现，过去一份调研文档（`06_导师信息调研与方法.md`）只列了通用平台，遗漏了**学科专用数据库**。这是一个关键缺陷——不同学科的"权威 ID 来源"不同。

### 2.1 跨学科通用 ID

| ID 类型 | 唯一性来源 | 覆盖率 | 可靠性 | 关键 API |
|:-------|:----------|:------:|:------:|:---------|
| **ORCID** | 注册时签发 | 低（特别是中文学者） | 极高（一旦注册就是唯一） | `pub.orcid.org/v3.0/expanded-search?q=email:...` |
| **OpenAlex Author ID** | ML 聚类（姓名+机构+合作者+概念+引用+ORCID） | 高 | 中（中文名+跨机构聚类不干净，实测张鹏举返回 25 候选） | `api.openalex.org/authors?search=...&filter=last_known_institutions.id:{ROR}` |
| **Semantic Scholar Author ID** | ML 聚类 | 高 | 中 | `api.semanticscholar.org/graph/v1/author/search?query=...` |
| **Google Scholar Profile** | 用户自建 | 中 | 中 | **无官方 API**，仅能通过 Web Search 抓 `site:scholar.google.com` 命中，Serper/Exa 部分支持；约 100 次请求后触发 CAPTCHA |
| **ArXiv Author ID** | 用户主动 claim 论文后生成（2005 起 authority records，2009 起公开） | 中（看领域活跃度） | 高（一旦生成） | `arxiv.org/a/{lastname}_{firstinitial}_{n}` 或 ORCID 索引 |
| **CrossRef 作者标识** | DOI 元数据中的 Author ID | 中 | 中 | CrossRef API |

### 2.2 学科专用数据库（v2.0 关键补充）

| 学科 | 数据库 | URL / API | 权威性 | 适用场景 |
|:-----|:------|:-----------|:------|:---------|
| **高能物理/粒子物理/核物理** | **INSPIRE-HEP** | `inspirehep.net/api/{authors\|literature}` | **事实标准**——该领域学者人人都有 | 第一首选 ID（百度万级） |
| **天文学/天体物理** | **NASA ADS** | `api.adsabs.harvard.edu/v1/search/query`（需免费 API key） | 事实标准 | 第一首选 |
| **凝聚态/原子分子物理/光学** | 无单一权威——依赖 OpenAlex + ArXiv + APS | — | — | 通用 ID |
| **计算机科学** | DBLP（`dblp.org/pid/...`） | `dblp.org/search/author/api` | 事实标准 | 物理领域用不到 |
| **生物医学** | PubMed / ORCID | `ncbi.nlm.nih.gov/entrez/eutils` | — | 物理领域用不到 |
| **中文社科/部分理工** | CNKI / 万方 | 需授权/付费 | — | 暂不接入 |

### 2.3 ID 体系的真实工作方式

**核心观察**：几乎所有"权威 ID"都不是全自动生成的——它们依赖学者本人主动注册/claim。

- ORCID：学者主动注册，否则没有
- ArXiv Author ID：学者主动 claim 自己提交的论文，否则可能错配
- Google Scholar Profile：学者主动注册
- INSPIRE-HEP：高能物理领域基本人手一个（机构级别强制要求）
- OpenAlex / Semantic Scholar：**唯一不依赖学者主动行为的**——它从论文元数据自动聚类，但代价是准确率有限

**对我们工具的含义**：必须并行查询多个 ID 源，根据候选人出现与否决定后续权重。

### 2.4 物理学科 ID 查询优先级（v2.0 新增）

根据张鹏举（IPHY 超快物质科学中心，特聘研究员）的实测和学科判断，建议优先级：

```
1. INSPIRE-HEP（如方向属高能/核物理 → 必查，权重最高）
2. ArXiv Author ID（如作者在 ArXiv 活跃 → claim 论文后即有 ID）
3. OpenAlex（通用 fallback，覆盖最广）
4. Semantic Scholar（补充 TLDR 和引用图）
5. ORCID（黄金锚点，但中国学者覆盖率低）
6. Google Scholar Profile（仅通过 Web Search 搜，不直调）
7. 机构主页（用户给定的 URL）
```

**测试实例张鹏举**：超快科学/强场物理/阿秒科学方向，属于凝聚态+原子分子光物理交叉。ArXiv 活跃（强场物理领域 ArXiv 是主要发布渠道），应优先查 ArXiv Author ID。

## 3. 数据源层（v2.0 完整版）

按数据源**抓取协议**和**对工具的意义**分类：

### 3.1 学术数据库 API（核心数据）

| 数据源 | 协议 | 免费额度 | 覆盖率 | 在工具中的角色 |
|:-------|:----|:--------:|:-------|:--------------|
| OpenAlex | REST API (无 key) | 10 req/s（带 email 为 polite pool），100K/天 | 2.5 亿+论文 | 通用 fallback——任何导师都至少能拿到一份 |
| Semantic Scholar | REST API（可选 key） | 100 req/5min（无 key），更高（带 key） | 2 亿+论文 | TLDR、引用图、影响力指标 |
| ArXiv API | REST API | 1 req/3s | 预印本 | 最新论文，强场物理等领域必查 |
| ORCID | REST API | ∞ | 研究者 profile | 黄金锚点（按 email 查） |
| INSPIRE-HEP | REST API | ∞ | 高能物理领域 100% | 高能/核物理方向首选 |
| NASA ADS | REST API（需 key） | 5000 query/天 | 天体物理 | 天文方向才用 |
| CrossRef | REST API | 50 req/s（polite pool） | DOI 注册库 | DOI 解析和元数据校验 |
| PubMed | REST API | 3 req/s | 生物医学 | 物理领域用不到 |

### 3.2 网页抓取（中文/特有信息）

| 来源 | 协议 | 内容 | 在工具中的角色 |
|:-----|:----|:-----|:--------------|
| 机构官网 | HTTP 抓取（MCP fetch） | 中文介绍、招生信息、实验室介绍 | 用户给定 URL 后深度抓取 |
| Google Search | Serper / Exa / Tavily | 综合搜索结果，含 Google Scholar profile 链接 | 第一层发现层 |
| 百度百科 | Web 搜索 | 人物词条（覆盖率低） | 补充 |
| 知乎/小红书/微博 | Web 搜索 | 学生评价、学者动态 | 补充（**不主动采评价，仅作参考**） |
| INSPIRE-HEP 网页 | HTTP 抓取 | 学者主页（含完整论文列表） | 当 API 受限时的 fallback |
| Google Scholar 个人页 | HTTP 抓取（限频） | h-index、引用统计 | 仅当 Web Search 命中后抓取，避免 CAPTCHA |

### 3.3 不引入

- ❌ Scopus / Web of Science——付费
- ❌ CNKI / 万方——需授权
- ❌ 重型图数据库（Neo4j 等）——零部署原则

## 4. 多源协作的真实技术模式

调研了 9 个同类工具后，归纳出 4 种主流技术模式：

### 4.1 supervisor 模式：Web Search 优先

**代表**：`ybq22/supervisor`（137 stars），从"ArXiv 优先"明确转向"Google Search 优先"。

**核心做法**：
- 4 种 Google Search 并行：一般+机构、Wikipedia、Google Scholar、个人主页
- Puppeteer 浏览器自动化（不适合我们的环境）
- 用 ArXiv 和 Semantic Scholar 仅做补充

**对我们的启示**：Web Search 是发现阶段的主力，API 是确认和深挖阶段的工具。

### 4.2 scholar-megasearch 模式：Fan-out + 去重

**代表**：`TaewoooPark/scholar-megasearch`，20+ 数据库并行 fan-out。

**核心做法**：
- 把不同数据库分配给独立 subagent
- Dedup 策略：DOI → arXiv-id → 归一化标题（≥85% 相似度）
- 排名公式：`(number of sources, citation count, year)`

**对我们的启示**：多源采集后必须有去重和排序，不是简单堆数据。

### 4.3 ai-talent-graph 模式：4 API 并行

**代表**：`rrrrrredy/ai-talent-graph`，4 个免费 API（OpenAlex、S2、ORCID、arXiv）。

**核心做法**：
- 4 个 API 并行调，不做 Web Search
- 数据靠"互相补"，OpenAlex 不准的地方用 ORCID 校准

**对我们的启示**：纯 API 路线可行，但要承认覆盖率局限。

### 4.4 CrossND / LEAD 学术消歧算法

**代表**：`zfjsail/CrossND`（清华）、`LEAD`（2025）。

**核心做法**：
- ML 聚类 + 跨源一致性校验
- 用 LLM 提取语义特征 + 合作者网络结构特征
- F1 = 0.967（LEAD 在意大利数据集上）

**对我们的启示**：纯 ML 方案准确率高但重——我们靠官网 URL ground truth + 论文标题匹配，避免 ML。

## 5. 反思：之前的设计错在哪

v1.0 文档（`技术方案概述.md` 原版）有两个关键缺陷：

### 5.1 数据源层太泛

只列了 OpenAlex/Semantic Scholar/ORCID/arXiv/机构主页/中文百科六个。**漏掉了：**

- 学科专用数据库（INSPIRE-HEP、NASA ADS、DBLP）——对物理领域尤其致命
- ArXiv Author ID 的真实存在——v1.0 把它当成"纯文本搜索工具"是错的
- Google Scholar 的反爬限制——v1.0 没说明怎么访问

### 5.2 误以为 OpenAlex 是唯一主源

v1.0 把 OpenAlex 当作默认入口，其他都做补充。**真实情况是：**

- 对中文名字 + 跨机构，OpenAlex ML 聚类不干净（张鹏举返回 25 候选）
- 物理学科常用 INSPIRE 和 ArXiv，OpenAlex 不是首选
- Web Search 才是发现阶段的主力工具，API 才是确认阶段的主力

**v2.0 修正**：明确按学科分层，并行查多个 ID 源，不预设 OpenAlex 为唯一主源。

## 6. 待补充：阶段 0 和阶段 1 的具体 Workflow

v2.0 文档到此只更新了"数据源层"和"调研结论"。**阶段 0（身份验证）和阶段 1（基础画像）的具体 Workflow 设计**还需要在 `需求确认.md` 第 2 章 2.2 节单独定稿——这是下一步的工作。

设计原则（从这次调研提炼）：

1. **不要预设单一主源**——根据学科和候选人实际情况调整优先级
2. **Web Search 和 API 并行**——Web Search 做发现，API 做确认和深挖
3. **每个数据源失败都有 fallback**——OpenAlex 找不到用 ArXiv，ArXiv 找不到用 INSPIRE
4. **人闸必须**——任何 ID 锁定前用户确认，避免连锁错误
