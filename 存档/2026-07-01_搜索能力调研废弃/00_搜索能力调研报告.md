## 二、直接相关项目

### 2.1 ai-talent-graph（rrrrrredy/ai-talent-graph）

**定位**：Claude Code Skill，输姓名/机构 → 全平台学者画像

**数据源**：OpenAlex（主）+ Semantic Scholar + ORCID + arXiv

**结构**：
```
SKILL.md
scripts/
  ├── talent_graph.py     # 统一 CLI 入口
  ├── openalex_api.py     # 每个源一个独立 wrapper
  ├── arxiv_api.py
  ├── semantic_scholar.py
  ├── orcid_api.py
  └── github_api.py
```

**核心能力**：
- 按研究主题搜学者（带 h-index 过滤）
- 输姓名生成完整画像（论文/引用/h-index/机构/履历）
- 按机构查 top 学者

**身份消歧**：无显式人闸。依赖 OpenAlex API 自带的消歧 + ORCID 验证。

**可借鉴**：脚本架构（每个源独立 wrapper + 统一入口），不依赖 Claude 内联 Bash。

### 2.2 scholar-lab-radar（TaewoooPark/scholar-lab-radar）

**定位**：Claude Code Skill，输 PI 身份 → 拉全组论文 → 构建时间线知识图谱

**数据源**：OpenAlex（主）+ ORCID + Crossref + Semantic Scholar

**核心能力**：
- 分辨 PI → 拉全组论文 → 知识图谱（Topic/Theme/Method/合作者关系）
- 区分 OpenAlex Topics 和研究组特有的 Themes
- 时间线标注（研究方向的演化）
- 合作网络可视化

**身份消歧**（原文引用）：
> "Author disambiguation is the make-or-break step. The corpus is exactly the confirmed PI author id(s); a wrong id taints the whole report. Always confirm — prefer ORCID, and merge ids when OpenAlex splits a person."

**可借鉴**：
- 和我们的阶段2/3理念一致（领域调研后再回看论文）
- 区分 Topic（通用分类）和 Theme（组特有方向）
- 强调消歧是成败关键

### 2.3 nature-skills/academic_search.py（Yuan1z0825/nature-skills）

**定位**：Claude Code Skill，学术搜索工具

**数据源**：OpenAlex

**核心能力**：
- 按主题搜论文（支持排序过滤）
- 按作者搜论文（姓名 → 机构过滤 → OpenAlex ID 合并）
- 按 ORCID 搜（最可靠）
- 按 OpenAlex ID 精确搜
- `--list-authors` 列出所有同名候选人，让你选

**身份消歧（精华）**：

```python
def resolve_author(name, affiliation=None):
    1. 搜 OpenAlex 候选人
    2. 用机构名过滤（同姓名但不同机构的排除）
    3. 同姓名+同机构的 API 记录合并为一个 ID
    4. 排序：选发文最多的机构候选为锚
    5. 返回 IDs + 候选人完整列表（供人工审查）
```

**可借鉴**：
- 机构约束消歧是最直接解决中文重名的方法
- `--list-authors` 交互让用户选
- 比我们的论文标题比对更简洁

**定位**：Claude Code Skill，把某位导师"蒸馏"成一个可复用的对话 Skill

**核心能力**：

## 四、技术路线对比

### 身份消歧方案

| 方案 | 代表项目 | 优点 | 缺点 |
|:-----|:---------|:-----|:------|
| ORCID 优先 | scholar-lab-radar, ai-talent-graph | 一对一，最快 | 注册率低 |
| 姓名+机构过滤 | nature-skills/academic_search.py | 消歧+合并一步完成 | 依赖机构数据准确 |
| 论文标题比对 | pilot-test（当前） | 不依赖机构 | 操作繁琐，依赖官网有论文列表 |
| OpenAlex XGBoost | OpenAlex 官方 | 全自动 | 中文名跨机构不可靠 |

### 脚本架构方案

| 方案 | 代表项目 | 优点 |
|:-----|:---------|:------|
| 每个源独立 .py + 一个主入口 | ai-talent-graph | 清晰，可单独升级 |
| 全在一个大文件里 | pilot-test（当前identity_resolver.py） | 混乱 |
| 通过 Agent/MCP 工具调 API | Scholar-MCP, academic-research-mcp | 灵活，API 调用实时可控 |

## 五、对我们的启示

1. **身份消歧简化方案**：搜 OpenAlex → 用机构 ROR 过滤 → 同姓名同机构合并 → 你确认 → 锁定。把论文标题比对当备选方案（ROR 过滤失败时启用）

2. **脚本架构重构**：`openalex_api.py` + `arxiv_api.py` + `orcid_api.py` + `main.py`（统一入口），不依赖 `python -c` 内联

3. **补充中文数据源**：百度学术、知网（CNKI）国内学者覆盖率远超 OpenAlex

4. **双源去重合并**：OpenAlex + Semantic Scholar 交叉去重，论文更全

5. **MCP 工具化**：长期应该做成 MCP 工具链，而不是纯 Python 脚本

## 六、搜索质量的反思

本次搜索暴露的问题：
1. 搜索工具单一（仅 Exa）
2. 关键词覆盖不足（中文、前端、论文库方向没搜）
3. 搜到的项目没有深读代码（只看 README）
4. 没有交叉验证（同样关键词应该用 Serper/Tavily 再搜一轮确认无遗漏）

**根本原因**：我没有将"搜索"本身视为需要系统性设计的能力，而是当成了一个一次性的查资料动作。
