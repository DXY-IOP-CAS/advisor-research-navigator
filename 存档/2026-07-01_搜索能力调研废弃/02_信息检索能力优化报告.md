## 二、信息检索的核心流程

搜遍所有材料后，业界的共识收敛为同一个流程：

```
问题 → 分解 → 多路并行搜索 → 评估结果 → 识别缺口 → 深挖 → 交叉验证 → 综合输出
                                                    ↑                      ↓
                                              缺口还多 → 继续搜     写报告
```

这不是线性流程，是**带反馈回路**的循环。核心思路来源：

| 思路 | 出处 | 一句话 |
|:-----|:------|:-------|
| Plan-and-Solve | GPT-Researcher (arXiv:2305.04091) | 先规划子问题，再并行执行 |
| 多角度提问 | STORM (arXiv:2402.14207) | 同个问题从不同角度搜，覆盖更广 |
| 搜索-评估循环 | Web Search Agent Loop (AgentPatterns) | 搜→评→补→再搜，直到信息充足 |
| 由宽到窄 | Anthropic Multi-Agent Research | 先宽泛搜索摸清全貌，再聚焦深挖 |
| 缺口驱动 | Gap-driven Reformulation (同上) | 基于已找到的信息判断缺什么，而非原地换词 |
| 交叉验证 | Multi-Agent Research (RAGentA) | 多源证据对同一事实校准 |

## 三、九条核心原则

### 原则 1：问题分解（Question Decomposition）

最基础也最高效的手段。**复杂问题拆成独立子问题，分别搜索再合并。**

反面例子：
```
"调研张鹏举" → 一次搜出所有信息 → 结果混乱
```

正面例子：
```
"调研张鹏举"
  → 子问题1: 身份信息（姓名/机构/邮箱）
  → 子问题2: 学术履历（教育/工作经历）
  → 子问题3: 论文列表（研究方向/引用数据）
  → 子问题4: 公开信息（新闻/采访/讲座）
  → 分别执行 → 合并结果
```

**来源**：Anthropic Question Decomposition 论文 / TypeGraph Agentic Retrieval

### 原则 2：先宽后窄（Broad-to-Narrow）

**不要一上来就搜长尾关键词，先搜宽泛的摸清全貌，再收窄。**

Anthropic 原文：
> "Agents often default to overly long, specific queries that return few results. We counteracted this tendency by prompting agents to start with short, broad queries, evaluate what's available, then progressively narrow focus."

实测效果：Anthropic 报告称多Agent搜索相比单Agent搜索提升了 90.2%。

**来源**：Anthropic "How we built our multi-agent research system"

### 原则 3：多角度提问（Multi-Perspective）

**对一个主题，从不同角度生成搜索查询，不是原地换同义词。**

STORM (Stanford) 的做法：
1. 识别多个相关角度
2. 每个角度派一个"记者"提问
3. 多轮对话深挖
4. 综合结果

**示例**：调研开源项目
- 角度1：功能维度 → "what features does X have"
- 角度2：对比维度 → "X vs Y comparison"
- 角度3：生态维度 → "X GitHub contributors issues"
- 角度4：用户维度 → "X review tutorial getting started"

**来源**：Stanford STORM / arXiv:2402.14207

### 原则 4：缺口驱动迭代（Gap-driven Iteration）

**不是机械地搜固定次数。每次搜索后评估已找到的信息还缺什么，下一个查询针对缺口。**

核心循环：
```
search → evaluate → identify gaps → generate follow-ups → search
```

评估维度：
- 覆盖度：这个角度的信息够了？
- 矛盾度：不同来源说法一致？
- 置信度：来源权威吗？是官方还是一面之词？
- 时效性：信息够新吗？

**来源**：Web Search Agent Loop / Anthropic Multi-Agent Research

### 原则 5：工具选择策略（Tool Routing）

**不同搜索任务用不同的工具，不是一把抓。**

| 场景 | 首选工具 | 原因 |
|:-----|:---------|:------|
| 搜学术论文/arXiv | Exa | 语义搜索理解论文标题 |
| 搜 GitHub 项目 | Exa | 语义搜索，理解项目描述 |
| 搜新闻/事实核查 | Serper (Google) | 时效性好，覆盖面广 |
| 深度综合调研 | Tavily | 多源融合，长文本 |
| 需要交叉验证 | 2+ 工具同时搜 | 同一关键词，不同结果对比 |

**来源**：Anthropic "Writing effective tools for AI agents" / You.com Tool Design

### 原则 6：分离发现与深读（Separate Discovery from Deep Reading）

**搜索（discovery）和读取（fetch）是两个不同阶段。**

- 搜索阶段：用 `web_search` 批量化获取标题和摘要，筛选潜在价值
- 深读阶段：只对筛选后的结果用 `web_fetch` 读全文
- 目标导向读取：读页面时只提取与当前目标相关的部分

这样做的好处是搜索阶段的上下文消耗极低（只看摘要），深读阶段的上下文花的值得（只读高价值内容）。

**来源**：You.com Tool Design / AgentPatterns.ai

### 原则 7：信息评估与过滤（Evaluation & Filtering）

**搜索到的结果不是全都用。每次搜到信息后先做一轮评估：**

```
1. 相关性：和当前问题有关？是的话保留
2. 来源可信度：官方文档 > 技术博客 > 论坛帖子 > 未经验证的第三方
3. 时效性：需要最新信息还是历史信息？
4. 交叉验证：多个独立来源说法一致？
5. 低质量结果过滤：低于阈值的不进上下文
```

**来源**：Self-Corrective RAG / Redis RAG Best Practices

### 原则 8：信息综合（Synthesis）

**不是简单地粘贴搜索结果，而是综合多源信息产出结构化的输出。**

综合要点：
- 不同来源的**共识**是什么
- 不同来源的**矛盾**是什么（列出不仲裁）
- 哪些信息是**验证过的**（多源确认）
- 哪些信息是**单一来源**（标注，未验证）

**来源**：GPT-Researcher / RAGentA Multi-Agent Framework

### 原则 9：追踪来源（Source Tracking）

**每条信息标记来源 URL，让最终输出可溯源。**

标准做法：
- 每条外部信息带来源链接
- 引用格式统一
- 区分"已验证"（多源）和"仅X来源"（单源）

**来源**：GPT-Researcher citation mechanism / Marting Fowler Reliable Agentic AI

## 四、具体操作层面的改进

### 4.1 搜索前

| 步骤 | 时间 | 产出的东西 |
|:-----|:-----|:-----------|
| 明确问题 | 10秒 | 一句话描述要搜什么 |
| 分解子问题 | 30秒 | 2-5 个子问题列表 |
| 选择工具 | 10秒 | 每个子问题选工具 |
| 设计搜索词 | 30秒 | 每个子问题 1-2 个搜索词 |
| 规划深挖方向 | 30秒 | 预期一轮搜索后可能跟进的方向 |

### 4.2 搜索中

```
一. 第一轮（宽泛搜索）
   → 每种子问题用 1-2 个工具并行发
   评估结果

二. 识别缺口
   → 哪些子问题已覆盖？哪些还没？
   缺口有哪些？

三. 第二轮（深挖）
   → 针对缺口换关键词/换角度
   可能换工具

四. 交叉验证
   → 关键事实用不同工具再查一轮
   矛盾点标注

五. 综合
   → 所有结果合并
   去重 → 评估 → 结构化输出
```

### 4.3 搜索后

输出结构化的报告，包含：
- 信息来源（URL）
- 验证状态（多源/单源/未验证）
- 仍存在的缺口或不确定点
