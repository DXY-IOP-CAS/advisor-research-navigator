## 二、核心发现：AI Agent 搜索的质量为何差，怎么修

### 2.1 为什么我之前的搜索质量差

对照当前研究，我的搜索问题可归纳为：

| 问题 | 当前做法 | 正确做法 |
|:-----|:---------|:---------|
| **无规划** | 想到一个关键词搜一次，不顺就换 | 先分解问题 → 生成多组关键词 → 并行搜索 |
| **无多角度** | 同一角度换词重搜 | 每次搜索都换角度（学术/工程/中文/前沿论文） |
| **无多轮迭代** | 搜一次就停止 | 第一轮搜宽，第二轮选关键点深搜，第三轮深读代码 |
| **无并行搜索** | 串行一次调用一个工具 | 同时发起 3-5 路搜索 |
| **无交叉验证** | 一个工具查不到就放弃 | 同样的关键词用不同工具（Exa+Tavily+Serper）各搜一次 |
| **无深读** | 只看 README 简介 | 关键项目要读源代码和论文 |

### 2.2 业界成熟的搜索方法论

#### 方法一：Plan-and-Solve（GPT-Researcher 核心）

**论文**：Plan-and-Solve Prompting (arXiv:2305.04091)

**流程**：
```
用户查询
   ↓
1. 规划阶段：分解为 N 个研究子问题
   ↓
2. 执行阶段：每个子问题派一个搜索 Agent（并行）
   ↓
3. 汇总阶段：合并所有结果 → 去重 → 引用溯源
   ↓
4. 校验阶段：Reviewer Agent 检查准确性
   ↓
5. 输出阶段：生成带引用的报告
```

**关键参数**：
- `breadth`（广度）：每层生成的搜索查询数。值越大，覆盖越广
- `depth`（深度）：递归探索的层级数。值越大，挖掘越深
- 典型配置：`breadth=4, depth=2` → 首层4条搜索 → 每结果深挖2层

**来源**：https://github.com/assafelovic/gpt-researcher

#### 方法二：STORM 多角度提问（Stanford）

**论文**：STORM (arXiv:2402.14207)

**核心洞察**：搜索质量的关键不是"怎么搜"，而是"提出好问题"。LLM 直接提问效果差，需要：

```
输入主题
   ↓
1. 发现角度：在相似主题的文章中提取不同视角（perspectives）
   ↓
2. 模拟对话：每个角度派一个"记者"，和"专家"做多轮对话
   ↓
3. 分解问题：每个问题拆成多个搜索查询
   ↓
4. 筛选来源：按可信度过滤
   ↓
5. 综合回答：基于来源生成答案，再提出更深的问题
```

**示例**：调研"张鹏举"这个老师
- 角度1：超快光学方向 → 搜"张鹏举 阿秒 HHG"
- 角度2：液体动力学方向 → 搜"张鹏举 液体水 光电子"
- 角度3：合作网络 → 搜"ETH Zurich Wörner 超快 中国"
- 角度4：国内影响 → 搜"物理所 超快物质科学 张鹏举"

**来源**：https://github.com/stanford-oval/storm

#### 方法三：Agentic RAG（多阶段检索）

区别于传统 RAG（单次向量检索），Agentic RAG 的 Agent 控制整个检索流程：

```
问题 → Agent 判断需要什么信息
   → 决定查哪个索引（论文/履历/新闻）
   → 调用对应工具
   → 读结果，判断是否需要补充搜索
   → 迭代直到信息充足
   → 综合回答
```

**关键优势**：不是一次性搜索，而是**迭代推进**——每次搜索的结果决定下一步搜什么。

**来源**：https://www.mindstudio.ai/blog/what-is-agentic-rag-multi-layer-retrieval

## 五、对我们项目的具体改进方案

### 5.1 搜索流程重构

当前（一次搜索）：
```
想到一个关键词 → 搜一次 → 不满意 → 换词再搜 → 放弃
```

改进后（结构化搜索）：
```
搜索任务
   ↓
1. 分解子问题（Plan）
   生成 3-5 个搜索角度
   ↓
2. 第一轮并行搜索（Execute）
   每个角度用不同关键词 + 不同工具
   Exa + Serper + Tavily 同时发
   ↓
3. 筛选关键结果（Analyze）
   找出最有价值的 2-3 个项目/文章
   ↓
4. 第二轮深搜（Deep Dive）
   读代码、读论文、读 README
   ↓
5. 交叉验证（Verify）
   相同关键词用不同工具再查一轮
   ↓
6. 输出报告（Synthesize）
   结构化文档，标注来源
```

### 5.2 搜索工具选择策略

| 场景 | 首选 | 备选 |
|:-----|:------|:------|
| 找 GitHub 项目 | Exa（语义搜索效果最好） | Serper |
| 找前沿论文 | Exa（学术语义） | Semantic Scholar MCP |
| 找中文内容 | Serper（Google 中文） | Exa 中文搜索 |
| 深度调研 | Tavily Research（多源融合） | 多轮 Exa + Serper |
| 事实核查 | Serper（Google 时效性好） | WebSearch 内置 |

### 5.3 搜索的核心原则

1. **搜索 = 迭代过程，不是一次性动作**。首轮搜宽，后续收窄
2. **同时用 2+ 个工具**。同样的关键词用不同工具交叉验证
3. **每轮至少换一个角度**。不要原地换词，要换视角
4. **搜到关键项目后必须深读**。README 不决定一切，代码才决定
5. **文档记录搜索过程**。记录搜了什么、找到了什么、放弃了什么
