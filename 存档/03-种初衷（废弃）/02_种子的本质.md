# 02. 种子的本质

**本件不变性**：克隆时必须带着。

---

## 种子不是文档

种子不是一份说明书、一份规范、一份标准答案。文档是死的，种子是活的。

文档的价值在"读完之后人懂了"——种子的价值在"用了几次之后工具变好了"。

---

## 种子不是框架

框架是固定结构——四阶段、五个角色、若干规则。框架的成长靠"人改框架"。

种子是结构 + 生长机制——结构给骨架，机制让工具按反馈自动逼近用户需求。种子的成长靠"用 + 反馈"，不需要人定期改种子本身。

---

## 种子不是 Agent

种子不替代 Agent、不指挥 Agent、不和 Agent 抢活。种子**驱动** Agent 的行为——告诉 Agent"在这个项目里你应该按什么流程做事"。

Agent 是手和脚，种子是大脑里的工作习惯。

---

## 种子 vs 工具的关系

```
种子 ──克隆──> 工具项目（pilot-test 是第一个实例）
                │
                ├─> CLAUDE.md（宪法，约束 Agent 行为）
                ├─> SKILL.md（流程，引导 Agent 做事）
                ├─> Hooks（机械层，强制规则不可绕过）
                ├─> 产出/（工具产生的实际成果）
                └─> 初衷/（种子的基因，克隆时必带）
```

**克隆 = 把 `pilot-test/` 整个复制一份到新文件夹**——新文件夹里 Agent 一启动就自动加载这些文件，按种子的流程做事。

---

## 为什么是"种子"而不是"模板"

- 模板是**死答案**——给你 5 步你照做
- 种子是**活机制**——给你"先问清楚再动手"的本能，你每一步问清楚当下场景的具体答案

模板不能进化（除非人改）。种子可以进化（用户每次反馈都会让种子长出的工具变好）。

---

## 这份文件何时被改

- 不变。种子的本质定义本身就是种子的"基因"。
- 如果将来发现种子本质表述错了——只追加新理解，不改旧表述（避免上下文腐烂后失去原意）。
