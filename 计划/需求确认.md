## Block 1: 需求背景

### 1.1 起因 / 工具目标

每年硕士/博士生选导师是刚需。通常学生在研一 3-4 月面临报志愿套磁,需要快速判断与某个导师研究方向的匹配度。但在调研过程中,普遍遇到四个痛点:

- **官网信息粗糙**——老师官网只有论文列表和标题,没有科普级解读,看不出这个方向到底在做什么
- **有论文看不懂**——论文建立在研究生以上知识基础上,学生只有本科基础,缺少前置知识
- **资源散乱无索引**——论文、教材、视频、B站、博客各在各处,没有统一的路径索引
- **调研极易绕圈**——不知道"该读什么、按什么顺序",在信息海里漫游浪费大量时间

为解决这些问题,工具按四段层层嵌套的方式运作。每一段不是孤立执行,而是前一段的输出为后一段提供**认知上下文**——范围逐段收窄,理解逐段加深。工具同时承担三个职能:**调研**(全面收集信息)、**综述**(分析串联知识)、**教学**(设计递进学习路线),三职能递进,最终产出学习讲义:

| 段 | 做什么 | 如何做(高层次) | 为什么要做 |
|:---|:---|:---|:---|
| 第1段 | 老师基础信息调研 | 地毯式收集老师多维信息(官网、学术平台、中文百科、采访稿等),多源交叉比对 | 输出老师基础画像,明确老师所在领域范围 |
| 第2段 | 相关领域发展调研 | 基于第1段明确的领域范围,追溯该子领域的发展脉络,理清技术路径和当前阶段 | 理解该领域在大专业中的位置,输出认知地图 |
| 第3段 | 老师论文串联与定位 | 以第2段的领域认知为框架,从专业视角重新审视老师的论文序列 | 把老师的研究放在领域框架里定位,输出论文时间线+前沿定位 |
| 第4段 | 从零到前沿学习讲义 | 基于前三段对老师和领域的深入理解,从高数+线代起步反向设计学习路线 | 输出单线递进式讲义(小字典),让用户快速入门老师方向 |

嵌套逻辑概括: **大领域→小领域→老师定位→学习路径**,后一段建立在前一段构建的认知之上,信息量逐层收窄、理解深度逐层加深。

> 注:四段的具体技术实现细节(用什么调研工具、搜索策略、信息组织方式等)将在技术调研阶段进一步确定和修正。当前表格中的"如何做"仅描述高层次思路,供理解工具设计逻辑。

### 1.2 目标用户画像

工具的四段嵌套服务于这样的用户:

**用户实际场景**: 大二/大三物理相关专业本科生,正在准备申请硕博选导师。研一 3-4 月面临报志愿套磁,需要快速判断与导师研究方向的匹配度。

**工具假设的起点水平**: 工具假设用户从高等数学和线性代数这两个最基础的起点开始推导。实际用户可能已有更多基础(四大力学、固体物理等),但工具仍从最基础起步——因为所有高阶知识(复变函数、数理方法、高等量子力学、群论、量子多体等)本质上都是高数和线代的逐步推广。从根开始推导,保证零跳步,建立真正的理解而非表面记忆。

**"大字典"——教材体系在AI时代的角色转变**:

本科+研一的教材在AI时代不再是"必须一字不差学完的课本",而是**字典/指南针**:

| 层面 | 课程 | 定位 |
|:---|:---|:----|
| 数学基础(确认掌握) | 高数、线代、概率论、复变、数理方法 | 推导的起点 |
| 物理基础(确认掌握) | 普物(力热电磁光原子)、四大力学(理力/电动/初量/热统)、固体物理 | 物理直觉的来源 |
| 高阶延伸(领域范围) | 高量、群论、量子多体、量子场论、规范场论 | "大字典"的边界——定义了该领域触及的范围 |

教材从"课本"到"字典"的角色转变,原因有二:

- 教材面向通用人才培养,涵盖的是一个领域的完整体系,可以构建学生对相关领域的认知框架。但具体到某个方向只用到其中部分知识点，如果学生逐页通读,会在不相关的内容上浪费大量精力。
- AI时代学生可以通过与AI持续对话,基于自己的水平逐步理解知识点。教材的作用是提供**参考坐标**——告诉学生这个领域"有哪些东西、按什么顺序展开、什么重要什么不相关",防止在琐碎知识上跑偏。学生在AI帮助下按需读取字典中的条目,带着研究方向去学,既能深化对具体知识点的理解,又能强化对整个框架的认知。

**"小字典"——工具产出讲义的定位**:

第4段产出的讲义,实质上是一个**从大字典中提取的个性化小字典**:

- 从本科+研一的教材体系(大字典)中,只提取第1-3段调研后确认真正需要的知识点
- 把这些知识点串联成一条从高数+线代到老师前沿的逻辑链
- 让用户走通这条线,而不是把整本教材学完

"小字典"扎根于"大字典"意味着: 讲义以学术界检验过的教材体系为参照系,可以约束AI的创作边界,防止AI幻觉和信息收集偏离轨道。

**一个可扩展的方向**: 工具还可附带教用户如何利用AI配合小字典进行递进学习。AI相比教材的核心优势在于适配性——能根据用户当前水平和弱点动态调整。但很多人并不擅长用AI辅助学习,因此加入"如何用AI学"的引导,可以进一步降低用户门槛。这个方向暂未确定,作为后续讨论的参考。

### 1.3 工具差异化与定位（调研后更新）

同类调研覆盖 18+ 工具（详见 `计划/需求/05_同类工具调研报告.md`），按四段流水线分段评估后发现：**市场无直接竞品**。现有工具分三类，各占一个和我们不重叠的位置。

| 类别 | 代表工具 | 做什么 | 与我们的重叠 |
|:-----|:---------|:-------|:-----------:|
| 导师评价平台 | 导师评价网/GradChoice/advisor-calculator | 口碑评分（人品、管理风格、毕业难度） | ❌ 不重叠—我们不碰评价 |
| 导师匹配平台 | PhdFit/StreamlinedAI/PhDuo | 从学生 CV 出发推荐导师 | ❌ 不重叠—我们不做推荐 |
| 学者画像工具 | supervisor/SciAtlas/ai-talent-graph | 信息聚合（论文+主页 → 报告） | ⚡ 第1段部分重叠 |

学者画像工具与我们在第1段有部分重叠，但它们停在"导师长什么样"——我们的流水线继续往前推到领域发展、论文定位和教学讲义。

**与最接近的工具的边界**：

| 工具 | 它做到哪 | 我们做了它没做的 |
|:-----|:---------|:---------------|
| **supervisor**（GitHub: ybq22/supervisor） | 输出可对话的导师画像 | 领域脉络 + 论文定位 + 从零到前沿学习讲义 |
| **SciAtlas**（GitHub: zjunlp/SciScholar） | 学术KG学者报告（英文） | 中文/机构/新闻多维信息 + 教学流水线 |
| **paper-to-course**（GitHub: KaguraTart/paper-to-course） | 单论文生成6模块课程 | 多论文整合 + 从高数+线代起步的完整路径 |
| **GoAI**（arXiv 2503.08549） | AI领域学习路径生成（CS限定） | 物理等自然科学领域适配 |

**工具的准确定位**：

- **通用模板引擎**：对任何机构的任何老师都适用，跨老师复用零边际成本。第一个测试实例为张鹏举（IPHY 超快物质科学中心）
- **开源方法论**：工具是「Claude Code Skill + Markdown 模板」形态，不是网站/App，零部署
- **从调研走到教学**：调研+综述+教学三职能递进，不只是信息聚合
- **调研驱动设计**：每个设计决策经过同类工具验证，不凭空发明

**一条根本差异**：同类工具的目的是帮学生"找到谁合适"（匹配引擎），或帮学生"知道老师什么样"（画像工具）。我们的目的是帮学生**学懂一个方向**——这个需求没有人在做。

**工具不做的事**：
- ❌ 不写教材正文——讲义是路径索引+知识链，不是教科书
- ❌ 不替代学生主动学习——给路线，不给答案
- ❌ 不点评老师/推荐方向——价值判断用户自己做
- ❌ 不捏造论文/教材链接——所有引用必须有 URL
- ❌ 不评价导师口碑/人品——不陷入争议
- ❌ 不搭建网站——不做部署运维

## Block 2: 工具设计

### 2.1 第1段：身份验证与基础画像

第1段是整个流水线的入口，也是最需要精细设计的环节。核心问题只有一个——**如何从用户给的"张鹏举 + 中科院物理所 + 官网URL"精确锁定学术数据库中的唯一研究者？**

这个问题不是我们独创的。学术界称之为"作者消歧"(Author Name Disambiguation)，OpenAlex、Semantic Scholar、AMiner 等平台都在用机器学习做这件事（考虑因素：姓名相似度、合作者网络、机构归属、研究主题一致性、引用模式、ORCID）。但我们的场景比它们更简单——**用户提供了官网URL，这是一个很强的锚点**。两段式设计利用了这个锚点：

#### 2.1.1 数据流总览

```
用户输入: 姓名 + 机构 + 官网URL
    │
    ▼
┌─ Phase 1a：身份验证（浅调研）─────────────────────────────────────────┐
│  目的：确认我们要爬的是正确的人                                      │
│  手段：用官网URL锚定身份，多源交叉验证                                │
│  产出：00_身份验证卡.md（用户确认后才能进入下一阶段）                  │
└──────────────────────────────────────────────────────────────────────┘
    │  用户确认
    ▼
┌─ Phase 1b：基础画像（深调研）─────────────────────────────────────────┐
│  目的：地毯式获取一切公开学术信息                                   │
│  手段：用确认后的唯一ID拉取各平台数据                                │
│  产出：01_基础画像.md（9节完整画像）                                 │
└──────────────────────────────────────────────────────────────────────┘
```

#### 2.1.2 Phase 1a：身份验证 —— 原理与步骤

**原理**：官网URL是用户提供的，它必然指向目标老师本人。从页面提取的 email、论文标题、研究方向可以当作"身份指纹"，用于在学术数据库中锁定唯一ID。不需要复杂的消歧算法——只需把页面信息和API返回信息做交叉匹配。

**具体执行步骤**：

```
Step 1 ── 抓机构官网页面
  目的：提取身份锚点
  方法：Tavily Extract 或 HTTP fetch 读取 URL 页面内容
  提取字段：
    ├─ 姓名（中文/英文拼写）
    ├─ 职称
    ├─ 邮箱（关键锚点——ORCID 搜索可以用邮箱精准匹配）
    ├─ 研究方向（一段话或几个标签）
    ├─ 论文列表（标题、期刊、年份——如果有的话）
    └─ 教育/工作经历（如果有）
  异常处理：页面无法访问 → 用户确认URL是否正确的链接
            页面不含论文列表 → 跳过论文比对步骤

Step 2 ── 搜索 ORCID 注册库
  目的：找到该老师在ORCID中的唯一ID（如果有）
  方法：调 orcid_resolve_researcher（MCP工具）
        优先用邮箱搜：用 Step1 提取的 email 作为 query
        备用用姓名+机构：姓名 + affiliation 参数
  返回：0-5个候选人，每人附带 name_match_type + institution_overlap 信号
  结果判断：
    ├─ 返回1人且institution_overlap=True → 大概率是本人
    ├─ 返回多人 → 排名靠前的候选
    ├─ 返回0人 → 说明没注册ORCID，跳过此步
  异常处理：ORCID API超时 → 跳过，不影响流程

Step 3 ── 搜索 OpenAlex
  目的：找到该老师在 OpenAlex 中的 Author ID
  方法：调 OpenAlex /authors 接口
        URL: https://api.openalex.org/authors?search={姓名}&filter=last_known_institutions.id:{机构ROR}
  参数：
    ├─ search: 姓名（先用中文，无结果则用英文拼音）
    └─ filter: 机构ROR（从OpenAlex查询机构名称获得）
        中科院物理所的ROR: https://ror.org/05qbk4x57
  返回：候选作者列表，每人带 works_count, cited_by_count, last_known_institutions
  结果判断：
    ├─ 返回且某候选人机构含"中科院物理所"或"ETH Zurich"等 → 高置信度
    ├─ 返回多人 → 用 Step1 的论文标题列表与候选人的论文列表做标题匹配
    └─ 不返回或结果不匹配 → 不阻塞，记录缺失
  异常处理：中文名URL编码问题 → 改用英文拼音重试

Step 4 ── 搜索 Semantic Scholar
  目的：辅助验证身份，补充引用数据
  方法：调 Semantic Scholar /author/search 接口
        URL: https://api.semanticscholar.org/graph/v1/author/search?query={姓名}
  返回：候选作者列表，带 paperCount, citationCount
  结果判断：与 Step1-3 的数据交叉比

Step 5 ── 论文标题交叉比对（核心消歧手段）
  目的：在多个候选人中确定谁是目标本人
  方法：把 Step1 从官网抓到的论文标题分别与每位候选人的论文列表做匹配
        匹配度 = 官网论文列表中有多少比例出现在该候选人的论文列表中
  判定规则：
    ├─ 匹配度 > 80% → 锁定，高置信度
    ├─ 匹配度 50-80% → 最可能候选人，标注置信度
    └─ 匹配度 < 50% → 多个候选人都有部分匹配 → 让用户选择

Step 6 ── 输出身份验证卡
  格式：00_身份验证卡.md（半页，现场出结果）
  内容见下方模板
```

#### 2.1.4 Phase 1b：基础画像深调研

用户确认身份后，用已锁定的唯一ID进行纵深数据采集。

**采集方法**：

| 数据维度 | API/工具 | 参数 | 产出字段 |
|:---------|:---------|:-----|:---------|
| 论文列表+引用 | OpenAlex `/authors/{id}/works` | Author ID, sort=cited_by_count:desc | 标题、DOI、期刊、年份、引用数、作者列表 |
| 论文TLDR/领域 | Semantic Scholar `/author/{id}/papers` | S2 Author ID | TLDR摘要、领域标签、影响力 |
| 最新预印本 | arXiv API `search` | 姓名+机构 | 最新论文（可能有OpenAlex未收录的） |
| 合作者分析 | OpenAlex works 返回的作者 | 从论文列表中提取 | 高频合作者列表 |
| 中文网络信息 | Web Search | 姓名+机构 | 新闻、采访、学术报告通知 |
| 机构页面补充 | HTTP fetch | 机构域名 | 实验室介绍、团队信息 |

**重要经验**（从实际测试跑张鹏举所得）：

1. **OpenAlex 对中文名消歧不够可靠**。同名字跨多个机构、没有清晰主条目的情况很常见。所以不能单靠 OpenAlex 的搜索结果做决定，必须用官网论文做交叉验证。
2. **邮箱是ORCID搜索的黄金锚点**。ORCID 的 expanded-search 接口支持按 email 搜索（`email-self`字段），如果官网页面有邮箱，ORCID 命中率很高。
3. **Semantic Scholar 的论文列表通常不如 OpenAlex 完整**。物理领域的论文在 Semantic Scholar 中覆盖率不如 CS 领域，建议 OpenAlex 为主、S2为辅。
4. **arXiv API 对中文名搜索支持差**，arXiv 上的中文作者名通常是拼音，很难直接匹配。

#### 2.1.5 基础画像输出格式（01_基础画像.md）

每节固定，字段固定，没有的数据留空不编：

```
01_基础画像.md  →  9 节结构

Frontmatter（机器可读）:
  affiliation / department / tags / source_updated /
  各平台ID（orcid / openalex_id / s2_id 如有）

1. 身份标识
   姓名（中/英/其他拼写）| 职称 | 机构 | 部门 | 邮箱 | 主页 | 各平台ID
   纯客观事实

2. 学术履历时间线
   教育：年份 → 学校 → 学位
   工作：年份 → 机构 → 职位
   来源标注（每条来自哪个URL）

3. 研究方向
   3a. 总体概览（一段话，让外行看懂）
   3b. 具体方向（分点，每点1-2句）

4. 代表性论文
   表格：序号 | 标题 | 期刊 | 年份 | DOI | 引用数（如有）| 一句话说明
   分两群：近期工作 / 高引工作
   来源交叉比对标示

   总引用数、h-index（如有）、获奖、人才项目
   数据驱动，不编分析

   博导 / 博后老板 / 高频合作者（列出合作关系说明）
   来源标注

   来源URL

   多源比对一致性表
   缺失信息标注（哪些没找到）
```

#### 2.1.6 关键设计原则（本段的约束）

- **所有信息必须标注来源URL**。不标注来源的信息不写入画像。
- **缺失比编造好**。某数据源返回为空 → 标注"未找到"，不生成AI推断。
- **交叉比对是核心**。同一条信息至少有2个来源时才写入，单一来源标注"仅单源"。
- **多源冲突时**。在画像中列出多个来源的不同信息，不加AI仲裁。
- **"洞见"仅限于研究主题的时间变化、机构轨迹、合作网络密度**。这三类是从数据中直接可读的，不是AI编的故事。

### 2.4 数据源

| 源 | 协议 | 免费额度 | 用途 |

### 2.5 数据组织

每个老师一个独立目录（`项目/导师/<姓名>/`），全身只在这里存一份。`01_基础画像.md` 顶部用 YAML frontmatter 标注分类元数据：

```yaml
---
affiliation: 中科院物理所
department: 超快物质科学中心
tags: [超快光学, 强场物理, 阿秒科学]
---
```

`build_index.py` 扫描 `项目/导师/` 下所有 frontmatter，自动重建两个索引：

- **按机构.md**：学校→学院→部门→老师（树形）
- **按方向.md**：研究方向→老师（跨机构散点网络）

新增一个老师 → 脚本重跑 → 索引自动更新。不加新老师时不需要跑脚本。

### 2.6 贡献与迭代模式

| 层 | 门槛 | 内容 |
|:---|:---:|:-----|
| 数据修正 | 极低 | PR 改 `.md` 文件里的论文标题/DOI/URL |
| 新增导师 | 低 | 跑一次流水线，核实产出，PR 合入 `项目/导师/<姓名>/` |
| 方法论改进 | 中 | 优化 SKILL.md / 模板结构 |

核心迭代循环：

```
用户跑实例 → 发现问题 → 修正 → PR 合入 → 下一个人拿到修正版
       ↕
维护者更新 SKILL.md/模板 → 所有未来实例受益
```

PR 合并天然零冲突（每个导师独立目录）。
