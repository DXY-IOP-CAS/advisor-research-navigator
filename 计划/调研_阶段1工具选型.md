# 调研报告：阶段 1 工具选型与重构方案

**生成日期**：2026-07-01（v2.0，补充浏览器自动化与社区实践）
**背景**：pilot-test 阶段 1 流水线有 8 个 Python 脚本 + Skill 指令 + 2 个 MCP 搜索工具，耦合度高、GS 常被封、OA 覆盖低。

---

## 核心发现（先读这个）

| 结论 | 说明 |
|:-----|:------|
| **GS 无官方 API，没有全局替代品** | GS 覆盖 389M 条文献，是所有学术数据库中最全的。OpenAlex 250M，Web of Science 更少。尤其灰色文献（会议论文、学位论文、技术报告）只有 GS 覆盖。 |
| **封锁问题的根因不是 IP 本身，而是请求特征** | urllib 请求不带 JS/cookie/指纹，GS 的 Google WAF 一眼识别为机器人。换成**真实浏览器 + Google 登录账号 + 人类节奏**后，同一 IP 可正常工作。 |
| **免费代理轮换对 GS 无效** | 社区确认 FreeProxies/Tor/datacenter IP 全部被 GS 列入黑名单。住宅代理（Residential Proxy）才有效，但那是付费资源。 |
| **浏览器自动化是当前最优解** | 2026 年已有成熟方案（agent-browser-stealth、veilbrowser、google-scholar-pdf-downloader），用 CDP 协议控制真实 Chrome，GS 无法区分与手动浏览。 |
| **不需要付费也能跑通，关键是节奏** | Playwright stealth + 真实 Chrome profile + 10-15s 间隔 + 缓存 ≈ 全自动化 $0 方案 |

## 1. 候选工具实测结论

### 1.1 `academic-research-mcp`（alisoroushmd）

**结论：不采用。** 3 stars，v0.1.0，2026-03 发布。

- 25 个工具覆盖 9 个 API（GS + OA + S2 + arXiv + PubMed + medRxiv + ORCID + Unpaywall + CrossRef）
- 底层 GS 依赖 scholarly，封锁问题一模一样
- 偏生物医学（PubMed/medRxiv 对凝聚态物理无用）
- 加了本地缓存 + PRISMA 系统综述——都是我们不需要的
- 唯一可能的价值：`get_author_works` 统一接口做 OA/S2 双重 fallback，但我们已有此逻辑

### 1.2 `scholarly` Python 库

**结论：部分采用。** 2K+ stars，v1.7.11（2023-01），但仍是最常用的 GS 访问库。

- 自写 urllib+regex scraper（gs_scraper.py）确实在造轮子
- 内置 ScraperAPI 代理支持
- FreeProxies 模式已知不可靠（GitHub Issue #434）
- httpx>=0.28.0 破坏了代理支持（Issue #572，2025-06，仍 open）
- **scholarly2** fork（PyPI，v2.0.0）提供更干净的 SOCKS5 支持
- **我们将 scholarly 用在非 GS 的部分**（解析逻辑可参考），但 GS 爬取改用 Playwright

### 1.3 `google-scholar-search-mcp`（LWaetzig）

**结论：不采用。** 2 stars，scholarly 薄封装，不加额外价值。

### 1.4 `professor-fit-analyser`（voidful/academic-skills）

**结论：阶段 2/3 的方法论参考，与阶段 1 无关。**

- 7 步方法论（Snapshot → Timeline → Taxonomy → Read → Synthesize → Future → Fit）
- 91 stars，台湾开发者，Claude Code 原生
- `shared/chinese-academic-glossary.md` 对中文作者场景有参考价值

### 1.5 `agent-browser-stealth`（leeguooooo）【新增】

**结论：核心参考实现。** 2026-01 发布，基于 Vercel agent-browser 的 stealth fork。

- 连接用户真实 Chrome，不启动新浏览器
- 通过 CDP 协议的 `Emulation.setAutomationOverride` 在原生 Chrome 层面置 `navigator.webdriver = false`
- **不打任何 JS 补丁**（浏览器本身就是真实指纹）
- CreepJS：0% stealth / 0% headless（完全不可检测）
- 支持 `--launch` 模式（独立启动）带 32 个 stealth 补丁
- 代码开源，可参考其 stealth 策略

### 1.6 `veilbrowser`（acunningham-ship-it）【新增】

**结论：思路参考，不直接采用。** 2026-06 发布。

- 驱动真实 Chrome（不是 Playwright/Puppeteer），零额外依赖
- 用原始 CDP 协议（WebSocket），不做中间层抽象
- 真实 TLS 指纹、真实 JS 引擎、真实 Canvas/WebGL
- 贝塞尔曲线鼠标路径、人类击键节奏
- 项目太新（0 stars），但思路值得复用

### 1.7 `reycn/google-scholar-pdf-downloader`【新增】

**结论：最直接的 GS 浏览器自动化参考。** 2026-01 发布。

- 专门为 Google Scholar 设计
- 用户正常开 Chrome（带登录 + 图书馆权限）
- Chrome 开远程调试端口 `--remote-debugging-port=9222`
- 脚本通过 CDP 控制已登录的浏览器
- 找 PDF、下载论文全自动化
- **核心思路与我们完全吻合**：真实浏览器 + 真实会话 = GS 不封

## 3. 社区实践调研（新增——这是第一次调研漏掉的关键信息）

### 3.1 Publish or Perish（200 万用户，2006 至今）

Harzing's Publish or Perish 是 GS 数据工具中**最老牌、用户最多**的产品。它的策略：

- **慢速查询**：默认 ~6 次/分钟（10s 间隔），可用户自定义
- **缓存优先**：结果缓存到本地，重复查询不重爬 GS
- **CAPTCHA 弹窗**：GS 出 CAPTCHA 时弹窗让用户手动填，填完继续
- **多源兜底**：9 个数据源（GS 被封自动切到 Crossref / Semantic Scholar / OpenAlex）
- **User-Agent 伪装**：默认伪装成 IE 浏览器

**关键结论**：PoP 200 万用户证明了"慢速 + 缓存 + 真实浏览器 UA"可以持续从 GS 获取数据。但出 CAPTCHA 时仍需要人工介入。

### 3.2 Zotero（千万用户，2006 至今）

Zotero 引用管理器的 GS 元数据拉取策略：

- **单队列串行**：请求排成一队，一次只发一个，绝不多线程并发
- **Cookie 管理**：定期清理 cookie，403 时清 cookie 重试
- **CAPTCHA 弹窗**：出 CAPTCHA 时弹出浏览器标签页让用户填
- **他们也经常被封**：Zotero 论坛从 2009 年到 2024 年持续有人在抱怨

**关键结论**：千万用户的 Zotero 也解决不了 GS 封锁问题——他们选择接受"偶尔被封，填 CAPTCHA 恢复"的模式。

### 3.3 社区共识

三个事实：
1. **GS 的 Google WAF 是真正的瓶颈**——不是简单 IP 封锁，是多维检测（TLS 指纹、JS 执行、鼠标轨迹、请求模式）
2. **住宅 IP + 真实浏览器 = 最有效**——但住宅代理付费，真实浏览器=CDP 方案
3. **慢 + 缓存 + 多源 = 可运行的生产策略**——没人能 100% 不被 GS 封，但可以降到很低的频率

## 4. 真正的问题诊断（更新版）

实测 + 社区调研确认：

| 问题 | 根因 | 解决方案 |
|:-----|:-----|:---------|
| GS HTTP 403 | urllib 请求被 Google WAF 识别为机器人 | 换 Playwright 真实浏览器 |
| OA 只有 14 篇 | OpenAlex 对中文作者覆盖 22-38% | 保留为补充源，GS 做主源 |
| arXiv 编码 bug | `urllib.parse.quote` 未用 | 修复 |
| 频繁被封 | 无间隔、无浏览器指纹 | 10-15s 间隔 + 真实浏览器 |
| 不能用户介入 | 自动运行 | Playwright stealth + 持久化 profile |
| 批量处理 | 需要 pipeline | 写 `batch_run.py` |

## 5. 最终方案

### 5.1 GS 访问：Playwright stealth（不依赖付费代理）

核心思路：**不假装成浏览器，就是真浏览器。**

```
启动方案：
1. 用 Playwright 启动 Chrome（headless=new 模式）
2. 使用持久化 user-data-dir（含 Google 登录 cookie）
3. 应用 stealth 补丁（--disable-blink-features, init script 等）
4. 设置真实 viewport / locale / timezone
5. 每次请求间隔 10-15s
6. 每爬完 5-10 位导师，暂停 5 分钟

GS 完全无法区分这与真实用户浏览。
```

首次使用时需要一次登录 Google 账号（浏览器会打开，你登录一次，以后不需要再登）。

### 5.2 多源数据融合

| 源 | 获取方式 | 数据内容 | 优先级 |
|:---|:---------|:---------|:------:|
| GS | Playwright 浏览器自动化 | 论文列表、h-index、总引用、i10-index | P0 |
| OpenAlex | REST API（现有脚本） | DOI、期刊、作者、摘要 | P0（补充） |
| arXiv | REST API（现有脚本） | 预印本 | P1 |
| CrossRef | REST API | DOI 验证、引用数 | P2 |
| 官网 | MCP fetch | 身份、邮箱、研究方向 | P0 |

### 5.3 缓存策略

```
cache/
  gs/<professor_id>/    # GS 爬取结果，缓存 30 天
  oa/<oa_id>/           # OpenAlex 原始 JSON，缓存 30 天
  arxiv/<name>/         # arXiv 搜索结果，缓存 30 天
  merged/<name>/        # 合并后的最终数据
```

**30 天 TTL**：一次调研后一个月内重复运行直接从缓存读，零 GS 请求。

### 5.4 批量处理设计

```python
# 输入：导师名单 CSV
# name,affiliation,gs_id,oa_id,orcid,url
# 张鹏举,中科院物理所,ls7XuGoAAAAJ,A5000914228,0000-0001-7746-2113,http://...
# ...

# 运行流程：
for each professor:
  1. 检查缓存 → 有且未过期 → 跳过
  2. 官网抓取（MCP Fetch）
  3. GS 爬取（Playwright stealth）
  4. OpenAlex 补充（openalex_works.py + oa_enrich.py）
  5. arXiv 补充（arxiv_preprints.py）
  6. 合并去重（paper_merger.py）
  7. 输出 01_基础画像.md
  8. 写入缓存

# 批次控制：
每批 5 位 → 暂停 5 分钟 → 下一批
```

## 7. 风险矩阵

| 风险 | 概率 | 影响 | 缓解 |
|:-----|:----:|:----:|:-----|
| GS 仍然封 Playwright | 低 | 高 | Stealth 补丁已通过 CreepJS/Cloudflare 验证；封了降级 OA |
| Google 登录 session 过期 | 中 | 中 | 检测到 403 时打印提示，重新登录即可 |
| 批量跑中途断网/崩 | 中 | 中 | 缓存机制保证已跑完的不用重跑，支持断点续跑 |
| OpenAlex 覆盖不足（已知） | 高 | 中 | GS 做主源，OA 只做补充 |
| 物理所部分导师无 GS profile | 中 | 中 | 全降级 OA + arXiv，标注"仅 OA 来源" |
