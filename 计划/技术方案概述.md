# 技术方案概述

**生成日期**：2026-06-30
**来源**：需求确认讨论 + 同类工具调研（`05_同类工具调研报告.md`）

---

## 1. 工具定位（已确认）

**一句话**：给定一个导师，输出四份结构化文档——从调研到教学，帮助学生快速理解该导师的研究方向。

**三不原则**：
- ❌ 不做导师口碑/评分——不评价人品、管理风格、毕业难度，避开所有争议漩涡
- ❌ 不做学生→导师匹配——不与 PhdFit/PhDuo 等匹配平台竞争
- ❌ 不做专业内容创作——写路径索引和学习路线，不写教材正文

**三大职能**：调研（收集信息）→ 综述（分析串联）→ 教学（设计递进路线），三职能递进，最终产出学习讲义。

## 2. 四段流水线（架构定稿）

```
输入: 导师姓名 + 学校/机构 + 方向关键词（必要时补充主页 URL）
  │
  ▼
┌─────────────────────────────────────────────────────┐
│  第1段 · 基础信息调研                                 │
│  输入: 姓名 + 学校 + 方向关键词                        │
│  动作: 多源信息采集（OpenAlex / Semantic Scholar /    │
│        ORCID / arXiv / 机构主页 / 中文百科）           │
│  产出: 01_基础画像.md                                  │
│        → 教育背景、职位、研究领域标签、代表性论文列表    │
│        → 多源交叉比对的完整 profile                    │
└─────────────────────────────────────────────────────┘
  │  输出作为第2段的"领域范围"输入
  ▼
┌─────────────────────────────────────────────────────┐
│  第2段 · 领域发展脉络                                 │
│  输入: 第1段产出的研究方向标签 + 代表性论文             │
│  动作: 追溯引文网络 → 识别关键里程碑论文                │
│        → 理清子领域的技术路线分岔                      │
│  产出: 02_领域脉络.md                                  │
│        → 该子领域的发展树（起点→分岔→当前前沿）         │
│        → 每篇里程碑论文的贡献一句话                     │
└─────────────────────────────────────────────────────┘
  │  输出作为第3段的"认知框架"输入
  ▼
┌─────────────────────────────────────────────────────┐
│  第3段 · 论文序列与定位                               │
│  输入: 第2段的领域脉络 + 第1段的论文列表               │
│  动作: 把导师的论文放在领域脉络里标注位置                │
│        → 识别"什么节点加入、做了哪个分支、当前在哪"     │
│  产出: 03_论文定位.md                                  │
│        → 时间线叙事（入职→早期→转型→当前）            │
│        → 每篇论文在领域树上的位置标注                  │
└─────────────────────────────────────────────────────┘
  │  输出作为第4段的"目标前沿"输入
  ▼
┌─────────────────────────────────────────────────────┐
│  第4段 · 从零到前沿学习讲义                            │
│  输入: 第1段的基础 + 第2段的领域地图 + 第3段的前沿定位 │
│  动作: 从高数+线代开始反推知识链                        │
│        → 只提取"大字典"中真正需要的知识点               │
│        → 串联成一条递进学习路径                        │
│  产出: 04_学习讲义.md                                  │
│        → 章节路径 + 每章知识点 + 推荐资源索引          │
│        → 遵循"起承转合"叙事流                          │
└─────────────────────────────────────────────────────┘
  │
  ▼
输出: 项目/<导师名>/ 下的四份 .md 文件
```

每段完成后，AI 先自检（查来源是否可验证、信息是否一致），再进入下一段。用户可在任一段输出后介入修正。

## 3. 交付形态

**不是网站/App**，是一套可被 AI 读取的指令文件 + 极简脚本。

```
仓库根目录/
├── CLAUDE.md                        # 宪法层：项目身份 + 纪律
├── .claude/skills/
│   └── research-advisor/
│       ├── SKILL.md                 # 流水线指令（四段执行步骤）
│       └── 模板/                     # 每段的 .md 输出骨架
│           ├── 01_基础画像.md
│           ├── 02_领域脉络.md
│           ├── 03_论文定位.md
│           └── 04_学习讲义.md
├── 配置/                             # 数据源配置
│   └── apis.json                    # OpenAlex / S2 / arXiv API 端点
├── 脚本/                             # 辅助脚本（纯可选）
│   ├── fetch_papers.py              # 调 OpenAlex 获取论文列表
│   └── validate_output.py           # 检查输出字段完整性
└── 项目/                             # 每个导师的产出
    ├── 张鹏举/
    │   ├── 01_基础画像.md
    │   ├── 02_领域脉络.md
    │   ├── 03_论文定位.md
    │   └── 04_学习讲义.md
    └── (后续导师)/
```

核心在 **SKILL.md**——不是传统程序，是一份 AI 能按步骤执行的流程文档。用户只需：

```
装 Skill → /research-advisor "张鹏举" "中科院物理所" "超快物质科学"
       ↓
Agent 自动跑四段 → 项目/张鹏举/ 下生成四份 .md
       ↓
用户读 → 有错改文件 → 没事就不用管
```

零服务器、零数据库、零部署。

## 4. 数据源层

| 数据源 | 协议 | 免费额度 | 覆盖范围 | 用法 |
|:-------|:----|:--------:|:--------|:-----|
| OpenAlex | REST API (无 key) | ∞ | 2.5亿+论文，含机构/作者消歧 | 第1段主源：论文列表、引用、合作者 |
| Semantic Scholar | REST API (无 key) | 100 req/s | 2亿+论文，含 TLDR/influence | 第1段补充：论文影响力、领域分类 |
| arXiv API | REST 搜索 | ∞ | 预印本 | 第1段补充：最新论文 |
| ORCID | REST API (无 key) | ∞ | 研究者 profile | 第1段：职业历史交叉验证 |
| 机构主页 | HTTP 抓取 | ∞ | 官方介绍 | 第1段：中文/机构特有信息 |
| 中文百科/新闻 | Web 搜索 | 取决于搜索工具 | 中文信息 | 第1段补充：背景、采访、报道 |

**不引入**：Neo4j 等重型图数据库、CNKI（需授权/付费）、Scopus（需付费）。

## 5. 社区模型

### 5.1 用户故事

**用户 A（你自己）**：跑张鹏举实例 → 用产出帮助理解方向 → 修正信息 → PR 合入主线。

**用户 B（朋友/同学）**：装 Skill → 跑自己要申的导师 → 核实内容 → PR 合入导师数据（不用改代码）。

**用户 C（陌生贡献者）**：发现数据有误 → 提 Issue 或直接 PR 改 `.md` 文件。

### 5.2 贡献渠道分三层

| 层 | 门槛 | 能做什么 |
|:---|:---:|:---------|
| 数据修正 | 极低 | 改 `.md` 文件里的论文标题/DOI/URL |
| 新增导师 | 低 | 提供导师名+学校，跑一次流水线，核实产出 |
| 方法论改进 | 中 | 优化 SKILL.md 的步骤描述、模板结构 |

要合并用户新增的导师数据到主线只需要：用户 A 的 `项目/张鹏举/` 目录 → 合并到仓库的 `项目/张鹏举/`。每个导师是独立目录，互不依赖，合并没有冲突。

### 5.3 迭代模式

```
用户跑实例 → 发现问题 → 反馈/PR → 合并 → 下一个人拿到修正版
       ↕
维护者更新 SKILL.md/模板 → 所有未来实例受益
```

第一层（数据）和第二层（方法）的迭代互不阻塞。

## 6. 不做的事（边界清单）

- ❌ 不写具体教材内容——讲义是路径索引，不是教科书
- ❌ 不替代学生主动学习——给路线，不给答案
- ❌ 不点评导师/推荐方向——价值判断用户自己做
- ❌ 不捏造论文/链接——所有引用必须有 URL
- ❌ 不收取费用——免费开源，有人赞助是以后的事
- ❌ 不搭建 Web 服务器——不部署、不运维
