本计划书统一按以下规则组织。

**编号体系**：h2（##）用于章（第X章），h3（###）用于节（X.Y），h4（####）用于节内子段（X.Y.Z）。禁止 h5+。

## 第1章: 需求背景

### 1.1 起因 / 工具目标

每年硕士/博士生选导师是刚需。每年研一 3-4 月学生面临报志愿和套磁，核心痛点在四个环节。官网信息粗糙：只有论文列表和标题，没有科普级解读。有论文看不懂：论文要求研究生以上知识，学生只有本科基础。资源散乱无索引：论文、教材、视频各在各处。调研易绕圈：不知道该读什么、按什么顺序。

工具用四段嵌套解决这些问题。每一段的输出为后一段提供认知上下文，范围逐段收窄，理解逐段加深。工具同时承担三个递进职能：调研（全面收集信息）、综述（分析串联知识）、教学（设计递进学习路线）。

| 段 | 做什么 | 做法（高层次） | 目的 |
|:---|:---|:---|:---|
| 第1段 | 老师基础信息调研 | 地毯式收集官网、学术平台、中文百科、采访稿等多维信息，多源交叉比对 | 输出基础画像，明确领域范围 |
| 第2段 | 领域发展调研 | 基于第1段的领域范围，追溯子领域发展脉络和当前阶段 | 输出认知地图，理解位置 |
| 第3段 | 论文串联与定位 | 以第2段认知框架重新审视论文序列 | 标注每篇论文在领域树上的位置 |
| 第4段 | 从零到前沿讲义 | 从高数+线代起步反向设计学习路线 | 输出递进式讲义（小字典） |

嵌套逻辑：大领域一>小领域一>老师定位一>学习路径。后一段建在前一段的认知之上。

技术实现细节在第2章和附录 A。

### 1.2 目标用户画像

工具的目标用户是大二/大三物理本科生，正在准备申请硕博选导师。核心场景是研一 3-4 月报志愿时，用工具判断与导师研究方向的匹配度。

工具假设用户从高数和线代起步。实际用户可能已有四大力学、固体物理等基础，但所有高阶知识（复变、数理方法、高量、群论、量子多体）本质上是高数和线代的逐步推广。从根推导才能零跳步建立真理解。

教材在 AI 时代不再是逐页通读的课本，而是字典。本科到研一的教材体系分三个层面：

| 层面 | 课程 | 定位 |
|:---|:---|:----|
| 数学基础 | 高数、线代、概率论、复变、数理方法 | 推导起点 |
| 物理基础 | 普物、四大力学、固体物理 | 物理直觉来源 |
| 高阶延伸 | 高量、群论、量子多体、量子场论、规范场论 | 大字典边界—领域触及的范围 |

教材从课本转向字典，有两个原因。第一，教材面向通用人才培养，覆盖完整体系，但具体方向只用其中部分知识点。逐页通读在无关内容上浪费大量时间。第二，学生可在 AI 辅助下按需读取字典条目，带着研究方向去学，既能深化具体知识点理解，又能强化整体框架认知。

第4段产出的讲义是个性化小字典。它从教材体系（大字典）中只提取第1-3段调研确认需要的知识点，串联成一条从高数+线代到前沿的逻辑链。小字典扎根于大字典，以教材体系为参照系，约束 AI 创作边界，防止偏离轨道。

一个可扩展的方向：工具可附带教用户如何配合 AI 递进学习。AI 的优势是能根据用户水平动态适配，但多数人不擅长用 AI 辅助学习。加入"如何用 AI 学"的引导可降低门槛。此方向未确定，作为后续讨论参考。

### 1.3 工具差异化与定位

18+ 同类工具调研后确认：市场无直接竞品。现有工具分三类，各占一个不重叠的位置。

| 类别 | 代表工具 | 做什么 | 与我们的重叠 |
|:-----|:---------|:-------|:-----------:|
| 导师评价 | 导师评价网、GradChoice、advisor-calculator | 口碑评分（人品、管理、毕业） | 不重叠——我们不碰评价 |
| 导师匹配 | PhdFit、StreamlinedAI、PhDuo | 从学生 CV 推荐导师 | 不重叠——我们不做推荐 |
| 学者画像 | supervisor、SciAtlas、ai-talent-graph | 论文+主页一>信息聚合报告 | 第1段部分重叠 |

学者画像工具停在"导师长什么样"的层面。我们继续往前推到领域发展、论文定位和教学讲义。

工具的准确定位包含四层含义。第一，通用模板引擎：对任何机构的任何老师适用，跨老师复用零边际成本，首个实例为张鹏举（IPHY 超快物质科学中心）。第二，开源方法论：Claude Code Skill + 项目级 scripts/config + Markdown 模板形态，不是网站/App，零部署。第三，从调研走到教学：调研+综述+教学三职能递进。第四，调研驱动设计：每项决策经同类工具验证，不凭空发明。

一条根本差异区分了我们和所有竞品：同类工具帮学生找谁合适（匹配引擎）或知道老师什么样（画像工具）；我们帮学生学懂一个方向。

工具六不做：不写教材正文；不替代主动学习；不点评老师或推荐方向；不捏造论文/教材链接；不评价导师口碑；不搭建网站。

## 第2章: 工具设计

**本章节为当前活跃章节**——阶段 0/1 优先，阶段 2/3/4 待后续。

**注**：原 `工具设计.md` 文件因混乱被拆分出去。当前内容以本计划书为准。

### 2.1 整体工作流

用户提供导师姓名 + 机构 + 官网 URL，工具执行五份文档流水线。每份文档的认知上下文来自前一份，深度逐段加深。

**流水线逻辑**：

```
阶段0：浅层收集 → 列出调研计划让你审（链接/锚点）→ 确认后批量查询 → 锁定身份
    ↓
阶段1：深调研 → 论文按学术履历阶段分组 → 结构化画像 + 完整论文列表
    ↓
阶段2：领域脉络调研 → 子领域发展树、里程碑、当前前沿
    ↓（认知回环，非单向）
阶段3：以领域框架重审论文 → 标注每篇论文在领域树上的位置
    ↓
阶段4：从高数到前沿的递进学习路径
```

阶段2建立的领域认知，反过来让阶段3的论文串联更有深度——不是简单的时间线排列，而是在"老师做了什么研究"的基础上讲清"为什么做、为什么这么做"。五份文档每一份都服务于下一份，阶段3回馈阶段2的认知积累形成闭环。

**阶段一览**：

| 阶段 | 输入 | 产出 | 核心任务 | 人闸 |
|:-----|:-----|:-----|:---------|:----:|
| 0 浅调研 | 姓名+机构+URL | `00_身份验证卡.md` | 官网抓取 → 列调研计划给你审 → 确认后批量查 ID → 论文比对锁定身份 | 两次 |
| 1 基础画像 | 确认的身份 | `01_基础画像.md` | 全平台深调研，论文按学术履历分组，9 节结构化画像 | 一次 |
| 2 领域脉络 | 画像+论文列表 | `02_领域脉络.md` | 追溯子领域发展树和里程碑 | TBD |
| 3 论文串联 | 领域框架 | `03_论文定位.md` | 每篇论文在领域树上的位置标注 | TBD |
| 4 学习讲义 | 前三段全部产出 | `04_学习讲义.md` | 从高数+线代到前沿的递进学习路径 | TBD |

**六条设计原则**贯穿全部阶段：
1. 来源必标 URL：每条外部信息末尾带来源链接
2. 缺失优于编造：找不到的字段标 `[未找到]`，不捏造
3. 双源才标已验证：单一来源的信息标注"仅X来源"
4. 冲突列出不仲裁：不同源数据矛盾时并列写出，你判断
5. 身份消歧不走全自动：最终锁定权在你
6. 每段自检：通过后才进入下一段

### 2.2 阶段 0：浅调研 + 人闸确认

#### 2.2.1 工作流程

不是"全自动查完所有源再让你确认"，而是"浅层收集 → 列出调研计划让你审（链接、锚点、数据源）→ 确认无误后再批量查询"。

**六步流程**：

```
Step 1: 抓官网
  用请求或 MCP fetch 读你给的 URL。
  提取: email/论文标题（至少 2 篇）/职称/研究方向关键词
  容错: 404 就停，让你确认 URL。缺字段标缺失。

Step 2: 学科识别
  → python scripts/discipline_classifier.py --text "方向关键词" --affiliation "机构名"
  匹配结果决定后续启用哪些数据源（高能→INSPIRE-HEP，天文→NASA ADS）。
  解析 JSON 输出，取 primary 字段。

Step 3: 列出调研计划 → 你审
  输出一份简要清单，包括：
  - 锚定论文（官网提取的论文标题，作为后续比对的 ground truth）
  - 将查询的 ID 源（OpenAlex / ORCID / S2 / ArXiv / INSPIRE / ADS）
  - Web 搜索关键词（采访 / 新闻 / 讲座 / B站 / 百度百科）
  你确认：这些链接对不对？这些论文是这个老师的？锚点没问题？
  确认后继续，modify 修改输入重跑，no 取消。

Step 4: 批量执行查询
  并行查 6 个 ID 源（技术原理见 2.2.2）。
  全降级模式：查不到不报错退出，跳过后继续。

Step 5: 论文标题交叉比对
  identity_resolver.py（或 AI 手动）拿官网论文标题 vs OpenAlex 候选人的论文列表。
  归一化标题匹配，算每个候选人的命中率。

Step 6: 输出 00_身份验证卡.md
  按模板渲染（.claude/skills/research-advisor/assets/00_身份验证卡.md）。
  → STOP，等你的 yes/modify/no。
```

**人闸位置**：
- 第一次：Step 3，你审调研计划，确认锚点无误
- 第二次：Step 6，你确认身份锁定正确

**全降级模式**：所有 ID 源查不到时，输出降级版卡片，列已尝试的源，让你补充信息（如 ORCID 链接、Google Scholar 链接）。

#### 2.2.2 技术原理

**核心验证机制**：
- 官网论文做 ground truth
- OpenAlex 候选人做池（ML 聚类，提供候选列表）
- 论文标题归一化匹配做筛
- 你做终审

```
官网提取 3+ 篇论文标题（ground truth）
       ↓
OpenAlex 返回 N 个候选人（每个含论文列表）
       ↓
对每个候选人：官网 n 篇论文中有几篇出现在此人的论文列表中？
命中率 = 命中数 / 官网论文总数

≥80% → 高置信
50-80% → 中置信
<50% → 低置信，需人工判断
```

为什么不用纯 ML 消歧？OpenAlex 的 XGBoost 对中文名跨机构已证明不可靠（张鹏举实测 25 个候选人）。论文标题比对更简单、可解释、可逐条复查。

**标题归一化**：去标点 + 小写 + 去空格，做子串匹配（防止小误差）。

**ID 源分工表**（互相补充，不是可选替代）：

| 源 | 查什么 | 优势 | 弱点 |
|:---|:-------|:-----|:-----|
| ORCID | by email | 一对一，最可靠 | 注册率低 |
| OpenAlex | 姓名+机构 ROR | 覆盖最广，论文最全 | ML 聚类，中文不准 |
| Semantic Scholar | 姓名拼音 | TLDR 摘要，引用数据 | 同 ML 聚类问题 |
| ArXiv Author ID | 姓名 | 物理/数学常用 | 需主动 claim |
| INSPIRE-HEP | 姓名 | 高能物理事实标准 | 仅高能物理 |
| NASA ADS | 姓名 | 天文事实标准 | 仅天体物理 |

**学科自动识别**：`discipline_classifier.py` 把研究方向文本匹配预设关键词字典（`config/sources.json` 的 `disciplines` 段）。匹配到高能/天体物理时，后续启用对应的专业数据源。其他学科（凝聚态 / AMO / 量子信息）走通用源 OpenAlex + arXiv + S2。

**不引入的外部源**：Scopus、Web of Science（付费）；CNKI/万方（需授权）；DBLP（CS 专用）；基础数据不存 Neo4j（重）。

#### 2.2.3 输出规格

| 文件 | 位置 | 内容 |
|:-----|:-----|:------|
| `00_身份验证卡.md` | `项目/导师/<姓名>/` | 姓名/机构/邮箱/职称、各 ID 源匹配情况、论文比对命中率及详情、数据源清单、综合建议、用户决策点（yes/modify/no） |

模板见 `.claude/skills/research-advisor/assets/00_身份验证卡.md`，用 `{{placeholder}}` 标注模板字段。

### 2.3 阶段 1：深调研 + 基础画像

#### 2.3.1 工作流程

身份确认后，多源拼图组装完整画像。四步：

```
Step 1: 用锁定 ID 拉 profile
  OpenAlex  /authors/{id} → 基本数据（总引用/总论文数/h-index）
  OpenAlex  /authors/{id}/works → 论文列表（最多 200 篇，按引用排序）
  S2        /author/{id}/papers?fields=tldr → 每篇论文的 AI 摘要
  ORCID     /{orcid-id}/educations, employments → 履历（如有注册）

Step 2: 不需 ID 的并行采集
  arXiv:        au:姓名拼音 → 最新预印本（补 OpenAlex 的时间差）
  Web Search:   姓名 + 机构 + 采访/讲座/新闻/B站/知乎/百度百科
  Google Scholar: 搜 profile URL（有则抓引用数据）
  官网深度抓取: 实验室介绍、招生信息、团队成员页

Step 3: 去重排序
  去重规则：DOI 优先 → arXiv ID → 归一化标题 ≥85%
  排序键：多源共证数 → 引用数 → 年份（降序）
  关键操作：论文按学术履历阶段分组
    - 博士期间论文
    - 博后期间论文
    - 独立工作后论文

Step 4: 输出 01_基础画像.md
  9 节 + YAML frontmatter（模板见 assets/01_基础画像.md）
  AI 自检：所有信息有 URL、字段无缺失、矛盾信息标出
  完成后给你审阅
```

**与旧设计的差异**：论文不是按引用数简单排序，而是按学术履历阶段分组。这样能看清"这个人在不同职业阶段做了什么研究、研究方向有什么变化"。

#### 2.3.2 技术原理

**画像 9 节结构**：

| # | 节 | 内容 | 主要来源 |
|:-:|:---|:-----|:---------|
| 1 | 身份标识 | 姓名/职称/机构/邮箱/各平台 ID | 官网 + API |
| 2 | 学术履历 | 教育→工作（含导师、机构、起止年份） | ORCID + 官网 + Web Search |
| 3 | 研究方向 | 总体方向 + 具体方向（中英文关键词） | 官网 + 论文自动提取 |
| 4 | 全部论文（按履历分组） | **全部已发表+预印本**，每阶段一组，每篇含标题/期刊/年份/引用数/角色/TLDR | OpenAlex + S2 + arXiv |
| 5 | 影响力指标 | 总引用/总论文/h-index | OpenAlex |
| 6 | 合作网络 | 博士导师、高频合作者、机构 | OpenAlex 论文共著 |
| 7 | 公开信息 | 新闻/采访/讲座/B站/知乎 | Web Search |
| 8 | 技术关键词 | 实验技术/理论工具/研究对象/应用方向 | 研究方向 + 论文关键词 |
| 9 | 来源说明 | 各数据源状态（成功/失败/未启用） | 自检 |

**数据源互补关系**：

```
OpenAlex → 论文骨架（标题/期刊/引用数/DOI）
    ↓
S2 → 论文内容理解（TLDR 摘要）→ 快速判断每篇论文的核心贡献
    ↓
arXiv → 最新预印本（补正式发表前的时间差）
    ↓
ORCID → 履历（教育/工作/导师）→ 论文分组依据
    ↓
Web Search → 公开活动（采访/讲座/新闻）→ OpenAlex/S2 给不了的信息
```

每个源提供一部分信息，重复信息做交叉验证，独有信息做互补覆盖。

**去重逻辑**：
- DOI 相同 → 同一篇（最高优先级）
- arXiv ID 相同 → 同一篇
- 归一化标题 ≥85% → 同一篇
- 共证源数 = 可信度标记（3 个源都有比只有 1 个源更可靠）

**论文分组逻辑**：

```
解析 ORCID 履历或官网 CV 中的起止年份：
  - 博士阶段: start_phd ~ end_phd
  - 博后阶段: start_postdoc ~ end_postdoc
  - 独立阶段: start_independent ~ present

每篇论文的发表年份落入哪个时间段就归入哪组。
如无精确年份，按机构 affiliation 推断。
跨阶段工作（如博士工作延至博后才发表）归属于实际完成阶段。
```

#### 2.3.3 输出规格

| 文件 | 位置 | 内容 |
|:-----|:-----|:------|
| `01_基础画像.md` | `项目/导师/<姓名>/` | YAML frontmatter + 9 节 + 完整论文列表（按履历分组） |

YAML frontmatter 统一键：`affiliation`、`department`、`tags`、`source_updated`、`orcid`、`openalex_id`、`arxiv_id`、`inspire_id`、`s2_id`、`google_scholar_url`。

未找到的字段标注 `[未找到]`。每条论文条目标注来源 URL。

## 第4章: 使用最佳实践

本章写给使用者（你自己）。**纠正两点：AI 不能替代 session 管理，工具不是万能胶。**

### 4.1 session 分割原则

每轮 session 只做一类工作。不要在同一轮里同时"设计文档 + 写代码 + 修 bug + 整理目录"——做混了就是我在这个窗口犯的错。

推荐分法：

| session | 任务 | 时长参考 |
|:--------|:-----|:--------|
| 设计 | 写计划书章节、定 workflow | 30-60 分钟 |
| 编码 | 写脚本、测试 | 60-90 分钟 |
| 整理 | 归档、改目录结构 | 15-30 分钟 |
| 调研 | 搜索资料、读工具文档 | 30-45 分钟 |

### 4.2 checkpoint 机制

每完成一个独立任务，写一段 checkpoint 到 `计划/上下文交接.md`。格式很简单：

```
2026-06-30 checkpoint: 跑通 identity_resolver 的 OpenAlex 查询（有 bug 未修）
```

写完后 `/clear` 再开下一个任务。

### 4.3 什么时候该停

- 纠正同一件事两次后 → 不继续，直接写 checkpoint 然后 `/clear`
- AI 开始自行加文档/重构/提议 → 停下，明确说"只做 X，别做 Y"
- 超过 45 分钟 → 该 session 已经够长了，收尾 or `/compact`

### 4.4 `/compact` 的正确用法

```
/compact 保留以下决策：A.1-A.7（数据源体系和架构形态），附录 C（部署清单），2.2.1（阶段 0/1 workflow）。丢弃：过往对话历史、旧代码输出、调试记录。
```

不要在自动 compact 触发后才处理。自动 compact 会丢关键早期决策。

## 附录 A: 已沉淀的技术决策

本附录收录本轮三窗口中已确定的技术决策。后续如有变更需走"用户拍板"流程。

### A.1 整体方向（已确认）

**A.1.1 信息源不是单一的**。早期讨论陷入"OpenAlex 是主源，其他补充"的死胡同。修正：物理学科按学科分层数据源优先级。

**A.1.2 身份消歧不走纯 ML**。OpenAlex 等 ML 聚类对中文名+跨机构不可靠（实测张鹏举 25 个同名候选）。我们的方法：官网论文做 ground truth，论文标题直接比对。

**A.1.3 必须有人闸**。身份消歧不允许全自动。任何 ID 锁定前用户确认，避免连锁错误。

**A.1.4 不报错退出**。所有 API/数据源失败时降级到其他源，不抛异常。

### A.2 数据源体系（已确认）

按学科+ID 类型分层：

**跨学科通用 ID 体系**：
- ORCID（by email，注册过就有，最可靠）
- OpenAlex Author ID（ML 聚类，覆盖广但中文不准）
- Semantic Scholar Author ID（ML 聚类，覆盖广，TLDR 资源）
- ArXiv Author ID（用户主动 claim 后生成，物理/数学/CS 常见）
- Google Scholar Profile（用户自建，无官方 API）
- CrossRef

**物理学科专用**：
- **INSPIRE-HEP** — 高能物理/粒子物理/核物理领域事实标准，API 全
- **NASA ADS** — 天体物理事实标准，需 API token

**网页抓取层**：
- 机构官网（HTTP fetch）
- Web Search（Serper/Exa/Tavily）
- 百度百科/知乎/B站（Web Search）

**不引入**：Scopus、Web of Science（付费）；CNKI/万方（需授权）；DBLP（CS 专用）；Neo4j（重）。

详见 `config/sources.json`（项目级共享配置）。

### A.3 架构形态（已确认）

工具不是"AI 在空白 Claude Code 里按文档操作"，而是：

```
pilot-test/
├── .claude/skills/research-advisor/   # Claude Code Skill（指令 + 模板）
│   ├── SKILL.md                       # 入口，渐进式披露
│   ├── assets/                        # 输出模板
│   └── references/                    # 按需加载的细节
├── config/
│   ├── sources.json                   # 数据源 + 学科字典
│   └── disciplines.json               # [TBD] 学科→关键词独立文件 [可选]
├── scripts/                           # 跨阶段共享 Python 脚本
│   ├── discipline_classifier.py       # [已写 v0.1]
│   └── identity_resolver.py           # [已写 v0.1，OpenAlex paper 匹配待测]
├── cache/                             # 临时缓存
└── 项目/导师/<姓名>/                  # 每位导师的产出
    ├── 00_身份验证卡.md
    ├── 01_基础画像.md
    ├── ...
```

**渐进式披露原则**：SKILL.md ≤500 行，只含路由 + 当前阶段入口。详细步骤进 `references/<phase>.md`。本原则来自 Trail of Bits `designing-workflow-skills`。

**Skill 不内嵌 scripts/config**——脚本和配置归项目根，SKILL.md 用相对路径引用。这是按 CLAUDE.md "隔离边界" 原则。

### A.4 学科 → 数据源映射（已确认）
